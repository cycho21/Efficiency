# 코드 리뷰 체크리스트

이 체크리스트는 Claude를 활용한 효율적인 코드 리뷰 프로세스를 안내합니다.

**사용 시점**: Pull Request 생성 전, 코드 품질 검토, 리팩토링 검증 시  
**예상 소요 시간**: 준비 5-10분, 리뷰 15-45분 (변경 범위에 따라)

---

## 1. 리뷰 준비

### 1.1 리뷰 범위 명확히 정의
- [ ] 리뷰 초점을 명확히 정의했다 (보안/성능/스타일/로직 중 선택)

**중요한 이유**:
리뷰 범위를 명확히 하면 집중도가 높아지고 불필요한 토큰 소비를 방지합니다.

**리뷰 범위 예시**:
- **보안**: 인증/권한, 입력 검증, 민감 정보 처리
- **성능**: 알고리즘 효율성, 데이터베이스 쿼리, 메모리 사용
- **스타일**: 코딩 컨벤션, 네이밍, 가독성
- **로직**: 비즈니스 로직 정확성, 엣지 케이스 처리, 에러 핸들링
- **전체**: 모든 측면 (시간이 더 소요됨)

**Claude에게 전달할 정보**:
```
리뷰 범위: [보안/성능/스타일/로직/전체]
우선순위: [Critical/Important/Nice-to-have]
특별히 확인할 사항: [구체적 우려사항]
```

---

### 1.2 변경 파일만 선택적으로 제공
- [ ] 실제 변경된 파일만 리뷰 대상으로 제공했다

**이유**:
- 변경되지 않은 파일은 리뷰가 불필요
- 토큰 낭비 방지
- 리뷰 집중도 향상

**효율적인 접근**:

❌ **비효율적**:
```bash
# 전체 프로젝트를 Claude에게 제공
"모든 파일을 읽고 PR을 리뷰해줘"
→ 수백 개 파일 읽기, 토큰 낭비
```

✅ **효율적**:
```bash
# 변경된 파일만 확인
git diff --name-only main...feature-branch

# diff만 제공
git diff main...feature-branch > changes.diff
"이 diff를 리뷰해줘"
→ 변경 사항만 집중 리뷰
```

**실용 팁**:
- 10개 파일 미만: 파일별로 제공
- 10개 이상: diff 형식으로 제공
- 대규모 변경: 주요 파일만 선택하여 리뷰

---

### 1.3 PR 설명 충분히 작성
- [ ] PR 설명에 변경 목적, 주요 변경 사항, 테스트 방법을 명시했다

**PR 설명에 포함할 내용**:
```markdown
## 변경 목적
[왜 이 변경이 필요한가?]

## 주요 변경 사항
- [파일명]: [변경 내용 요약]
- [파일명]: [변경 내용 요약]

## 테스트 방법
- [ ] 단위 테스트 추가/통과
- [ ] 통합 테스트 통과
- [ ] 수동 테스트 완료

## 특별히 리뷰해줄 사항
[리뷰어가 집중해야 할 부분]
```

**Claude에게 제공**:
```
다음 PR을 리뷰해줘:

PR 설명:
[위 내용 붙여넣기]

변경 내용:
[diff 또는 파일 내용]
```

## 2. 효율적인 리뷰 요청

### 2.1 전체 파일 대신 diff만 제공
- [ ] 가능한 경우 전체 파일 대신 diff를 제공했다

**diff 사용의 이점**:
- 토큰 사용량 최소화 (변경 부분만)
- 변경 컨텍스트 명확 (before/after)
- 리뷰 시간 단축

**diff 생성 방법**:
```bash
# 브랜치 간 diff
git diff main...feature-branch

# 특정 파일만
git diff main...feature-branch -- src/auth.service.ts

# 통계 포함
git diff --stat main...feature-branch
```

**예외 상황** (전체 파일 제공 필요):
- 새로 생성된 파일 (diff로는 컨텍스트 부족)
- 대규모 리팩토링 (diff가 오히려 복잡)
- 아키텍처 리뷰 (전체 구조 이해 필요)

---

### 2.2 우선순위 지정
- [ ] 리뷰 코멘트의 중요도를 명시했다 (Critical/Important/Nice-to-have)

**우선순위 분류**:

**Critical** (반드시 수정):
- 보안 취약점
- 데이터 손실 위험
- 심각한 성능 이슈
- 프로덕션 장애 가능성

**Important** (수정 권장):
- 비즈니스 로직 오류
- 테스트 누락
- 코드 품질 저하
- 유지보수성 이슈

**Nice-to-have** (선택적):
- 스타일 개선
- 네이밍 제안
- 리팩토링 아이디어
- 문서 추가

**프롬프트 예시**:
```
다음 코드를 리뷰하고, 각 이슈에 우선순위를 붙여줘:

형식:
- [Critical/Important/Nice-to-have] 이슈 설명
- 위치: [파일:라인]
- 제안: [구체적 수정 방법]

코드:
[변경 내용]
```

---

### 2.3 자동 수정 vs 제안만 구분
- [ ] 자동으로 수정해도 되는 것과 제안만 받을 것을 구분했다

**자동 수정 가능**:
- 간단한 스타일 수정 (세미콜론, 들여쓰기 등)
- 명백한 타이포 수정
- import 정리
- 간단한 리팩토링 (변수명 변경 등)

**제안만 받기**:
- 로직 변경
- 아키텍처 개선
- 성능 최적화
- API 시그니처 변경

**프롬프트 예시**:

✅ **자동 수정 요청**:
```
다음 코드의 스타일 이슈를 찾아서 즉시 수정해줘:
- ESLint 규칙 적용
- import 정리
- 네이밍 컨벤션 통일

[코드]
```

✅ **제안만 요청**:
```
다음 코드의 성능 개선점을 찾아서 제안만 해줘:
- 각 제안에 예상 효과와 트레이드오프 명시
- 구체적인 코드 예시 포함
- 우선순위 지정

[코드]
```

## 3. 리뷰 후

### 3.1 제안사항 검토 및 적용
- [ ] Claude의 제안을 검토하고 유효한 것만 선택적으로 적용했다

**검토 프로세스**:
1. **이해**: 제안의 의도와 근거 파악
2. **검증**: 제안이 실제로 개선인지 확인
3. **테스트**: 적용 후 테스트 통과 확인
4. **선택**: 유효한 제안만 적용

**주의사항**:
- ❌ 모든 제안을 무조건 수용하지 마세요
- ✅ 각 제안의 타당성을 비판적으로 평가하세요
- ✅ 프로젝트 컨텍스트에 맞는지 확인하세요
- ✅ 팀 컨벤션과 일치하는지 검토하세요

**제안 재검토 프롬프트**:
```
방금 제안한 [구체적 제안]에 대해:
- 이 변경이 필요한 이유를 다시 설명해줘
- 잠재적 부작용은 무엇인가?
- 더 나은 대안은 없는가?
```

---

### 3.2 중요 피드백은 CLAUDE.md에 반영
- [ ] 반복적으로 발견되는 패턴은 CLAUDE.md에 규칙으로 추가했다

**CLAUDE.md 반영 기준**:
- 2회 이상 반복되는 이슈
- 팀 전체에 적용 가능한 규칙
- 자동으로 검증 가능한 패턴

**예시**:

반복 이슈:
```
"모든 PR에서 에러 핸들링이 누락됨"
```

CLAUDE.md 추가:
````markdown
## 코딩 규칙

### 에러 핸들링
- 모든 async 함수는 try-catch 블록 사용
- 에러는 적절한 로그 레벨로 기록
- 사용자에게는 일반화된 에러 메시지 반환

나쁜 예:
```js
async function getUser(id) {
  return await db.users.findById(id);
}
```

좋은 예:
```js
async function getUser(id) {
  try {
    return await db.users.findById(id);
  } catch (error) {
    logger.error('Failed to get user', { id, error });
    throw new UserNotFoundError('User not found');
  }
}
```
````

---

### 3.3 반복 패턴은 Hook으로 자동화 고려
- [ ] 자동으로 검증 가능한 패턴은 Hook이나 lint 규칙으로 자동화를 고려했다

**자동화 가능한 패턴**:
- 코드 스타일 (ESLint, Prettier)
- import 정렬
- 테스트 커버리지 확인
- 민감 정보 검사 (.env, API keys)
- 타입 체크
- 라이선스 헤더

**Hook 설정 예시**:

**참고**: Hook 설정 형식은 Claude Code 공식 문서를 참고하세요. 아래는 개념적 예시입니다.

```json
// .claude/settings.json
{
  "hooks": {
    "pre-commit": {
      "tasks": [
        {
          "name": "코드 스타일 검사",
          "command": "npm run lint"
        },
        {
          "name": "민감 정보 검사",
          "prompt": "커밋할 파일에 .env, API key, 비밀번호가 포함되어 있는지 확인해줘"
        }
      ]
    }
  }
}
```

**참고**: [메인 가이드 3.3절 Hooks 설정](../sections/03-project-setup/03-3-hooks.md)

## 4. 프롬프트 템플릿

효율적인 코드 리뷰를 위한 재사용 가능한 프롬프트 템플릿입니다.

### 템플릿 1: 일반 코드 리뷰

````markdown
다음 코드 변경 사항을 리뷰해줘:

## PR 정보
- 제목: [PR 제목]
- 목적: [변경 목적]
- 영향 범위: [영향받는 기능/모듈]

## 리뷰 범위
- 초점: [보안/성능/스타일/로직/전체]
- 우선순위: [Critical/Important/Nice-to-have]

## 변경 내용
```diff
[git diff 내용 붙여넣기]
```

## 요구사항
다음 형식으로 리뷰해줘:

### Critical Issues
- [이슈 설명]
  - 위치: [파일:라인]
  - 이유: [왜 문제인가]
  - 제안: [구체적 수정 방법]

### Important Issues
- [이슈 설명]
  - 위치: [파일:라인]
  - 제안: [개선 방법]

### Nice-to-have
- [제안 사항]

### Good Practices
- [칭찬할 부분]
````

---

### 템플릿 2: 보안 중심 리뷰

````markdown
다음 코드의 보안 이슈를 집중적으로 리뷰해줘:

## 변경 내용
```diff
[diff]
```

## 보안 체크리스트
다음 항목을 중점적으로 확인해줘:
- [ ] 입력 검증 (SQL injection, XSS 등)
- [ ] 인증/권한 검증
- [ ] 민감 정보 처리 (비밀번호, 토큰 등)
- [ ] 에러 메시지에 민감 정보 노출 여부
- [ ] CORS/CSRF 설정
- [ ] 안전하지 않은 의존성

## 출력 형식
각 이슈에 대해:
- 취약점 유형: [OWASP Top 10 분류]
- 위치: [파일:라인]
- 위험도: [Critical/High/Medium/Low]
- 설명: [구체적 공격 시나리오]
- 수정 방법: [코드 예시 포함]
````

---

### 템플릿 3: 성능 중심 리뷰

````markdown
다음 코드의 성능을 리뷰해줘:

## 변경 내용
[코드 또는 diff]

## 성능 체크리스트
- [ ] 알고리즘 시간 복잡도 (O(n²) 이상 경고)
- [ ] 불필요한 반복문/중첩
- [ ] 데이터베이스 쿼리 효율성 (N+1 문제 등)
- [ ] 메모리 사용 (대용량 데이터 처리)
- [ ] 캐싱 기회
- [ ] 비동기 처리 최적화

## 출력 형식
### 성능 이슈
- 이슈: [설명]
  - 현재 복잡도: [O(n)]
  - 개선 후 복잡도: [O(log n)]
  - 예상 효과: [구체적 수치 또는 비율]
  - 코드 예시:
    ```
    [최적화된 코드]
    ```

### 최적화 기회
- 제안: [설명]
  - 예상 효과: [...]
  - 트레이드오프: [...]
````

### 템플릿 4: 리팩토링 검증

````markdown
다음 리팩토링이 적절한지 검토해줘:

## 리팩토링 목표
[리팩토링의 목적과 기대 효과]

## 변경 전 코드
```
[원본 코드]
```

## 변경 후 코드
```
[리팩토링된 코드]
```

## 검토 요청 사항
- [ ] 기능적 동등성 (behavior-preserving)
- [ ] 가독성 개선 여부
- [ ] 유지보수성 향상 여부
- [ ] 새로운 버그 가능성
- [ ] 테스트 커버리지 유지

## 출력 형식
### 검증 결과
- 기능적 동등성: [✅ 유지됨 / ❌ 변경됨]
  - [차이점 설명]

### 개선 사항
- [긍정적 변화]

### 우려 사항
- [잠재적 문제]

### 추가 제안
- [더 나은 접근법이 있다면]
````

### 템플릿 5: 테스트 코드 리뷰

````markdown
다음 테스트 코드를 리뷰해줘:

## 테스트 대상 코드
```
[테스트할 프로덕션 코드]
```

## 테스트 코드
```
[테스트 코드]
```

## 체크리스트
- [ ] 주요 경로(happy path) 커버
- [ ] 엣지 케이스 커버
- [ ] 에러 케이스 커버
- [ ] 테스트 격리 (독립적 실행 가능)
- [ ] 명확한 테스트명
- [ ] Arrange-Act-Assert 패턴
- [ ] Mock/Stub 적절성
- [ ] 테스트 가독성

## 출력 형식
### 커버리지 분석
- 커버된 케이스: [...]
- 누락된 케이스: [...]

### 테스트 품질
- Good: [잘된 부분]
- Improve: [개선 필요 부분]

### 추가 테스트 제안
- [구체적 테스트 케이스와 이유]
````

## 코드 리뷰 완료 체크리스트

리뷰 완료 전 최종 점검:

### 리뷰 품질
- [ ] 모든 Critical 이슈를 확인했는가?
- [ ] 보안 취약점을 검토했는가?
- [ ] 성능 영향을 고려했는가?
- [ ] 테스트 커버리지가 충분한가?

### 피드백 품질
- [ ] 건설적이고 구체적인가?
- [ ] 우선순위가 명확한가?
- [ ] 수정 방법을 제시했는가?
- [ ] 긍정적인 부분도 언급했는가?

### 후속 조치
- [ ] 제안사항을 검토하고 적용했는가?
- [ ] 반복 패턴을 문서화했는가?
- [ ] 자동화 가능한 부분을 식별했는가?
