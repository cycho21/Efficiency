# 문서 작성 체크리스트

이 체크리스트는 Claude를 활용한 효율적인 문서 작성 프로세스를 안내합니다.

**사용 시점**: 기술 문서, API 문서, README, 가이드 등 문서 작성 시  
**예상 소요 시간**: 준비 5-10분, 작성 20-60분 (문서 크기에 따라)

---

## 1. 문서 작성 준비

### 1.1 대상 독자 정의
- [ ] 대상 독자를 명확히 정의했다 (예: 신입 개발자, 시니어 엔지니어, 비개발자 등)

**중요한 이유**:
대상 독자에 따라 문서의 톤, 기술 깊이, 설명 수준이 달라집니다.

**예시**:
- 신입 개발자 → 기본 개념 설명, 단계별 상세 가이드
- 시니어 엔지니어 → 핵심만 간결하게, 아키텍처 중심
- 비개발자 → 전문 용어 최소화, 비유와 다이어그램 활용

**Claude에게 전달할 정보**:
```
대상 독자: [신입/중급/시니어] [개발자/PM/디자이너/경영진 등]
기술 수준: [초급/중급/고급]
배경 지식: [있음/없음] - 구체적으로 명시
```

---

### 1.2 톤 앤 매너 결정
- [ ] 문서의 톤과 스타일을 결정했다 (예: 공식적/친근한, 간결한/상세한 등)

**고려 사항**:
- **공식 문서** (API 레퍼런스, 아키텍처 문서): 객관적, 간결, 정확
- **튜토리얼** (가이드, HOW-TO): 친근, 단계별, 예시 풍부
- **내부 문서** (설계 문서, ADR): 실용적, 명확, 결정 사항 중심
- **마케팅 문서** (블로그, 소개 자료): 흥미롭고 설득력 있게

**Claude에게 전달할 정보**:
```
톤: [공식적/친근한/실용적]
스타일: [간결한/상세한/대화형]
문체: [객관적/주관적/설득적]
```

---

### 1.3 아웃라인 먼저 작성 및 승인
- [ ] 문서의 전체 구조를 아웃라인으로 작성했다
- [ ] 아웃라인을 이해관계자에게 공유하고 승인받았다 (필요시)

**아웃라인 작성 이점**:
- 전체 문서 분량과 범위 파악 가능
- 중복/누락 섹션 사전 발견
- 이해관계자와 방향성 조기 합의
- Claude가 일관된 맥락으로 작성 가능

**프롬프트 예시**:
```
다음 내용에 대한 기술 문서 아웃라인을 작성해줘:

주제: [문서 주제]
목적: [문서의 목적]
대상 독자: [독자 정보]

아웃라인 형식:
- 각 섹션 제목
- 2-3줄 설명
- 예상 분량 (짧음/보통/김)
```

**실패시 조치**:
아웃라인 없이 바로 작성하면 구조가 산만해지고, 나중에 대규모 수정이 필요할 수 있습니다. 최소 10분 투자로 2-3시간 절약 가능합니다.

## 2. 효율적인 작성

### 2.1 섹션별로 나누어 요청 (한 번에 전체 X)
- [ ] 문서를 섹션 단위로 나누어 순차적으로 작성을 요청했다

**이유**:
- 한 번에 전체 문서를 요청하면 품질 저하 위험
- 섹션별 피드백으로 방향 수정 가능
- 토큰 효율적 사용
- 각 섹션의 깊이와 스타일 조절 가능

**권장 접근법**:

❌ **비효율적**:
```
"README 파일 전체를 작성해줘 (10개 섹션 포함)"
→ 일관성 떨어짐, 수정 어려움, 토큰 낭비
```

✅ **효율적**:
```
세션 1: 아웃라인 작성 + 승인
세션 2: Introduction 섹션 작성
세션 3: Installation 섹션 작성
세션 4: Usage 섹션 작성
...
```

**실용 팁**:
- 1-3개 섹션씩 작성 요청
- 각 섹션 완료 후 간단히 검토
- 필요시 즉시 수정 요청
- 전체 완성 후 일관성 검토

---

### 2.2 템플릿 활용
- [ ] 문서 유형별 템플릿을 사용했다 (README, API 문서, ADR 등)

**템플릿 활용 이점**:
- 일관된 구조로 품질 유지
- 작성 시간 단축
- 필수 섹션 누락 방지
- 팀 전체 문서 표준화

**주요 템플릿**:
- **README**: 프로젝트 소개, 설치, 사용법, 기여 가이드
- **API 문서**: 엔드포인트, 요청/응답, 에러 코드
- **ADR** (Architecture Decision Record): 컨텍스트, 결정, 결과
- **튜토리얼**: 목표, 사전 요구사항, 단계별 가이드, 다음 단계
- **릴리즈 노트**: 새 기능, 개선, 버그 수정, 중단 변경

**Claude에게 템플릿 제공**:
```
다음 템플릿을 사용하여 [문서 유형] 작성해줘:

[템플릿 붙여넣기]

내용:
- [섹션1 내용]
- [섹션2 내용]
...
```

---

### 2.3 예시 코드는 최소한으로
- [ ] 예시 코드를 필요한 만큼만 포함했다 (과도한 예시 지양)

**이유**:
- 코드 예시는 토큰 소비가 크다
- 너무 많으면 문서가 산만해짐
- 유지보수 부담 증가 (코드 변경시 문서도 업데이트 필요)

**권장 사항**:

✅ **좋은 예시 사용**:
- 핵심 기능만 보여주는 최소 코드
- 실제 동작하는 완전한 예제
- 주석으로 설명 추가
- 1-2개 대표 예시로 충분

❌ **피해야 할 패턴**:
- 모든 함수마다 예시 추가
- 거의 비슷한 예시 반복
- 불완전한 코드 조각
- 500줄 이상의 긴 예시

**프롬프트 예시**:
```
다음 기능에 대한 예시 코드를 작성해줘:

기능: [기능 설명]
요구사항:
- 10-20줄 이내로 간결하게
- 핵심 사용법만 보여주기
- 주석으로 설명 추가
- 실행 가능한 완전한 코드
```

**실용 팁**:
- 복잡한 예시는 별도 파일(examples/)로 분리
- 문서에는 핵심 코드만, 전체 코드는 링크로 제공
- 코드 블록에 언어 명시 (syntax highlighting)

## 3. 문서 검증

### 3.1 맞춤법 검사
- [ ] 맞춤법과 문법을 검토했다

**확인 방법**:
- Claude에게 맞춤법 검사 요청
- IDE/에디터의 맞춤법 검사 도구 활용
- 팀원에게 간단한 리뷰 요청

**프롬프트 예시**:
```
다음 문서의 맞춤법과 문법을 검토해줘:

[문서 내용]

형식:
- 오류 위치: [섹션명, 라인]
- 현재: [잘못된 표현]
- 수정: [올바른 표현]
- 이유: [간단한 설명]
```

---

### 3.2 링크 유효성 확인
- [ ] 모든 내부/외부 링크가 정상 작동하는지 확인했다

**확인 항목**:
- [ ] 외부 링크 (HTTP 200 응답 확인)
- [ ] 내부 문서 링크 (파일 존재 확인)
- [ ] 앵커 링크 (섹션 ID 일치 확인)
- [ ] 이미지 링크 (이미지 표시 확인)

**자동화 도구**:
```bash
# Markdown 링크 확인
npx markdown-link-check docs/**/*.md

# 또는 수동 확인
grep -r "\[.*\](" docs/
```

**실패시 조치**:
끊어진 링크는 사용자 경험을 크게 저해합니다. 배포 전 반드시 확인하세요.

---

### 3.3 기술 정확성 검토
- [ ] 기술적 내용이 정확하고 최신 정보인지 검증했다

**검증 항목**:
- [ ] API 시그니처가 실제 코드와 일치
- [ ] 버전 정보가 정확 (라이브러리, 프레임워크 등)
- [ ] 명령어가 실제 실행 가능
- [ ] 설정 예시가 올바름
- [ ] 아키텍처 다이어그램이 현재 구조 반영

**프롬프트 예시**:
```
다음 문서 내용이 [프로젝트/코드]와 일치하는지 확인해줘:

문서 내용:
[기술적 설명 또는 코드 예시]

실제 코드:
[해당 코드 파일 내용]

불일치 사항을 리스트로 알려줘.
```

**실패시 조치**:
부정확한 문서는 신뢰를 떨어뜨리고 혼란을 야기합니다. 실제 코드와 교차 검증 필수.

---

### 3.4 코드 예시 실행 확인
- [ ] 문서에 포함된 모든 코드 예시가 실제로 실행되는지 확인했다

**확인 방법**:
1. 예시 코드를 복사하여 실제 환경에서 실행
2. 예상된 결과가 나오는지 검증
3. 에러 발생시 코드 수정 또는 설명 추가

**자동화 방법**:
```bash
# 문서에서 코드 블록 추출 및 테스트
# (doctest, 커스텀 스크립트 등 활용)
```

**베스트 프랙티스**:
- 예시 코드는 examples/ 디렉토리에도 저장
- CI/CD에서 예시 코드 실행 테스트
- 버전 변경시 예시 코드 재검증

## 4. 프롬프트 템플릿

효율적인 문서 작성을 위한 재사용 가능한 프롬프트 템플릿입니다.

### 템플릿 1: README 작성

```markdown
다음 프로젝트의 README.md를 작성해줘:

## 프로젝트 정보
- 이름: [프로젝트명]
- 목적: [프로젝트의 목적과 해결하는 문제]
- 주요 기능: [핵심 기능 3-5개]
- 기술 스택: [사용 기술]

## 대상 독자
- [신입/중급/시니어] 개발자
- [언어/프레임워크] 경험 [있음/없음]

## 톤 앤 매너
- [공식적/친근한/실용적]
- [간결한/상세한] 스타일

## 포함할 섹션
1. 프로젝트 소개 (2-3 문단)
2. 주요 기능 (bullet points)
3. 설치 방법 (단계별)
4. 빠른 시작 (최소 예시)
5. 사용 예시 (1-2개 주요 케이스)
6. 설정 옵션 (테이블 형식)
7. 기여 가이드 (간략히)
8. 라이선스

## 요구사항
- 코드 예시는 각 섹션당 1개 이하
- 설치 명령어는 실행 가능한 것만
- 전문 용어는 첫 사용시 간단히 설명
- 전체 분량 200-300줄 목표

먼저 아웃라인을 작성해줘. 승인 후 섹션별로 작성 진행.
```

---

### 템플릿 2: API 문서 작성

```markdown
다음 API 엔드포인트에 대한 문서를 작성해줘:

## API 정보
- 엔드포인트: [HTTP 메서드] [경로]
- 목적: [API의 기능]
- 인증: [필요/불필요] - [방법]

## 요청 정보
파라미터:
- [파라미터명]: [타입] - [설명] - [필수/선택]
- ...

요청 본문 (해당시):
[JSON 스키마 또는 예시]

## 응답 정보
성공 응답 (200):
[JSON 스키마 또는 예시]

에러 응답:
- [에러 코드]: [설명]
- ...

## 형식 요구사항
- 요청/응답 예시는 실제 작동하는 것만
- 파라미터는 테이블 형식으로
- 에러 코드는 모두 문서화
- cURL 예시 1개 포함

작성 순서:
1. 엔드포인트 개요
2. 파라미터 테이블
3. 요청 예시
4. 응답 예시
5. 에러 처리
6. 사용 예시 (cURL)
```

---

### 템플릿 3: 기술 가이드/튜토리얼 작성

```markdown
다음 주제에 대한 튜토리얼을 작성해줘:

## 튜토리얼 정보
- 주제: [구체적인 주제]
- 목표: [독자가 완료 후 할 수 있는 것]
- 난이도: [초급/중급/고급]
- 예상 소요 시간: [시간]

## 대상 독자
- 기술 수준: [수준]
- 사전 요구 지식: [필요한 배경 지식]
- 준비물: [필요한 도구, 환경 등]

## 구조
1. 소개 (목표와 학습 내용 요약)
2. 사전 준비 (환경 설정, 도구 설치 등)
3. 단계별 가이드:
   - 각 단계는 명확한 목표 제시
   - 코드 예시는 누적 방식 (이전 단계 위에 추가)
   - 각 단계 완료 후 검증 방법 제공
4. 문제 해결 (자주 발생하는 이슈와 해결법)
5. 다음 단계 (추가 학습 리소스)

## 작성 요구사항
- 각 단계는 5-10분 내 완료 가능하게
- 코드 예시는 복사-붙여넣기로 실행 가능
- 스크린샷은 [필요/불필요]
- 톤: 친근하고 격려하는 스타일
- 전문 용어는 설명과 함께 사용

먼저 전체 단계 아웃라인을 작성해줘.
```

### 템플릿 4: ADR (Architecture Decision Record) 작성

```markdown
다음 아키텍처 결정에 대한 ADR을 작성해줘:

## 결정 정보
- 제목: [간결한 결정 제목]
- 날짜: [결정 날짜]
- 상태: [제안됨/승인됨/거부됨/대체됨]

## 컨텍스트
문제/상황:
[어떤 문제를 해결하려고 하는가?]
[어떤 요구사항이 있는가?]
[어떤 제약사항이 있는가?]

## 고려한 옵션
1. [옵션 1]
   - 장점: [...]
   - 단점: [...]
   - 비용/복잡도: [...]

2. [옵션 2]
   - 장점: [...]
   - 단점: [...]
   - 비용/복잡도: [...]

3. [옵션 3] (선택됨)
   - 장점: [...]
   - 단점: [...]
   - 비용/복잡도: [...]

## 결정
[선택한 옵션과 이유]

## 결과
예상되는 영향:
- 긍정적: [...]

## 형식 요구사항

- 기술적 세부사항 포함
```

## 문서 작성 완료 체크리스트

작성 완료 전 최종 점검:

### 내용
- [ ] 목적이 명확하고 달성되었는가?
- [ ] 대상 독자에게 적합한 깊이와 톤인가?
- [ ] 모든 필수 섹션이 포함되었는가?
- [ ] 정보가 정확하고 최신인가?

### 품질
- [ ] 맞춤법과 문법이 올바른가?
- [ ] 링크가 모두 작동하는가?
- [ ] 코드 예시가 실행 가능한가?
- [ ] 일관된 용어와 스타일을 유지하는가?

### 사용성
- [ ] 목차가 있는가? (긴 문서의 경우)
- [ ] 섹션 구분이 명확한가?
- [ ] 시각 자료(다이어그램, 표 등)가 적절히 사용되었는가?
- [ ] 독자가 쉽게 필요한 정보를 찾을 수 있는가?

### 유지보수성
- [ ] 버전 정보가 명시되어 있는가? (해당시)
- [ ] 업데이트 날짜가 기록되어 있는가?
- [ ] 코드 예시가 별도 파일로 관리되는가? (복잡한 경우)
- [ ] 향후 업데이트가 쉬운 구조인가?
