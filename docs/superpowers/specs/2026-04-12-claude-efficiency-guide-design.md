# 클로드 효율적 이용 가이드 - 설계 문서

## 문제 정의

**How Might We**: 어떻게 하면 팀과 고급 사용자가 클로드를 사용할 때 더 나은 품질의 결과물을 더 적은 비용과 시간으로 얻을 수 있을까?

### 현재 문제점
- 토큰 낙비/과다 사용 (비효율적인 프롬프트, 불필요한 파일 읽기, 반복적인 컨텍스트 로딩)
- 베스트 프랙티스 부재 (체계적인 방법론이나 가이드 없음)
- 팀 간 일관성 부족 (각자 다른 방식으로 사용)
- 캐시 최적화 미흡 (낮은 캐시 히트율)

### 목표
1. **비용 절감**: 토큰 사용량 30-50% 감소, 캐시 히트율 70% 이상
2. **품질 향상**: 일관된 고품질 결과물 생성
3. **시간 절약**: 불필요한 반복 작업 제거, 효율적인 워크플로우
4. **표준화**: 팀 전체가 따를 수 있는 베스트 프랙티스

### 대상 사용자
- **주 타겟**: 팀/조직 전체, 고급 사용자
- **부 타겟**: 개인 사용자도 활용 가능
- **사용 시나리오**: 주로 코딩/개발, 글쓰기/문서화

### 성공 기준
- [ ] 즉시 적용 가능한 실용적인 가이드
- [ ] 측정 가능한 메트릭 포함 (토큰 사용량, 캐시 히트율)
- [ ] 나중에 자동 점검 skill로 변환 가능한 구조
- [ ] 팀 표준으로 채택 가능한 완성도

## 메인 가이드 상세 구조

### claude-efficiency-guide.md

#### 1. 시작하기 전에
**목적**: 가이드 사용법과 독자 수준별 읽는 법 안내

**내용**:
- 이 가이드를 읽는 법 (초보자 vs 고급 사용자)
- 각 섹션 개요 및 추천 읽기 순서
- 용어 정의 (토큰, 컨텍스트, 캐싱, 세션 등)
- 빠른 시작 가이드 (5분 안에 적용 가능한 핵심 3가지)

#### 2. 토큰 최적화 전략
**목적**: 비용 절감의 핵심 전략 제시

**내용**:

**2.1 기본 원칙 (11가지 규칙 확장)**
- 현재 CLAUDE.md의 11가지 규칙 각각을:
  - **왜**: 이 규칙이 중요한 이유
  - **언제**: 어떤 상황에서 적용하는지
  - **어떻게**: 구체적인 적용 방법
  - **예시**: Before/After 비교
  
예:
```markdown
규칙 1: 같은 세션에서 이미 읽은 파일을 다시 읽지 마라

왜: 
- 동일 파일 재읽기는 캐시되지만 여전히 비용 발생
- 대형 파일의 경우 수천 토큰 낭비

언제:
- 파일 내용이 변경되지 않았을 때
- 이미 컨텍스트에 정보가 있을 때

어떻게:
❌ Don't: "다시 확인해줘" → Claude가 파일을 다시 읽음
✅ Do: "위에서 읽은 user.service.ts 내용을 바탕으로..."

예시:
[실제 프롬프트 before/after]
```

**2.2 컨텍스트 캐싱 이해하기**
- 캐싱 작동 원리
  - 무엇이 캐시되는가: 시스템 프롬프트, CLAUDE.md, Tool 결과
  - 캐시 수명: 5분 (Claude API 기준)
  - 캐시 키: 정확히 동일한 내용만 히트
- 캐시 가능한 요소 목록
  - CLAUDE.md (가장 중요)
  - 자주 참조하는 설정 파일
  - 변경되지 않는 라이브러리/유틸 코드
- 캐시 무효화 조건
  - 파일 내용 변경시
  - 5분 경과시
  - 세션 종료시

**2.3 캐시 히트율 최적화 전략**
- 캐시 친화적 프로젝트 구조
  - 설정 파일 안정화 (자주 변경하지 않기)
  - 공통 코드 분리 (자주 참조되는 코드)
- 세션 관리 전략
  - 관련 작업을 묶어서 한 세션에
  - 세션 시작시 한 번에 필요한 파일 로드
- CLAUDE.md 최적화
  - 안정적인 규칙 위주로 작성
  - 자주 변경되는 내용은 별도 파일로
- 측정 및 모니터링
  - 캐시 히트율 계산법
  - 목표 기준: 70% 이상

#### 3. 프로젝트 설정
**목적**: 필수 설정 파일 및 자동화 구성

**내용**:

**3.1 CLAUDE.md 작성 가이드**
- 필수 섹션
  - 프로젝트 개요
  - 아키텍처 설명
  - 코딩 컨벤션
  - 토큰 최적화 규칙
  - 금지 사항
- 선택 섹션
  - 팀 워크플로우
  - 배포 프로세스
  - 테스팅 전략
- 작성 템플릿 제공
- 안티패턴
  - 너무 길게 작성 (2000줄 이상)
  - 너무 자주 수정 (캐시 무효화)
  - 코드 예시 과다 포함

**3.2 .claudeignore 설정**
- 필수 제외 항목
  - node_modules, .git, dist, build
  - 로그 파일, 캐시 파일
  - 바이너리, 미디어 파일
  - .env, 시크릿 파일
- 프로젝트별 패턴
  - Python: venv, __pycache__, *.pyc
  - JavaScript: node_modules, dist, coverage
  - Java: target, *.class
- 동적 업데이트 전략
  - 세션 중 불필요한 파일 발견시 즉시 추가
  - Hook으로 자동 제안

**3.3 Hooks 설정**
- Hook 종류 및 목적
  - pre-commit: 커밋 전 검증
  - post-edit: 파일 수정 후 체크
  - session-start: 세션 시작시 상태 요약
- 각 Hook 상세 예시는 `examples/hooks-examples.md` 참조
- 설정 방법 (Claude Code 기준)
- 주의사항
  - Hook은 빠르게 실행되어야 함 (< 1초)
  - 실패시 명확한 메시지

#### 4. 프롬프팅 베스트 프랙티스
**목적**: 작업 유형별 효과적인 프롬프트 작성법

**내용**:

**4.1 공통 원칙**
- 명확한 목표 명시
- 컨텍스트 최소화 (필요한 것만)
- 단계별 진행 (한 번에 너무 많이 요청하지 않기)
- 결과 형식 지정 (출력 포맷 명시)

**4.2 코딩 작업용 프롬프팅**
- 리팩토링
  - 목표와 제약 명확히
  - 변경 범위 한정
  - 테스트 유지 요구
- 버그 수정
  - 재현 단계 제공
  - 예상 vs 실제 동작 설명
  - 관련 파일만 언급
- 새 기능 추가
  - 명세 우선 작성
  - 단계별 구현 요청
  - 테스트와 함께 진행
- 코드 리뷰
  - 리뷰 범위 지정 (보안/성능/가독성 중 선택)
  - 변경 사항에만 집중
  - 자동 수정 vs 제안만 구분

**4.3 문서 작성용 프롬프팅**
- 아웃라인 먼저 승인
- 섹션별 작성 (한 번에 전체 X)
- 톤 앤 매너 지정
- 대상 독자 명시

**4.4 리팩토링/리뷰용 프롬프팅**
- 전체 스캔 vs 특정 부분
- 우선순위 지정 (critical/important/nice-to-have)
- 변경 영향도 고려 요청

**각 섹션**: 실제 템플릿은 `examples/prompt-templates.md` 참조

#### 5. 워크플로우 패턴
**목적**: 실전 시나리오별 효율적인 작업 방식

**내용**:

**5.1 개인 워크플로우**
- 일일 작업 시작
  - 세션 시작시 오늘 작업 컨텍스트 로드
  - 진행중인 작업 상태 확인
- 대규모 작업 관리
  - progress.md로 진행 상황 추적
  - 단계별 커밋으로 롤백 가능하게
- 컨텍스트 압축 타이밍
  - 60% 사용시 자동 압축
  - 중요 상태는 파일로 저장 후 압축

**5.2 팀 협업 워크플로우**
- CLAUDE.md를 팀 규칙으로
- PR 전 Claude 리뷰 활용
- 공통 프롬프트 템플릿 공유
- 토큰 사용량 모니터링 및 리뷰

**각 섹션**: 상세 시나리오는 `examples/workflow-scenarios.md` 참조

#### 6. 품질 관리
**목적**: 결과물 검증 및 일반적 실수 방지

**내용**:

**6.1 결과물 검증 방법**
- 코드 품질 체크
  - 컴파일/린트 통과 확인
  - 테스트 실행
  - 보안 취약점 스캔
- 문서 품질 체크
  - 맞춤법 검사
  - 링크 유효성
  - 기술 정확성

**6.2 일반적인 실수와 해결책**
- 실수 1: 너무 광범위한 요청
  - 증상: 불완전하거나 일반적인 답변
  - 해결: 작업을 작은 단위로 분할
- 실수 2: 컨텍스트 과다 로드
  - 증상: 높은 토큰 비용, 느린 응답
  - 해결: 필요한 파일만 선택적으로
- 실수 3: 캐시 무효화 유발
  - 증상: 낮은 캐시 히트율
  - 해결: 설정 파일 안정화
- [10개 정도의 실수 패턴]

#### 7. 측정 및 개선
**목적**: 지속적인 최적화를 위한 메트릭 추적

**내용**:

**7.1 추적할 메트릭**
- 토큰 사용량
  - 세션당 평균 토큰
  - 작업 유형별 토큰 소비
  - 월간 토큰 추이
- 캐시 히트율
  - 계산 방법: (캐시 히트 토큰) / (전체 입력 토큰)
  - 목표: 70% 이상
  - 낮은 경우 원인 분석
- 세션 효율성
  - 세션당 완료 작업 수
  - 재시도/수정 횟수
  - 평균 세션 길이

**7.2 캐시 히트율 분석 및 개선**
- 히트율 측정
  - API 로그에서 추출
  - 주간/월간 추이 분석
- 개선 전략
  - 낮은 히트율 원인 파악
    - CLAUDE.md 자주 변경?
    - 세션이 너무 짧음?
    - 파일 읽기 패턴 비효율적?
  - 타겟별 개선 방안
- Before/After 비교

**7.3 최적화 점검 체크리스트**
- 자동 체크 가능 항목 (skill 변환 대상)
  - [ ] .claudeignore 존재
  - [ ] CLAUDE.md 존재 및 기본 섹션 포함
  - [ ] 500줄 이상 파일 목록
  - [ ] 세션당 파일 재읽기 < 2회
  - [ ] 캐시 히트율 > 70%
  - [ ] Hook 설정 여부
  - [ ] 월간 토큰 사용량 추이
- 수동 체크 필요 항목
  - [ ] 프롬프트 품질
  - [ ] 컨텍스트 관련성
  - [ ] 팀 워크플로우 준수
  - [ ] 결과물 품질

#### 8. 고급 테크닉
**목적**: 숙련자를 위한 고급 활용법

**내용**:

**8.1 Subagent 활용**
- 언제 사용하는가
  - 대량 출력이 예상될 때 (20줄 이상)
  - 탐색 작업과 구현 작업 분리
  - 병렬 작업 수행
- 효과적인 활용 패턴
  - 탐색용 subagent (codebase 분석)
  - 구현용 main agent
- 주의사항
  - Subagent 과다 사용 X
  - 컨텍스트 공유 제한 인지

**8.2 컨텍스트 관리 전략**
- 세션 분할 전략
  - 관련 작업끼리 그룹화
  - 큰 작업은 여러 세션으로
- 컨텍스트 압축 타이밍
  - 자동: 60% 사용시
  - 수동: 작업 단위 전환시
- 중요 정보 보존
  - progress.md 작성
  - 핵심 결정사항 커밋 메시지로

**8.3 메모리 시스템 활용**
- Memory 타입 이해
  - user: 사용자 프로필/선호도
  - feedback: 작업 방식 피드백
  - project: 프로젝트 상태/결정
  - reference: 외부 리소스 링크
- 효과적인 메모리 저장
  - 무엇을 저장할지 (장기적 정보)
  - 무엇을 저장하지 말지 (코드, git 이력)
- 메모리 vs 파일 vs git
  - 적재적소 활용

**8.4 캐시 친화적 프로젝트 구조**
- 파일 구조 설계
  - 자주 변경되는 파일 vs 안정적인 파일 분리
  - 설정 파일 모듈화
- 세션 설계
  - 캐시를 최대한 활용하는 작업 순서
  - 워밍업 전략 (세션 시작시 공통 파일 로드)

**구조**:

```markdown
각 Hook마다:
```

**포함될 Hook**:
1. **Pre-commit Hook** (커밋 전 체크)
   - 대용량 파일 커밋 방지 (5MB 이상)
   - .env/.secret 파일 커밋 방지
   - CLAUDE.md 토큰 최적화 규칙 위반 체크
   - 코드 스타일 자동 검사

2. **Post-edit Hook** (파일 수정 후)
   - 파일 크기 경고 (500줄 이상시 분할 제안)
   - 자동 .claudeignore 업데이트 제안
   - 새로운 의존성 추가 감지

3. **Session-start Hook** (세션 시작시)
   - 프로젝트 최적화 상태 요약
   - 캐시 히트율 표시 (가능한 경우)
   - 마지막 세션 이후 변경사항 요약
   - 진행중인 작업 리마인더 (progress.md 기반)

4. **Pre-push Hook** (푸시 전)
   - 테스트 자동 실행
   - 린트 검사
   - 민감 정보 재확인

2. **문서 작성 배치 작업**
   - 문제: 50개 API 엔드포인트 문서화
   - 솔루션: 템플릿 활용, 캐싱 최적화, 배치 처리
   - 예상 절감: 70% 토큰

3. **레거시 코드 분석 및 현대화**
   - 문제: 5년된 코드베이스 이해하고 개선점 파악
   - 솔루션: 탐색/분석/실행 단계 분리
   - 예상 절감: 40% 토큰

4. **팀 협업 - 코드 리뷰 자동화**
   - 문제: PR당 수동 리뷰 30분 소요
   - 솔루션: Claude 1차 리뷰 → 사람 2차 리뷰
   - 예상 절감: 50% 시간, 안정적 품질

5. **긴급 버그 수정**
   - 문제: 프로덕션 이슈, 빠른 원인 파악 필요
   - 솔루션: 컨텍스트 최소화, 타겟 분석
   - 예상 절감: 70% 시간

6. **새 프로젝트 부트스트래핑**
   - 문제: 처음부터 최적화된 구조로 시작
   - 솔루션: 템플릿 활용, 초기 설정 자동화
   - 예상 절감: 초기 투자로 장기 절감

## 자동화 및 Skill 변환 계획

### Skill 개요

**Skill 이름**: `optimize-check` (또는 `/efficiency-audit`)

**목적**: 프로젝트의 Claude 사용 최적화 상태를 자동으로 점검하고 리포트 생성

**실행 방식**:
```bash
/optimize-check
```

### Skill 구현 설계

**Phase 1: 자동 체크 수행**
```yaml
체크 항목:
  프로젝트 설정:
    - .claudeignore 존재 여부
    - .claudeignore 내용 검증 (필수 패턴 포함)
    - CLAUDE.md 존재 여부
    - CLAUDE.md 크기 체크
    - Hook 설정 여부
  
  파일 분석:
    - 500줄 이상 파일 검색
    - 대용량 파일 감지 (5MB 이상)
    - .claudeignore 누락 파일 검색
  
  세션 분석 (가능한 경우):
    - 파일 재읽기 패턴
    - 세션 길이 분석
  
  캐싱 분석 (API 로그 접근 가능시):
    - 캐시 히트율 계산
    - CLAUDE.md 수정 빈도
```

**Phase 2: 점수 계산**
```python
카테고리별 가중치:
  - 프로젝트 설정: 25%
  - 토큰 효율성: 30%
  - 캐싱 최적화: 25%
  - 파일 관리: 20%

점수 산정:
  - 각 항목 pass/fail
  - 카테고리별 점수 합산
  - 가중치 적용하여 최종 점수 (0-100)
```
