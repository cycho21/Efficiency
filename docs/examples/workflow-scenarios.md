# 워크플로우 시나리오: 실전 사례

토큰 최적화 기법을 실제 프로젝트에 적용한 6가지 시나리오입니다.

**참고:** 이 문서의 측정 결과는 실제 프로젝트 패턴 분석을 바탕으로 한 예상 수치입니다. 실제 결과는 프로젝트 복잡도, 코드베이스 크기, 팀 숙련도에 따라 달라질 수 있습니다.

---

## 시나리오 1: 대규모 코드베이스 리팩토링

### 초기 상황
100개 파일, 10,000줄 규모의 레거시 React 컴포넌트를 ES5에서 ES6+로 마이그레이션해야 하는 상황. 전체 코드베이스를 한 번에 분석하려다 토큰 한계에 부딪힘.

### 목표
- ES5 문법을 현대적인 ES6+ 문법으로 전환
- 일관된 코드 스타일 적용
- 토큰 사용량을 최소화하며 안전하게 리팩토링

### 단계별 솔루션

1. **파일 목록 작성** - 전체 코드를 읽지 않고 glob/grep으로 리팩토링 대상 파일만 식별. `var` 키워드, `function` 선언, `.prototype` 패턴 등을 검색하여 우선순위 목록 작성.

2. **배치 분할** - 100개 파일을 10개씩 10개 배치로 분할. 각 배치는 관련성 높은 파일끼리 묶음 (예: 같은 디렉토리, 유사한 책임).

3. **배치별 리팩토링** - 각 배치마다 새 대화 시작. 해당 파일들만 읽고 변경 사항 적용. 완료 후 테스트 실행하여 검증.

4. **변경 사항 커밋** - 각 배치 완료 시 즉시 커밋. 문제 발생 시 롤백 범위를 최소화.

5. **패턴 문서화** - 첫 2-3개 배치에서 발견한 공통 패턴을 문서화. 이후 배치에서는 이 문서를 참조하여 일관성 유지.

6. **서브에이전트 활용** - 각 배치의 테스트 결과 분석을 서브에이전트에 위임. 메인 세션은 리팩토링 컨텍스트만 유지.

7. **점진적 통합** - 모든 배치 완료 후 전체 테스트 스위트 실행. 통합 이슈 발견 시 해당 파일만 재검토.

8. **최종 검증** - 변경 전후 번들 크기, 성능 메트릭 비교. 예상치 못한 회귀가 있는지 확인.

### 사용한 프롬프트 예시

1. **초기 분석**
   ```
   다음 디렉토리에서 ES5 패턴을 사용하는 파일 목록 작성:
   src/components/
   
   검색할 패턴:
   - var 키워드 사용
   - function 키워드로 된 컴포넌트
   - .prototype 사용
   
   출력: 파일 경로와 발견된 패턴 개수만 (코드는 읽지 말 것)
   ```

2. **배치 리팩토링**
   ```
   다음 10개 파일을 ES6+로 리팩토링:
   [파일 목록]
   
   변경 사항:
   - var → const/let
   - function → arrow function (적절한 곳만)
   - .prototype → class 문법
   
   각 파일 수정 후 즉시 Edit 도구 사용. 설명은 최소화.
   ```

3. **패턴 재사용**
   ```
   이전 배치에서 작성한 패턴 문서 참조:
   [문서 경로]
   
   동일한 변환 규칙을 다음 파일에 적용:
   [새 파일 목록]
   ```

### 사용한 기법/원칙

- **원칙 #3 (배치 분할로 컨텍스트 격리)**: 배치 분할로 컨텍스트 격리
- **원칙 #4 (한 번에 읽는 파일 수 제한)**: 한 번에 읽는 파일 수 제한
- **원칙 #5 (서브에이전트 활용)**: 서브에이전트로 테스트 결과 분석 위임
- **원칙 #9 (패턴 문서화)**: 배치 간 패턴 문서로 일관성 유지
- **기법 #1 (선택적 검색)**: grep으로 대상 파일만 식별
- **기법 #7 (점진적 커밋)**: 각 배치마다 즉시 커밋

### 측정 결과

- **Before**: 150K 토큰, 8시간 작업 시간
- **After**: 60K 토큰, 4시간 작업 시간
- **절감율**: 60% 토큰 절감, 50% 시간 절감

### 배운 점

- 배치 크기는 10-15개 파일이 최적. 너무 작으면 오버헤드, 너무 크면 컨텍스트 낭비.
- 첫 배치에서 패턴 문서를 작성하면 이후 배치가 훨씬 빨라짐.
- 각 배치 커밋 전 테스트 실행이 필수. 나중에 디버깅하면 원인 파악 어려움.

### 주의사항

- 파일 간 의존성이 강한 경우 배치 분할 시 주의. 관련 파일은 같은 배치에 포함.
- 50K 라인 이상 코드베이스는 배치 수가 많아져 관리 복잡도 증가. 자동화 스크립트 고려.
- 테스트 커버리지 낮은 프로젝트는 리팩토링 위험 높음. 먼저 핵심 로직 테스트 추가 권장.

## 시나리오 2: 배치 문서화 작업

### 초기 상황
50개 REST API 엔드포인트에 대한 문서가 전혀 없는 상태. 각 엔드포인트의 소스 코드, 요청/응답 스키마, 사용 예시를 분석하여 OpenAPI 문서 생성 필요.

### 목표
- 모든 엔드포인트에 대한 일관된 API 문서 작성
- 실제 코드와 동기화된 정확한 스키마 생성
- 최소한의 토큰으로 효율적인 문서화

### 단계별 솔루션

1. **엔드포인트 목록 추출** - 라우터 파일에서 grep으로 모든 엔드포인트 경로와 HTTP 메서드 추출. 전체 코드는 읽지 않음.

2. **템플릿 작성** - 첫 3-5개 엔드포인트를 수동으로 문서화하며 표준 템플릿 확립. 이후 모든 엔드포인트에 동일 구조 적용.

3. **배치 그룹핑** - 관련 엔드포인트끼리 묶음 (예: /users/* 관련 5개, /orders/* 관련 7개). 각 그룹은 공통 스키마 공유.

4. **스키마 우선 추출** - 각 배치마다 먼저 TypeScript 타입 정의나 validation 스키마만 읽음. 비즈니스 로직은 건너뜀.

5. **서브에이전트 문서화** - 각 배치를 서브에이전트에 위임. "이 5개 엔드포인트를 템플릿에 맞춰 문서화"만 지시.

6. **자동 예시 생성** - 테스트 코드에서 실제 요청/응답 예시 추출. 직접 작성하지 않고 기존 코드 활용.

7. **검증 및 병합** - 각 배치 결과를 메인 OpenAPI 파일에 병합. Swagger validator로 자동 검증.

### 사용한 프롬프트 예시

1. **엔드포인트 목록 작성**
   ```
   다음 라우터 파일들에서 모든 API 엔드포인트 추출:
   src/routes/**/*.ts
   
   출력 형식:
   - HTTP 메서드 | 경로 | 핸들러 함수명
   
   코드 내용은 읽지 말고 패턴만 추출.
   ```

2. **템플릿 확립**
   ```
   다음 3개 엔드포인트에 대한 OpenAPI 문서 작성:
   - GET /users/:id
   - POST /users
   - DELETE /users/:id
   
   각각 다음 포함:
   - 설명, 파라미터, 요청/응답 스키마, 예시, 에러 코드
   
   이 구조를 템플릿으로 저장: docs/api-template.md
   ```

3. **배치 문서화 위임**
   ```
   docs/api-template.md 템플릿 사용하여 다음 엔드포인트 문서화:
   
   /orders/* 그룹 (7개 엔드포인트)
   
   스키마는 src/types/order.ts에서 추출.
   예시는 tests/orders.test.ts에서 가져오기.
   ```

4. **자동 예시 생성**
   ```
   tests/users.test.ts에서 실제 요청/응답 페이로드만 추출:
   
   - POST /users 요청 본문
   - 200 응답 예시
   - 400 에러 예시
   
   코드 로직 설명 불필요. JSON만 출력.
   ```

### 사용한 기법/원칙

- **원칙 #3 (배치 분할로 컨텍스트 격리)**: 관련 엔드포인트 배치 처리
- **원칙 #5 (서브에이전트 활용)**: 각 배치를 서브에이전트에 위임
- **원칙 #9 (패턴 문서화)**: 표준 템플릿으로 일관성 확보
- **기법 #1 (선택적 검색)**: grep으로 엔드포인트 목록만 추출
- **기법 #3 (부분 읽기)**: 스키마 파일만 선택적으로 읽기
- **기법 #6 (코드 재사용)**: 테스트 코드에서 예시 재사용

### 측정 결과

- **Before**: 100K 토큰, 6시간 작업 시간
- **After**: 30K 토큰, 2.5시간 작업 시간
- **절감율**: 70% 토큰 절감, 58% 시간 절감

### 배운 점

- 템플릿 확립이 가장 중요. 첫 몇 개 엔드포인트에 시간 투자하면 나머지가 빠름.
- 테스트 코드는 문서 예시의 최고 소스. 실제 동작하는 코드라 정확성 보장.
- 서브에이전트 위임 시 명확한 입력(템플릿, 스키마 경로) 제공하면 품질 일관됨.

### 주의사항

- 테스트 코드 없는 엔드포인트는 예시 작성에 시간 소요. 우선순위 낮춤.
- TypeScript 사용 안 하는 프로젝트는 스키마 추출 어려움. validation 라이브러리 코드 활용.
- 엔드포인트 간 스키마 공유가 많으면 중복 문서화 주의. 공통 스키마 먼저 정의.

## 시나리오 3: 레거시 코드 분석

### 초기 상황
5년 전 작성된 10만 줄 규모의 Node.js 백엔드. 원래 개발자 퇴사, 문서 없음. 새 기능 추가 전 아키텍처와 주요 로직 이해 필요.

### 목표
- 전체 시스템 아키텍처 파악
- 핵심 비즈니스 로직 식별 및 문서화
- 기술 부채 및 리팩토링 우선순위 도출

### 단계별 솔루션

1. **구조 파악** - `tree` 명령으로 디렉토리 구조만 확인. 파일 내용은 읽지 않고 전체 조직 이해.

2. **진입점 분석** - package.json과 메인 파일(index.js, app.js)만 읽어 애플리케이션 시작 흐름 파악.

3. **의존성 그래프 작성** - grep으로 `require()`/`import` 문만 추출하여 모듈 간 의존성 맵 생성. 코드 로직은 건너뜀.

4. **핫스팟 식별** - `git log --numstat`으로 가장 자주 변경된 파일 찾기. 이 파일들이 핵심 로직일 가능성 높음.

5. **선택적 깊이 탐색** - 핵심 파일 10-15개만 상세히 읽기. 나머지는 함수 시그니처와 주석만 확인.

6. **다이어그램 생성** - 파악한 아키텍처를 Mermaid 다이어그램으로 시각화. 서브에이전트에 위임하여 생성.

7. **기술 부채 목록화** - 분석 중 발견한 안티패턴, deprecated 패키지, 보안 이슈 등을 별도 문서로 정리.

8. **질의응답 세션** - 팀원들이 궁금해할 질문 목록 작성. 각 질문에 대한 답변을 문서에 추가.

### 사용한 프롬프트 예시

1. **구조 파악**
   ```
   다음 디렉토리 구조 분석:
   [tree 출력]
   
   각 디렉토리의 역할 추측하여 요약.
   파일 내용 읽지 말 것.
   ```

2. **의존성 그래프**
   ```
   다음 파일들의 import/require 문만 추출:
   src/**/*.js
   
   출력: 모듈 A → [모듈 B, 모듈 C] 형식
   
   순환 의존성 있으면 표시.
   ```

3. **핵심 로직 분석**
   ```
   다음 파일의 주요 함수 시그니처와 역할 요약:
   src/services/payment.js
   
   각 함수:
   - 이름, 파라미터, 반환 타입
   - 1-2줄 설명
   
   구현 세부사항은 건너뜀.
   ```

4. **기술 부채 스캔**
   ```
   다음 패턴 검색:
   - TODO/FIXME 주석
   - try-catch 없는 async 함수
   - console.log (프로덕션 코드에서)
   
   발견된 위치와 간단한 컨텍스트만 출력.
   ```

### 사용한 기법/원칙

- **원칙 #1 (점진적 탐색)**: 점진적 탐색 (전체 읽지 않음)
- **원칙 #4 (한 번에 읽는 파일 수 제한)**: 핵심 파일만 상세 분석
- **원칙 #5 (서브에이전트 활용)**: 다이어그램 생성 서브에이전트 위임
- **원칙 #10 (상태 저장)**: 분석 결과를 문서로 저장
- **기법 #1 (선택적 검색)**: grep으로 패턴 검색
- **기법 #2 (Git 히스토리 활용)**: git history로 핫스팟 식별
- **기법 #9 (다이어그램 활용)**: Mermaid로 아키텍처 시각화

### 측정 결과

- **Before**: 200K 토큰, 12시간 작업 시간
- **After**: 120K 토큰, 7시간 작업 시간
- **절감율**: 40% 토큰 절감, 42% 시간 절감

### 배운 점

- 레거시 코드는 처음부터 모두 이해하려 하지 말 것. 필요한 부분만 깊게 파기.
- git history는 문서보다 정직함. 자주 변경된 파일이 진짜 핵심.
- 다이어그램은 천 마디 설명보다 효과적. 팀 공유 시 이해도 10배 향상.

### 주의사항

- git history 없는 코드베이스는 핫스팟 식별 어려움. 파일 크기와 복잡도로 대체.
- 비즈니스 도메인 지식 없으면 코드만으로 의도 파악 한계. 가능하면 기존 팀원 인터뷰.
- 대규모 모노레포는 의존성 그래프 생성에 시간 소요. 서브 디렉토리별로 분할 분석.

## 시나리오 4: 팀 코드 리뷰 자동화

### 초기 상황
하루 10-15개 PR이 생성되는 활발한 팀. 각 PR마다 수동 리뷰에 30분-1시간 소요. 리뷰어 부담 과중 및 병목 발생.

### 목표
- PR 리뷰 시간 50% 단축
- 일관된 코드 품질 기준 적용
- 리뷰어는 비즈니스 로직에만 집중

### 단계별 솔루션

1. **리뷰 체크리스트 작성** - 팀의 코딩 컨벤션, 자주 발견되는 이슈 패턴을 체크리스트로 문서화. Claude가 이를 기준으로 자동 검토.

2. **PR 변경사항만 분석** - `git diff main...feature-branch`로 변경된 부분만 가져오기. 전체 코드베이스는 읽지 않음.

3. **자동 검토 실행** - 체크리스트 기준으로 변경사항 분석. 발견된 이슈는 심각도별로 분류 (critical, warning, suggestion).

4. **컨텍스트 제한 읽기** - 변경된 함수 주변 10줄만 읽어 컨텍스트 파악. 파일 전체 읽지 않음.

5. **리뷰 코멘트 생성** - 각 이슈마다 파일명, 라인 번호, 설명, 개선 제안 포함한 구조화된 코멘트 작성.

6. **긍정적 피드백 추가** - 잘 작성된 코드에 대한 칭찬 코멘트도 포함. 개발자 동기부여.

7. **리뷰어 요약 제공** - 주요 변경사항, 발견된 이슈 수, 권장 승인 여부를 요약하여 리뷰어에게 제공. 리뷰어는 비즈니스 로직만 검토.

### 사용한 프롬프트 예시

1. **변경사항 분석**
   ```
   다음 PR 변경사항 검토:
   [git diff 출력]
   
   체크리스트:
   docs/code-review-checklist.md
   
   발견된 이슈를 심각도별로 분류.
   ```

2. **컨텍스트 파악**
   ```
   다음 함수가 변경됨:
   src/utils/format.js:45-60
   
   주변 10줄 컨텍스트:
   [코드]
   
   변경이 전체 로직에 미치는 영향 분석.
   ```

3. **리뷰 코멘트 생성**
   ```
   발견된 이슈에 대한 GitHub 코멘트 작성:
   
   형식:
   **[심각도]** 이슈 설명
   - 위치: 파일명:라인
   - 문제: ...
   - 제안: ...
   ```

4. **리뷰어 요약**
   ```
   이 PR에 대한 리뷰 요약 작성:
   
   - 주요 변경사항 (3-5 bullet points)
   - Critical 이슈: N개
   - Warning: N개
   - 권장 액션: [승인/수정 요청/재검토]
   ```

### 사용한 기법/원칙

- **원칙 #4 (한 번에 읽는 파일 수 제한)**: 변경사항만 읽기 (전체 코드 X)
- **원칙 #9 (패턴 문서화)**: 표준 체크리스트로 일관성
- **기법 #3 (부분 읽기)**: diff만 분석하여 토큰 절약
- **기법 #4 (컨텍스트 범위 제한)**: 주변 컨텍스트 제한 읽기
- **기법 #8 (템플릿 활용)**: 구조화된 리뷰 템플릿

### 측정 결과

- **Before (수동 리뷰)**: 평균 45분/PR, ~15K 토큰/PR (컨텍스트 로딩 + 분석), 리뷰어 6-8시간/일
- **After (Claude 1차 리뷰)**: 평균 20분/PR, ~8K 토큰/PR (포커스된 2차 검토), 리뷰어 3-4시간/일
- **절감율**: 56% 시간 절감, 47% 토큰 절감, 리뷰 품질 향상 (일관된 기준 적용)

### 배운 점

- 체크리스트가 구체적일수록 자동 리뷰 품질 높음. "명확한 변수명" 대신 "동사+명사 조합" 같이 구체화.
- 긍정적 피드백이 중요. 이슈만 지적하면 개발자 사기 저하.
- 리뷰어에게 요약 제공하면 중요한 부분에 집중 가능. 자동화는 보조 도구.

### 주의사항

- 비즈니스 로직의 정확성은 자동 검토 불가. 반드시 사람이 최종 검토.
- 체크리스트 과도하면 false positive 증가. 핵심 항목 10-15개가 적정.
- 새 팀원은 자동 리뷰에 의존할 수 있음. 학습 목적의 상세 리뷰는 별도 필요.

## 시나리오 5: 긴급 버그 수정

### 초기 상황
프로덕션에서 결제 처리 실패 이슈 발생. 고객 불만 증가 중. 30만 줄 코드베이스에서 원인 빠르게 찾아 수정 필요.

### 목표
- 1시간 내 원인 파악 및 수정
- 재발 방지 대책 마련
- 최소한의 코드만 읽어 빠른 해결

### 단계별 솔루션

1. **에러 로그 분석** - 프로덕션 로그에서 스택 트레이스와 에러 메시지만 추출. 관련 없는 로그는 필터링.

2. **영향 범위 역추적** - 스택 트레이스의 파일/라인만 정확히 읽기. 전체 파일 읽지 않고 해당 함수와 주변 10줄만.

3. **최근 변경사항 확인** - `git log --since="1 week ago" -- path/to/affected/file`로 최근 커밋 확인. 버그는 보통 최근 변경에서 발생.

4. **재현 조건 파악** - 에러 로그에서 요청 파라미터, 사용자 상태 등 공통 패턴 찾기. grep으로 유사 에러 검색.

5. **핫픽스 작성** - 최소 변경으로 문제 해결. 큰 리팩토링은 나중에. 변경 범위 5줄 이내 목표.

6. **테스트 케이스 추가** - 재현 조건을 테스트로 작성. 수정 전 실패, 수정 후 성공 확인.

7. **배포 및 모니터링** - 핫픽스 배포 후 에러율 모니터링. 30분간 관찰하여 해결 확인.

8. **사후 분석** - 왜 발생했는지, 어떻게 예방할지 간단히 문서화. 다음 스프린트 백로그에 근본 원인 해결 추가.

### 사용한 프롬프트 예시

1. **에러 로그 분석**
   ```
   다음 에러 로그에서 핵심 정보만 추출:
   [로그 파일]
   
   출력:
   - 에러 메시지
   - 스택 트레이스 (파일:라인)
   - 발생 빈도 및 시간대
   
   기타 로그는 무시.
   ```

2. **영향 범위 파악**
   ```
   다음 파일의 45-60줄만 읽기:
   src/services/payment.js
   
   processPayment 함수의 에러 처리 로직 확인.
   다른 함수는 무시.
   ```

3. **최근 변경사항**
   ```
   지난 1주일간 payment.js 변경사항:
   [git log 출력]
   
   각 커밋이 현재 버그와 관련 있을 가능성 평가.
   코드 전체 diff는 보지 말고 커밋 메시지만.
   ```

4. **테스트 작성**
   ```
   다음 재현 조건을 테스트 케이스로 작성:
   - 사용자: guest (비로그인)
   - 금액: 0원
   - 결제 수단: null
   
   expect(() => processPayment(params)).toThrow()
   ```

### 사용한 기법/원칙

- **원칙 #1 (점진적 탐색)**: 점진적 탐색 (에러 지점부터)
- **원칙 #4 (한 번에 읽는 파일 수 제한)**: 필요한 라인만 읽기
- **기법 #2 (Git 히스토리 활용)**: git log로 최근 변경 확인
- **기법 #4 (컨텍스트 범위 제한)**: 주변 컨텍스트 최소화
- **기법 #10 (테스트 우선)**: 즉시 테스트 작성

### 측정 결과

- **Before**: 평균 3-4시간 소요, 토큰 ~50K (전체 컨텍스트 로딩)
  - 코드 탐색: 2시간, ~30K 토큰
  - 수정: 1시간, ~15K 토큰
  - 테스트: 1시간, ~5K 토큰
- **After**: 평균 1시간 소요, 토큰 ~5K (타겟 파일만)
  - 탐색: 20분, ~2K 토큰 (에러 로그 + git blame)
  - 수정: 20분, ~2K 토큰 (최소 변경)
  - 테스트: 20분, ~1K 토큰 (재현 테스트)
- **절감율**: 70% 시간 절감, 90% 토큰 절감

### 배운 점

- 긴급 상황일수록 침착하게. 무작정 코드 읽지 말고 에러 로그부터.
- 최근 변경사항 90% 확률로 원인. git blame 적극 활용.
- 핫픽스는 최소 변경이 핵심. 리팩토링 욕구 참기.

### 주의사항

- 테스트 없이 배포 절대 금지. 긴급해도 재현 테스트는 필수.
- 핫픽스 후 반드시 사후 분석. 근본 원인 해결 안 하면 재발.
- 로그 부족한 시스템은 원인 파악 어려움. 평소 로깅 전략 중요.

## 시나리오 6: 새 프로젝트 부트스트래핑

### 초기 상황
새 마이크로서비스 프로젝트 시작. 기존 3개 서비스의 베스트 프랙티스를 참고하여 일관된 구조로 세팅 필요.

### 목표
- 일관된 프로젝트 구조 및 설정
- 기존 서비스의 검증된 패턴 재사용
- 장기적인 유지보수 비용 절감

### 단계별 솔루션

1. **패턴 추출** - 기존 3개 서비스의 공통 패턴 식별. 디렉토리 구조, 설정 파일, 미들웨어 등을 비교 분석.

2. **템플릿 생성** - 공통 패턴을 템플릿화. package.json, tsconfig.json, .eslintrc, 디렉토리 구조 등을 표준화.

3. **차이점 문서화** - 각 서비스의 고유한 설정은 별도 문서화. 새 프로젝트 특성에 맞게 선택 적용.

4. **자동 생성** - 표준 템플릿으로 초기 파일 구조 생성. 보일러플레이트 코드는 기존 서비스에서 복사.

5. **의존성 통일** - 3개 서비스의 package.json 비교하여 공통 의존성 버전 통일. 보안 취약점 체크.

6. **CI/CD 파이프라인 복제** - 기존 서비스의 GitHub Actions/Jenkins 설정 복사 후 프로젝트명만 변경.

7. **문서 템플릿 적용** - README, CONTRIBUTING, API 문서 등 표준 템플릿 적용. 프로젝트별 내용만 수정.

8. **팀 리뷰** - 생성된 구조를 팀과 리뷰. 개선 사항 반영 후 향후 프로젝트의 표준으로 확립.

### 사용한 프롬프트 예시

1. **패턴 비교**
   ```
   다음 3개 서비스의 디렉토리 구조 비교:
   - service-a/
   - service-b/
   - service-c/
   
   공통 패턴과 차이점 분석.
   파일 내용은 읽지 말고 구조만.
   ```

2. **설정 파일 통합**
   ```
   다음 파일들의 공통 설정 추출:
   - service-a/tsconfig.json
   - service-b/tsconfig.json
   - service-c/tsconfig.json
   
   통합 tsconfig.json 생성.
   각 서비스의 고유 설정은 주석으로 표시.
   ```

3. **보일러플레이트 생성**
   ```
   service-a의 다음 파일을 새 프로젝트에 맞게 수정:
   - src/middleware/logger.ts
   - src/middleware/errorHandler.ts
   - src/utils/response.ts
   
   service-a 고유 로직은 제거하고 범용적으로 변경.
   ```

4. **의존성 통일**
   ```
   3개 서비스의 package.json에서 공통 의존성 추출:
   
   각 패키지의 버전 비교하여:
   - 최신 안정 버전 권장
   - 보안 취약점 있는 버전 경고
   
   통합 package.json 생성.
   ```

### 사용한 기법/원칙

- **원칙 #3 (배치 분할로 컨텍스트 격리)**: 서비스별 배치 분석
- **원칙 #9 (패턴 문서화)**: 표준 템플릿 확립
- **원칙 #10 (상태 저장)**: 패턴 문서화로 재사용
- **기법 #1 (선택적 검색)**: 구조 비교 (내용은 나중)
- **기법 #6 (코드 재사용)**: 기존 코드 재사용

### 측정 결과

- **Before**: 새 프로젝트 세팅 2-3일 소요, 토큰 ~80K (매번 재학습)
  - 설정 조사: 1일, ~40K 토큰 (기존 프로젝트 분석)
  - 구현: 1-2일, ~40K 토큰 (반복 작업)
- **After**: 첫 프로젝트 세팅 1일, 토큰 ~40K (템플릿 작성)
  - 이후 프로젝트: 2-3시간, ~5K 토큰 (템플릿 적용)
- **절감율**: 초기 50% 절감, 장기적으로 90% 이상 절감 (토큰 & 시간)

### 배운 점

- 첫 통합 작업에 시간 투자하면 이후 모든 프로젝트가 빨라짐.
- 팀 리뷰 단계 필수. 독단적 표준화는 반발 초래.
- 완벽한 템플릿보다 80% 완성된 템플릿 + 유연성이 더 실용적.

### 주의사항

- 과도한 표준화는 프로젝트 특성 무시할 수 있음. 예외 허용 여지 필요.
- 템플릿은 주기적 업데이트 필요. 6개월마다 리뷰하여 최신 베스트 프랙티스 반영.
- 레거시 서비스의 나쁜 패턴까지 복사 주의. 각 패턴의 이유를 이해하고 선택.

## 종합 인사이트

### 공통 성공 요인

1. **점진적 접근**: 전체를 한 번에 처리하려 하지 말고 작은 단위로 분할
2. **컨텍스트 최소화**: 지금 필요한 정보만 읽고 나머지는 나중에
3. **패턴 재사용**: 한 번 확립한 패턴/템플릿을 반복 활용
4. **자동화 우선**: 반복 작업은 서브에이전트나 스크립트로 자동화
5. **문서화 습관**: 발견한 패턴과 결정 사항을 즉시 문서화

### 시나리오별 적용 기법 매트릭스

| 시나리오 | 주요 기법 | 토큰 절감율 | 시간 절감율 |
