# 프롬프트 템플릿 모음

Claude Code 작업에 즉시 활용 가능한 15개 프롬프트 템플릿. 각 템플릿은 목적, 구조, 옵션, 예시, 토큰 절약 팁, 예상 비용을 포함합니다.

---

## 코딩 템플릿 (6개)

### 1. 리팩토링 요청

#### 목적
기존 코드의 구조, 가독성, 유지보수성을 개선하면서 기능은 동일하게 유지

#### 구조

````
다음 파일을 리팩토링:
{{FILE_PATH}}

목표:
- {{GOAL_1}}
- {{GOAL_2}}
- {{GOAL_3}}

제약사항:
- {{CONSTRAINT_1}}
- {{CONSTRAINT_2}}

변경 후:
1. 주요 변경사항 요약 (3-5 bullet points)
2. Edit 도구로 파일 수정
3. 리팩토링이 기능에 영향 없음을 확인하는 테스트 명령 제안
````

#### 옵션
- `{{FILE_PATH}}`: 대상 파일 경로 (예: `src/components/Dashboard.tsx`)
- `{{GOAL_1-3}}`: 리팩토링 목표 (예: "함수 길이 50줄 이하로 분할", "중복 코드 제거")
- `{{CONSTRAINT_1-2}}`: 제약사항 (예: "공개 API 시그니처 유지", "기존 테스트 통과")

#### 예시 (채워진 템플릿)

````
다음 파일을 리팩토링:
src/utils/dataProcessor.js

목표:
- 500줄 함수를 10개 이하의 작은 함수로 분할
- 중복된 validation 로직을 별도 모듈로 추출
- ES6+ 문법 적용 (var → const/let, function → arrow)

제약사항:
- 공개 API (processData 함수 시그니처) 변경 금지
- 기존 테스트 100% 통과 필수

변경 후:
1. 주요 변경사항 요약 (3-5 bullet points)
2. Edit 도구로 파일 수정
3. 리팩토링이 기능에 영향 없음을 확인하는 테스트 명령 제안
````

#### 토큰 절약 팁
- 파일 경로만 제공하고 코드는 붙여넣지 않기 (Claude가 Read 도구로 읽음)
- 리팩토링 후 장황한 설명 요청하지 않기 ("요약만" 명시)
- 여러 파일 동시 리팩토링 시 배치로 분할 (한 번에 5-10개)

#### 예상 토큰 비용
- 입력: ~200 토큰 (프롬프트) + 파일 크기
- 출력: ~500-1,000 토큰 (요약 + 변경사항)
- 총: 중간 파일(300줄) 기준 ~2,000-3,000 토큰

### 2. 버그 수정 요청

#### 목적
재현 가능한 버그를 빠르고 안전하게 수정하며 재발 방지 테스트 추가

#### 구조

````
다음 버그 수정:

증상:
{{SYMPTOM}}

재현 단계:
1. {{STEP_1}}
2. {{STEP_2}}
3. {{STEP_3}}

예상 동작:
{{EXPECTED_BEHAVIOR}}

실제 동작:
{{ACTUAL_BEHAVIOR}}

관련 파일:
{{FILE_PATHS}}

에러 로그 (있다면):
{{ERROR_LOG}}

요구사항:
1. 원인 파악 및 최소 변경으로 수정
2. 재현 테스트 케이스 추가
3. 수정 전후 동작 차이 설명
````

#### 옵션
- `{{SYMPTOM}}`: 버그 증상 한 줄 요약
- `{{STEP_1-3}}`: 재현 단계 (구체적일수록 좋음)
- `{{EXPECTED_BEHAVIOR}}` / `{{ACTUAL_BEHAVIOR}}`: 기대 vs 실제 동작
- `{{FILE_PATHS}}`: 의심되는 파일 경로 (없으면 생략)
- `{{ERROR_LOG}}`: 스택 트레이스 또는 에러 메시지 (선택)

#### 예시 (채워진 템플릿)

````
다음 버그 수정:

증상:
결제 금액이 0원일 때 500 Internal Server Error 발생

재현 단계:
1. 비로그인 상태로 /checkout 접속
2. 쿠폰 코드 "100OFF" 입력 (100% 할인)
3. 결제하기 버튼 클릭

예상 동작:
금액이 0원이므로 결제 스킵하고 주문 완료 페이지로 이동

실제 동작:
500 에러 발생, "Cannot process payment with amount 0"

관련 파일:
src/services/payment.js
src/controllers/checkout.js

에러 로그 (있다면):
```
Error: Cannot process payment with amount 0
    at processPayment (payment.js:45)
    at checkout (checkout.js:78)
```

요구사항:
1. 원인 파악 및 최소 변경으로 수정
2. 재현 테스트 케이스 추가
3. 수정 전후 동작 차이 설명
````

#### 토큰 절약 팁
- 에러 로그는 핵심 스택 트레이스만 (전체 로그 파일 첨부 X)
- 재현 단계는 필수 정보만 (환경 세팅 등 불필요한 내용 제외)
- 관련 파일 추측 가능하면 먼저 제공 (Claude의 탐색 시간 단축)

#### 예상 토큰 비용
- 입력: ~400-600 토큰 (버그 설명 + 로그)
- 출력: ~800-1,200 토큰 (원인 분석 + 수정 + 테스트)
- 총: ~1,500-2,500 토큰

### 3. 새 기능 구현

#### 목적
명확한 요구사항을 바탕으로 새로운 기능을 체계적으로 구현

#### 구조

````
다음 기능 구현:

기능 설명:
{{FEATURE_DESCRIPTION}}

요구사항:
- {{REQUIREMENT_1}}
- {{REQUIREMENT_2}}
- {{REQUIREMENT_3}}

기술 스택:
{{TECH_STACK}}

파일 구조:
{{FILE_STRUCTURE}}

구현 단계:
1. {{IMPLEMENTATION_PLAN}} (생략 가능 - Claude가 제안)
2. 구현
3. 테스트 작성
4. 사용 예시 문서화

선택사항:
- 기존 패턴 참고: {{REFERENCE_FILE}}
- 성능 고려사항: {{PERFORMANCE_NOTE}}
````

#### 옵션
- `{{FEATURE_DESCRIPTION}}`: 기능 한 줄 요약 + 상세 설명
- `{{REQUIREMENT_1-3}}`: 명세 (기능, 비기능 요구사항)
- `{{TECH_STACK}}`: 사용 기술 (React, Express, PostgreSQL 등)
- `{{FILE_STRUCTURE}}`: 생성/수정할 파일 위치 제안
- `{{IMPLEMENTATION_PLAN}}`: 단계별 계획 (생략 시 Claude가 제안)
- `{{REFERENCE_FILE}}`: 참고할 기존 코드 (선택)
- `{{PERFORMANCE_NOTE}}`: 성능 제약사항 (선택)

#### 예시 (채워진 템플릿)

````
다음 기능 구현:

기능 설명:
사용자가 게시글에 태그를 추가/삭제하고, 태그별로 필터링할 수 있는 기능

요구사항:
- 게시글 작성/수정 시 태그 최대 5개 추가 가능
- 태그는 자동완성 지원 (기존 태그 목록에서)
- 태그 클릭 시 해당 태그 포함한 게시글만 표시
- 태그별 게시글 개수 표시

기술 스택:
React, TypeScript, React Query, Express, PostgreSQL

파일 구조:
- 프론트엔드: src/components/TagInput.tsx, src/components/TagFilter.tsx
- 백엔드: src/routes/tags.ts, src/models/Tag.ts
- 데이터베이스: migrations/add-tags-table.sql

구현 단계:
1. 구현 계획 제안 (단계별로 세분화)
2. 구현
3. 테스트 작성
4. 사용 예시 문서화

선택사항:
- 기존 패턴 참고: src/components/CategoryFilter.tsx
- 성능 고려사항: 태그 자동완성은 debounce 300ms 적용
````

#### 토큰 절약 팁
- 구현 계획 제안을 Claude에 맡기기 (직접 작성 불필요)
- 참고 파일은 경로만 제공 (코드 붙여넣기 X)
- 대규모 기능은 단계별로 분할하여 여러 세션으로 나누기

#### 예상 토큰 비용
- 입력: ~500-800 토큰 (요구사항 + 스택 정보)
- 출력: ~3,000-8,000 토큰 (계획 + 구현 + 테스트)
- 총: 중간 규모 기능 기준 ~5,000-10,000 토큰

### 4. 코드 리뷰

#### 목적
변경사항을 체계적으로 검토하여 품질, 보안, 성능 이슈 발견

#### 구조

````
다음 변경사항 코드 리뷰:

변경 범위:
{{CHANGE_SCOPE}}

리뷰 체크리스트:
{{CHECKLIST_PATH}} (또는 인라인 체크리스트)

포커스 영역:
- {{FOCUS_1}} (예: 보안, 성능, 가독성)
- {{FOCUS_2}}

리뷰 결과 형식:
1. 전반적 평가 (Good/Needs Work/Critical Issues)
2. Critical 이슈 (N개)
3. Warning (N개)
4. Suggestion (N개)
5. 잘된 점 (긍정적 피드백)

각 이슈:
- 파일:라인
- 설명
- 개선 제안 (구체적 코드 예시)
````

#### 옵션
- `{{CHANGE_SCOPE}}`: 변경 범위 (git diff, PR 번호, 파일 목록)
- `{{CHECKLIST_PATH}}`: 체크리스트 파일 경로 또는 직접 나열
- `{{FOCUS_1-2}}`: 중점 검토 영역 (선택)

#### 예시 (채워진 템플릿)

````
다음 변경사항 코드 리뷰:

변경 범위:
git diff main...feature/user-authentication

리뷰 체크리스트:
- 보안: 비밀번호 평문 저장 금지, SQL injection 방지
- 에러 처리: 모든 async 함수에 try-catch
- 테스트: 새 함수마다 단위 테스트 존재
- 코드 스타일: ESLint 규칙 준수
- 성능: N+1 쿼리 방지

포커스 영역:
- 보안 (인증 기능이므로 최우선)
- 에러 처리

리뷰 결과 형식:
1. 전반적 평가 (Good/Needs Work/Critical Issues)
2. Critical 이슈 (N개)
3. Warning (N개)
4. Suggestion (N개)
5. 잘된 점 (긍정적 피드백)

각 이슈:
- 파일:라인
- 설명
- 개선 제안 (구체적 코드 예시)
````

#### 토큰 절약 팁
- git diff 사용 시 변경사항만 포함 (전체 파일 읽기 X)
- 체크리스트는 파일로 저장 후 경로만 제공
- 대규모 PR은 파일별로 나눠서 리뷰

#### 예상 토큰 비용
- 입력: ~300 토큰 (프롬프트) + diff 크기
- 출력: ~1,000-3,000 토큰 (리뷰 결과)
- 총: 중간 PR(500줄 변경) 기준 ~3,000-6,000 토큰

### 5. 성능 최적화

#### 목적
느린 코드, 메모리 누수, 비효율적 알고리즘을 식별하고 개선

#### 구조

````
다음 코드 성능 최적화:

대상:
{{TARGET_CODE}}

성능 문제:
- 증상: {{SYMPTOM}}
- 측정 결과: {{MEASUREMENT}} (예: 응답 시간 3초, 메모리 500MB)

목표:
{{PERFORMANCE_GOAL}}

제약사항:
{{CONSTRAINTS}}

분석 및 최적화:
1. 병목 지점 식별 (프로파일링 결과 있으면 제공)
2. 최적화 방안 3개 제안 (트레이드오프 포함)
3. 권장 방안 구현
4. 성능 측정 방법 제안
````

#### 옵션
- `{{TARGET_CODE}}`: 파일 경로 또는 함수명
- `{{SYMPTOM}}`: 성능 문제 증상
- `{{MEASUREMENT}}`: 측정 결과 (응답 시간, 메모리 사용량 등)
- `{{PERFORMANCE_GOAL}}`: 목표 (예: "응답 시간 1초 이내", "메모리 100MB 이하")
- `{{CONSTRAINTS}}`: 제약사항 (예: "API 시그니처 변경 불가", "외부 라이브러리 추가 금지")

#### 예시 (채워진 템플릿)

````
다음 코드 성능 최적화:

대상:
src/services/reportGenerator.js - generateMonthlyReport 함수

성능 문제:
- 증상: 월간 리포트 생성 시 서버 응답 없음
- 측정 결과: 평균 15초 소요, 메모리 800MB 사용

목표:
응답 시간 3초 이내, 메모리 200MB 이하

제약사항:
- 리포트 형식 변경 불가 (기존 클라이언트 호환성)
- PostgreSQL 쿼리만 최적화 (Redis 등 추가 인프라 도입 불가)

분석 및 최적화:
1. 병목 지점 식별 (프로파일링 결과 있으면 제공)
2. 최적화 방안 3개 제안 (트레이드오프 포함)
3. 권장 방안 구현
4. 성능 측정 방법 제안
````

#### 토큰 절약 팁
- 프로파일링 결과가 있으면 먼저 제공 (Claude의 추측 시간 단축)
- 전체 파일 대신 느린 함수만 분석
- 최적화 후 벤치마크 코드는 Claude에게 작성 요청

#### 예상 토큰 비용
- 입력: ~400-600 토큰 (문제 설명 + 측정 결과)
- 출력: ~1,500-3,000 토큰 (분석 + 제안 + 구현)
- 총: ~2,500-5,000 토큰

### 6. 테스트 작성

#### 목적
기존 코드나 새 기능에 대한 테스트를 체계적으로 작성

#### 구조

````
다음 코드에 대한 테스트 작성:

테스트 대상:
{{TARGET_CODE}}

테스트 프레임워크:
{{TEST_FRAMEWORK}}

테스트 범위:
- {{TEST_TYPE_1}} (예: 단위 테스트, 통합 테스트, E2E)
- 커버리지 목표: {{COVERAGE_GOAL}}%

테스트 케이스:
1. {{CASE_1}} (생략 가능 - Claude가 제안)
2. {{CASE_2}}
3. Edge cases 포함

요구사항:
- 각 테스트는 독립적으로 실행 가능
- 설명적인 테스트 이름 (Given-When-Then 패턴)
- Mock/Stub 필요 시 명시
````

#### 옵션
- `{{TARGET_CODE}}`: 파일 경로 또는 함수명
- `{{TEST_FRAMEWORK}}`: Jest, Mocha, Pytest 등
- `{{TEST_TYPE_1}}`: 테스트 유형 (unit, integration, e2e)
- `{{COVERAGE_GOAL}}`: 커버리지 목표 (선택, 기본 80%)
- `{{CASE_1-2}}`: 테스트 케이스 (생략 시 Claude가 제안)

#### 예시 (채워진 템플릿)

````
다음 코드에 대한 테스트 작성:

테스트 대상:
src/utils/validator.js - validateEmail, validatePassword 함수

테스트 프레임워크:
Jest

테스트 범위:
- 단위 테스트
- 커버리지 목표: 100%

테스트 케이스:
1. validateEmail: 유효한 이메일, 잘못된 형식, null/undefined
2. validatePassword: 길이 체크, 특수문자 포함, 약한 비밀번호
3. Edge cases: 빈 문자열, 공백만 있는 문자열, 매우 긴 입력

요구사항:
- 각 테스트는 독립적으로 실행 가능
- 설명적인 테스트 이름 (Given-When-Then 패턴)
- Mock/Stub 필요 시 명시
````

1. 디렉토리 구조 및 역할
2. 주요 의존성 및 라이브러리
3. 아키텍처 패턴 (MVC, layered, microservices 등)
4. 코드 품질 지표
   - 대용량 파일 (500줄 이상)
   - 높은 복잡도 함수
   - 중복 코드
   - 테스트 커버리지 (추정)
5. 기술 부채 (TODO, FIXME, deprecated)
6. 보안 우려사항 (하드코딩된 시크릿, SQL injection 위험 등)

출력 형식:
- 요약 (1 페이지)
- 상세 분석 (섹션별)
- Mermaid 다이어그램 (아키텍처, 의존성 그래프)
- 권장 사항 (우선순위별)

제약:
- 파일 내용은 핵심 파일 15개만 읽기
- 나머지는 구조, import 문, 주석만 파악
````

#### 토큰 절약 팁
- tree 명령, git log, grep으로 먼저 메타데이터 수집
- 파일 내용 읽기는 최소화 (핵심 파일만)
- 다이어그램 생성은 서브에이전트에 위임

#### 예상 토큰 비용
- 입력: ~1,000-2,000 토큰 (메타데이터 + 핵심 파일)
- 출력: ~3,000-6,000 토큰 (분석 리포트)
- 총: 중간 규모 프로젝트 기준 ~5,000-10,000 토큰

````

{{SYSTEM_OVERVIEW}}

{{CURRENT_ARCHITECTURE}} (다이어그램, 설명 문서 경로)

리뷰 초점:
- {{FOCUS_1}} (예: 확장성, 성능, 보안, 유지보수성)
- {{FOCUS_2}}

평가 기준:
1. 설계 원칙 준수 (SOLID, DRY, KISS 등)
2. 확장성 (수평/수직 확장 가능성)
3. 성능 병목 지점
4. 단일 장애 지점 (SPOF)

7. 비용 효율성

출력:
- 현재 아키텍처 평가 (각 항목별 점수 + 설명)

````

- `{{SYSTEM_OVERVIEW}}`: 시스템 개요 (목적, 규모, 사용자 수 등)

### 15. 의존성 분석

#### 목적
프로젝트 의존성의 최신성, 보안, 라이선스, 사용 여부 분석

#### 구조

````
다음 프로젝트의 의존성 분석:

의존성 파일:
{{DEPENDENCY_FILE}} (package.json, requirements.txt, Gemfile 등)

분석 항목:
1. 버전 상태
   - 오래된 패키지 (1년 이상)
   - 메이저 버전 업데이트 가능한 패키지
   - deprecated 패키지
2. 보안 취약점
   - {{AUDIT_COMMAND}} 결과 (예: npm audit)
   - CVE 있는 패키지
3. 라이선스 검토
   - 상업적 사용 제한 (GPL 등)
   - 라이선스 충돌
4. 사용 여부
   - 설치되었으나 사용 안 하는 패키지
   - devDependencies vs dependencies 분류 오류
5. 번들 크기 영향 (프론트엔드만)

출력:
- 요약 (총 N개 패키지, 업데이트 필요 M개)
- 우선순위별 액션 아이템
  - 즉시 업데이트 (보안)
  - 검토 후 업데이트 (메이저 버전)
  - 제거 가능 (미사용)
- 업데이트 스크립트 제안

제약:
{{CONSTRAINTS}}
````

#### 옵션
- `{{DEPENDENCY_FILE}}`: 의존성 파일 경로
- `{{AUDIT_COMMAND}}`: audit 명령 (npm audit, pip-audit 등)
- `{{CONSTRAINTS}}`: 제약사항 (예: "Python 3.8 호환성 유지")

#### 예시 (채워진 템플릿)

````

의존성 파일:
package.json, package-lock.json

분석 항목:
1. 버전 상태
   - 오래된 패키지 (1년 이상)
   - 메이저 버전 업데이트 가능한 패키지
   - deprecated 패키지
2. 보안 취약점
   - npm audit 결과 (제공됨)
   - CVE 있는 패키지
3. 라이선스 검토
   - 상업적 사용 제한 (GPL 등)
   - 라이선스 충돌
4. 사용 여부
   - 설치되었으나 사용 안 하는 패키지 (grep import 문으로 확인)
   - devDependencies vs dependencies 분류 오류
5. 번들 크기 영향 (webpack-bundle-analyzer 결과 참고)

출력:
- 요약 (총 N개 패키지, 업데이트 필요 M개)
- 우선순위별 액션 아이템
  - 즉시 업데이트 (보안)
  - 검토 후 업데이트 (메이저 버전)
  - 제거 가능 (미사용)
- 업데이트 스크립트 제안

- `npm audit`, `npm outdated` 결과를 먼저 제공

- 라이선스는 `license-checker` 도구 결과 활용

#### 예상 토큰 비용
- 입력: ~500-1,000 토큰 (파일 + audit 결과)

## 템플릿 활용 가이드

### 사용 방법

1. **적절한 템플릿 선택**: 작업 유형에 맞는 템플릿 찾기
2. **플레이스홀더 채우기**: `{{변수}}` 부분을 실제 정보로 교체
3. **옵션 조정**: 필요에 따라 섹션 추가/제거
4. **Claude에 입력**: 완성된 프롬프트를 Claude Code에 입력

### 토큰 최적화 공통 팁

1. **파일 경로만 제공**: 코드는 Claude가 Read 도구로 읽도록
2. **배치 처리**: 유사한 작업 여러 개는 함께 처리
3. **출력 제한**: "요약만", "주요 내용만" 등 명시
4. **재사용**: 한 번 작성한 템플릿/패턴 재활용
5. **점진적 접근**: 대규모 작업은 단계별로 분할

### 템플릿 커스터마이징

각 프로젝트 특성에 맞게 템플릿 수정:

- 팀 코딩 컨벤션 추가
- 자주 사용하는 도구/프레임워크 명시
- 체크리스트를 별도 파일로 저장 후 경로 참조
- 프로젝트 고유 제약사항 포함

### 다음 단계

1. 가장 자주 하는 작업 3개 선택
2. 해당 템플릿을 팀 위키나 .claude/ 폴더에 저장
3. 실제 사용하며 개선 (플레이스홀더 추가, 옵션 조정)
4. 팀원과 공유하여 일관성 확보
