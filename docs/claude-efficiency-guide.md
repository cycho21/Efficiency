### 1.2 용어 정의

Claude를 효율적으로 사용하기 위해 알아야 할 핵심 용어들입니다.

#### 토큰 (Token)

**정의**: Claude가 텍스트를 처리하는 기본 단위입니다. 대략 4자 = 1토큰으로 계산됩니다.

**예시**:
- "안녕하세요" = 약 3-4 토큰
- "Hello, how are you?" = 약 5-6 토큰
- 1,000줄의 코드 파일 = 약 3,000-5,000 토큰

**왜 중요한가**: 
- API 비용은 토큰 수에 비례합니다
- 토큰을 절약하면 비용이 직접적으로 감소합니다
- 불필요한 토큰 사용은 응답 속도도 느리게 만듭니다

#### 컨텍스트 (Context)

**정의**: Claude가 현재 대화에서 참조할 수 있는 모든 정보입니다.

**포함되는 것**:
- 시스템 프롬프트 (CLAUDE.md 등)
- 대화 히스토리
- 읽은 파일 내용
- Tool 실행 결과

**컨텍스트 윈도우**: 
- Claude Sonnet 3.5: 최대 200,000 토큰
- 초과하면 오래된 정보부터 제거됩니다

**최적화 포인트**: 필요한 정보만 컨텍스트에 유지하는 것이 핵심입니다.

#### 캐싱 (Caching)

**정의**: 반복적으로 사용되는 정보를 저장해서 재사용하는 메커니즘입니다.

**캐시되는 것**:
- CLAUDE.md 같은 시스템 프롬프트
- 변경되지 않은 파일 내용
- 이전 대화의 일부

**캐시의 이점**:
- 캐시된 토큰은 비용이 90% 절감됩니다
- 응답 속도가 빨라집니다
- 같은 정보를 반복 전송하지 않아도 됩니다

**캐시 수명**: 
- 5분간 유지됩니다 (Claude API 기준)
- 5분 내에 다시 사용하면 연장됩니다

**예시**:
```
첫 번째 요청: CLAUDE.md 읽기 (2,000 토큰)
→ 비용: 2,000 토큰 전체

두 번째 요청 (5분 이내): 같은 CLAUDE.md 참조
→ 비용: 200 토큰 (90% 할인)

절감: 1,800 토큰 = 90%
```

#### 세션 (Session)

**정의**: Claude와의 연속된 대화 흐름입니다.

**세션 관리의 중요성**:
- 같은 세션에서 파일을 한 번만 읽으면 캐싱 효과가 극대화됩니다
- 관련 작업을 묶어서 한 세션에 수행하면 효율적입니다
- 너무 긴 세션은 컨텍스트가 흐려질 수 있습니다

**좋은 세션 예시**:
- "user.service.ts 리팩토링" 전체를 한 세션에
- 관련된 3-4개 파일을 함께 작업

**나쁜 세션 예시**:
- 매번 새 세션을 시작해서 캐시를 활용 못함
- 관련 없는 20개 작업을 한 세션에 몰아넣기

#### 캐시 히트율 (Cache Hit Rate)

**정의**: 전체 입력 토큰 중 캐시에서 재사용된 토큰의 비율입니다.

**계산 방법**:
```
캐시 히트율 = (캐시 히트 토큰 수) / (전체 입력 토큰 수) × 100%
```

**목표 기준**:
- 70% 이상: 우수
- 50-70%: 양호
- 50% 미만: 개선 필요

**예시**:
```
전체 입력: 10,000 토큰
캐시 히트: 7,000 토큰
캐시 히트율 = 7,000 / 10,000 = 70% ✅
```

**측정 방법**:

Claude API를 사용하는 경우, API 응답에서 캐시 메트릭을 확인할 수 있습니다:

```json
{
  "usage": {
    "input_tokens": 10000,
    "cache_creation_input_tokens": 3000,
    "cache_read_input_tokens": 7000
  }
}
```

Claude Code CLI를 사용하는 경우:
- 현재 자동 측정 기능은 제공되지 않습니다
- 수동 계산: 세션 중 반복되는 프롬프트/파일의 토큰 수를 추정하여 계산

**히트율이 낮은 경우**:
- CLAUDE.md를 너무 자주 수정함
- 세션이 너무 짧아서 캐시가 재사용되기 전에 만료됨
- 매번 다른 파일을 읽어서 캐싱 효과가 없음

### 1.3 빠른 시작 가이드

바쁜 분들을 위해, 5분 안에 즉시 적용 가능한 핵심 3가지를 소개합니다.

#### ⚡ 액션 1: .claudeignore 파일 생성 (2분)

**왜 중요한가**: 
불필요한 파일을 제외하면 토큰 낭비를 막고 Claude가 중요한 파일에 집중할 수 있습니다.

**즉시 실행**:

프로젝트 루트에 `.claudeignore` 파일을 만들고 다음 내용을 추가하세요:

```
# 의존성
node_modules/
vendor/
venv/
__pycache__/

# 빌드 결과물
dist/
build/
*.min.js
*.min.css

# 로그 및 임시 파일
*.log
.DS_Store
*.tmp

# 환경 설정 (보안)
.env
.env.local
*.key
*.pem

# Git
.git/

# IDE
.vscode/
.idea/
```

**예상 효과**: 
- 즉시 20-30% 토큰 절감
- 파일 읽기 속도 향상
- 보안 강화 (민감 파일 제외)

---

#### ⚡ 액션 2: 같은 세션에서 파일 재읽기 방지 (1분)

**왜 중요한가**: 
같은 세션에서 이미 읽은 파일을 다시 읽으면 캐시되더라도 비용이 발생합니다.

**즉시 적용**:

❌ **하지 말 것**:
```
사용자: "user.service.ts를 읽고 분석해줘"
Claude: [파일을 읽고 분석]

사용자: "다시 확인해줘"
Claude: [파일을 또 읽음] ← 불필요한 토큰 소비
```

✅ **이렇게 하세요**:
```
사용자: "user.service.ts를 읽고 분석해줘"
Claude: [파일을 읽고 분석]

사용자: "위에서 읽은 내용을 바탕으로 리팩토링 제안해줘"
Claude: [이미 가진 정보로 작업] ← 토큰 절약
```

**핵심 원칙**:
- "위에서 본", "방금 읽은", "앞서 분석한" 같은 표현 사용
- 명시적으로 "다시 읽어줘"라고 요청하지 않기
- Claude가 정보를 기억하고 있다고 신뢰하기

**예상 효과**: 
- 세션당 10-20% 토큰 절감
- 응답 속도 향상

---

#### ⚡ 액션 3: CLAUDE.md 기본 버전 생성 (2분)

**왜 중요한가**: 
CLAUDE.md는 프로젝트별 컨텍스트를 제공하고, 캐싱되어 모든 세션에서 재사용됩니다.

**즉시 실행**:

프로젝트 루트에 `.claude/CLAUDE.md` 파일을 만들고 다음 템플릿을 채우세요:

```markdown

- 목적: [한 줄 설명]

## 코딩 컨벤션
- 언어: [JavaScript/TypeScript/Python 등]
- 스타일: [Prettier, ESLint 등 사용 여부]
- 네이밍: [camelCase, snake_case 등]

## 금지 사항
- 이모지 사용하지 말 것
- 커밋 메시지는 한글로
- 테스트 없이 커밋하지 말 것

## 토큰 최적화 규칙
1. 같은 세션에서 파일 재읽기 금지
2. .claudeignore 적극 활용
```

**예상 효과**:

- 캐싱으로 장기적 토큰 절감

## 2. 토큰 최적화 전략

### 2.1 기본 원칙 (11가지 규칙)

**플랫폼 참고사항**: 
이 가이드는 Claude Code CLI와 Claude API 모두에 적용됩니다. 대부분의 규칙은 공통으로 적용되지만, 일부 기능은 플랫폼별로 다를 수 있습니다:
- `.claudeignore`, 서브에이전트: Claude Code CLI 전용 기능
- 프롬프트 캐싱: Claude API와 Claude Code 모두 지원 (구현 방식 다름)
- 컨텍스트 관리, 토큰 최적화 원칙: 모든 플랫폼 공통

---

이 섹션은 `.claude/CLAUDE.md`에 명시된 11가지 토큰 최적화 규칙을 상세히 확장합니다. 각 규칙마다 왜 중요한지, 언제 적용하는지, 어떻게 하는지, 그리고 실제 예시를 제공합니다.

**메트릭 해석 가이드**:
예시에서 사용되는 측정 단위:
- **토큰 (tokens)**: API 비용의 직접적 기준. 입력/출력 모두 포함
- **컨텍스트 사용률 (%)**: 200K 토큰 윈도우 중 사용 중인 비율 (예: 40,000 토큰 사용 = 20%)
- **절감률 (%)**: 최적화 전후 비교 (예: 6,000 → 2,000 토큰 = 67% 절감)

---

#### 규칙 1: 같은 세션에서 이미 읽은 파일을 다시 읽지 마라

##### 왜 (Why)

**중요한 구분**: 이 규칙은 두 가지 시나리오를 다룹니다:

**시나리오 A - 도구 호출 자체를 피하기** (최선):
- Claude가 이미 컨텍스트에 있는 정보를 활용
- 파일 재읽기 도구를 전혀 호출하지 않음
- 토큰 비용: **0 토큰** (도구 호출 없음)

**시나리오 B - 불가피한 재읽기** (차선):
- 파일을 다시 읽어야 하지만 캐시가 적용됨
- 도구를 호출하지만 캐시된 내용 사용
- 토큰 비용: **원래 비용의 10%** (90% 할인)

**실제 영향**:

시나리오 A (도구 호출 회피):
- 500줄 파일을 컨텍스트 정보로 처리: 0 토큰
- 하루 5번 회피: 0 토큰
- **최대 절감 효과**

시나리오 B (캐시된 재읽기):
- 500줄 파일 재읽기 (캐시 적용): 약 150-200 토큰 (원래 1,500-2,000의 10%)
- 하루 5번 재읽기: 750-1,000 토큰
- **90% 절감 (하지만 시나리오 A보다는 비효율적)**

**부가 문제**:
- **속도 저하**: 불필요한 파일 I/O로 응답 시간 증가
- **컨텍스트 오염**: 이미 가진 정보를 반복해서 컨텍스트에 추가하면 중요한 정보가 밀려날 수 있음

##### 언제 (When)

**적용해야 할 때**:
- ✅ 파일 내용이 변경되지 않았을 때
- ✅ 이미 컨텍스트에 파일 정보가 있을 때
- ✅ Claude가 방금 전에 해당 파일을 분석했을 때
- ✅ 같은 세션 내에서 연관 작업을 진행할 때

**예외 상황 (재읽기 필요)**:
- ❌ 파일을 직접 수정한 후 최신 내용 확인이 필요할 때
- ❌ 세션이 너무 길어져서 초기 파일 내용이 컨텍스트에서 밀려났을 때
- ❌ Claude가 특정 부분의 정확한 내용을 다시 확인해야 할 때

##### 어떻게 (How)

**방법 1: 명시적으로 이전 내용 참조**

❌ **Don't**:
```
사용자: "user.service.ts를 읽고 분석해줘"
Claude: [파일 읽고 분석 완료]

사용자: "다시 확인해서 개선점 알려줘"
Claude: [파일을 다시 읽음] ← 불필요
```

✅ **Do**:
```
사용자: "user.service.ts를 읽고 분석해줘"
Claude: [파일 읽고 분석 완료]

사용자: "위에서 분석한 내용을 바탕으로 개선점 알려줘"
Claude: [이미 가진 정보 활용] ← 효율적
```

**방법 2: 필요한 부분만 재확인**

❌ **Don't**:
```
사용자: "17번 줄이 정확히 뭐였지? 파일 다시 읽어줘"
Claude: [전체 파일 재읽기]
```

✅ **Do**:
```
사용자: "방금 본 user.service.ts의 17번 줄 내용이 뭐였지?"
Claude: [기억에서 답변]

또는

사용자: "user.service.ts의 15-20번 줄만 다시 확인해줘"
Claude: [필요한 부분만 읽기]
```

**방법 3: Claude에게 신뢰 표현**

✅ **효과적인 표현**:
- "앞서 읽은 파일에서..."
- "방금 분석한 내용 중에..."
- "위에서 본 코드를 바탕으로..."
- "이미 가진 정보로..."

##### 예시 (Example)

**시나리오**: 대형 서비스 파일 리팩토링

**Before (비효율적)**:
```
사용자: "user.service.ts 읽고 분석해줘"
Claude: [파일 읽기 - 2,000 토큰]

사용자: "이제 리팩토링 계획 세워줘. 파일 다시 확인 필요하면 읽어"
Claude: [파일 다시 읽기 - 2,000 토큰]

사용자: "계획대로 구현해줘. 원본 파일 확인하고"
Claude: [파일 또 읽기 - 2,000 토큰]

총 토큰: 6,000 토큰
```

**After (효율적 - 시나리오 A: 도구 호출 회피)**:
```
사용자: "user.service.ts 읽고 분석해줘"
Claude: [파일 읽기 - 2,000 토큰]

사용자: "위에서 분석한 내용으로 리팩토링 계획 세워줘"
Claude: [기존 컨텍스트 활용, 도구 호출 없음 - 0 토큰]

사용자: "계획대로 구현해줘. 방금 읽은 원본 기반으로"
Claude: [기존 컨텍스트 활용, 도구 호출 없음 - 0 토큰]

총 토큰: 2,000 토큰
절감: 4,000 토큰 (67% 감소) ✅
```

**만약 재읽기가 불가피하다면 (시나리오 B)**:
```
총 토큰: 2,000 (초기) + 200 (캐시 재읽기) + 200 (캐시 재읽기) = 2,400 토큰
절감: 3,600 토큰 (60% 감소)
여전히 좋지만 시나리오 A보다는 비효율적
```

**측정 가능한 결과**:
- 토큰 절감 (시나리오 A): 67%
- 토큰 절감 (시나리오 B): 60%
- 시간 절약: 파일 I/O 2번 제거로 응답 속도 향상
- 컨텍스트 효율: 중복 정보 제거로 더 많은 작업 공간 확보

**핵심**: 가능한 한 시나리오 A (도구 호출 자체 회피)를 추구하되, 불가피할 경우 시나리오 B (캐시 활용)도 여전히 큰 이득입니다.

#### 규칙 2: 불필요한 도구 호출을 피하라

##### 왜 (Why)

- **도구 호출 오버헤드**: 각 도구 호출마다 입출력 토큰이 발생합니다
- **컨텍스트 증가**: 도구 결과가 컨텍스트에 누적됩니다
- **응답 지연**: 불필요한 파일 읽기, 검색 등으로 시간 소요
- **복잡도 증가**: 도구 호출이 많을수록 에러 가능성 증가

**실제 영향**:
- 불필요한 파일 읽기 1번: 500-3,000 토큰
- 불필요한 Grep 1번: 100-500 토큰
- 하루 10번 불필요한 도구 호출: 5,000-15,000 토큰 낭비

##### 언제 (When)

**도구 호출을 피해야 할 때**:
- ✅ 사용자가 이미 충분한 정보를 제공했을 때
- ✅ 컨텍스트에 이미 필요한 정보가 있을 때
- ✅ 추측/추론으로 충분한 경우
- ✅ 도구 결과가 실제 작업에 사용되지 않을 때

**도구 호출이 필요한 때**:
- ❌ 정확한 파일 내용 확인이 필수적일 때
- ❌ 코드 변경 전 현재 상태 확인이 필요할 때
- ❌ 사용자가 명시적으로 요청했을 때

##### 어떻게 (How)

**방법 1: 도구 호출 전 필요성 검증**

❌ **Don't**:
```
사용자: "React 컴포넌트 패턴 설명해줘"
Claude: [프로젝트 파일 검색, 읽기...] ← 불필요
```

✅ **Do**:
```
사용자: "React 컴포넌트 패턴 설명해줘"
Claude: [일반적인 지식으로 설명] ← 도구 호출 없음
```

**방법 2: 사용자 제공 정보 활용**

❌ **Don't**:
```
사용자: "이 코드 리팩토링해줘:
function foo() { /* 코드 */ }"

Claude: "파일 경로 알려주시면 전체 파일을 읽고..." ← 불필요
```

✅ **Do**:
```
사용자: "이 코드 리팩토링해줘:
function foo() { /* 코드 */ }"

Claude: "제공하신 코드를 바로 리팩토링하겠습니다..." ← 즉시 작업
```

**방법 3: 필요한 도구만 선택적 사용**

❌ **Don't**:
```
Claude 생각: "리팩토링을 위해..."
1. Glob으로 모든 파일 찾기
2. Grep으로 패턴 검색
3. 관련 파일 모두 읽기
4. 리팩토링 수행

[5개 도구 호출]
```

✅ **Do**:
```
Claude 생각: "사용자가 이미 파일 경로를 제공했으므로..."
1. 해당 파일만 읽기
2. 리팩토링 수행

[1개 도구 호출]
```

##### 예시 (Example)

**시나리오**: 코드 리뷰 요청

**Before (비효율적)**:
```
사용자: "PR #123 리뷰해줘. 변경 파일은 user.service.ts랑 user.test.ts야"

Claude: 
1. Glob으로 전체 파일 검색 - 500 토큰
2. Grep으로 관련 파일 찾기 - 300 토큰
3. Git diff 확인 - 200 토큰
4. user.service.ts 읽기 - 2,000 토큰
5. user.test.ts 읽기 - 1,500 토큰
6. 관련 파일 3개 더 읽기 - 3,000 토큰

총 토큰: 7,500 토큰
```

**After (효율적)**:
```
사용자: "PR #123 리뷰해줘. 변경 파일은 user.service.ts랑 user.test.ts야"

Claude:
1. user.service.ts 읽기 - 2,000 토큰
2. user.test.ts 읽기 - 1,500 토큰
3. 리뷰 수행 - 0 토큰 (도구 호출 아님)

총 토큰: 3,500 토큰
절감: 4,000 토큰 (53% 감소) ✅
```

**핵심 원칙**:
- 사용자가 제공한 정보를 신뢰하라
- "확인 차원"의 불필요한 도구 호출을 피하라
- 필요한 것만, 한 번만 호출하라

#### 규칙 3: 가능한 경우 병렬 도구 호출을 실행하라

##### 왜 (Why)

- **시간 절약**: 여러 도구를 동시에 실행하면 순차 실행 대비 시간 단축
- **효율성**: 독립적인 작업을 묶어서 처리하면 컨텍스트 전환 최소화
- **토큰 최적화**: 한 번의 턴에서 여러 작업을 완료하므로 중복 설명 제거
- **사용자 경험**: 빠른 응답으로 더 나은 경험 제공

**실제 영향**:
- 순차 실행: 3개 파일 읽기 = 3턴 = 중복 설명 토큰 발생
- 병렬 실행: 3개 파일 읽기 = 1턴 = 설명 1번만
- 시간 절약: 50-70% 단축 가능

##### 언제 (When)

**병렬 실행이 가능한 경우**:
- ✅ 여러 파일을 동시에 읽어야 할 때
- ✅ 서로 독립적인 검색/분석 작업
- ✅ 의존성이 없는 여러 검증 작업
- ✅ 여러 파일에 동일한 변경 적용

**순차 실행이 필요한 경우**:
- ❌ 이전 도구 결과에 따라 다음 도구가 결정될 때
- ❌ 파일을 읽고 내용에 따라 다른 파일을 선택해야 할 때
- ❌ 순서가 중요한 작업 (예: 테스트 → 결과 확인)

##### 어떻게 (How)

**방법 1: 독립적인 파일 읽기를 병렬로**

❌ **Don't (순차 실행)**:
```
Claude:
1. user.service.ts 읽기
   [결과 대기]
2. user.controller.ts 읽기
   [결과 대기]
3. user.dto.ts 읽기
   [결과 대기]

총 시간: T1 + T2 + T3
```

✅ **Do (병렬 실행)**:
```
Claude:
1. user.service.ts, user.controller.ts, user.dto.ts 동시 읽기
   [모든 결과 한 번에 대기]

총 시간: max(T1, T2, T3) ≈ 33-50% 시간 절감
```

**방법 2: 검색과 읽기 병렬화**

✅ **효율적인 패턴**:
```
Claude:
// 동시에 실행
1. Grep으로 "User" 패턴 검색
2. Glob으로 "*.test.ts" 파일 찾기
3. package.json 읽기

[모든 결과를 받아서 종합 분석]
```

**방법 3: 여러 파일 수정 병렬화**

✅ **효율적인 패턴**:
```
Claude:
// 여러 파일에 동일한 패턴 적용시
1. file1.ts 수정 (Edit)
2. file2.ts 수정 (Edit)
3. file3.ts 수정 (Edit)

[동시 실행으로 빠른 완료]
```

##### 예시 (Example)

**시나리오**: 새 기능 구현 전 관련 파일 분석

**Before (순차 실행 - 비효율적)**:
```
Claude 내부 실행:
Turn 1: user.service.ts 읽기
  - 읽기: 1초
  - 설명: "user.service.ts를 읽었습니다..."
  
Turn 2: user.controller.ts 읽기
  - 읽기: 1초
  - 설명: "user.controller.ts를 읽었습니다..."
  
Turn 3: user.dto.ts 읽기
  - 읽기: 1초
  - 설명: "user.dto.ts를 읽었습니다..."

Turn 4: 종합 분석
  - 설명: "세 파일을 분석한 결과..."

총 시간: 약 3-4초
총 토큰: 읽기 토큰 + 설명 토큰 × 3
```

**After (병렬 실행 - 효율적)**:
```
Claude 내부 실행:
Turn 1: user.service.ts, user.controller.ts, user.dto.ts 동시 읽기
  - 읽기: 약 1초 (병렬)
  - 설명: "세 파일을 읽고 분석한 결과..." (한 번만)

총 시간: 약 1-1.5초 (60% 단축) ✅
총 토큰: 읽기 토큰 + 설명 토큰 × 1 (중복 제거) ✅
```

**측정 가능한 결과**:
- 시간 절약: 60-70%
- 토큰 절감: 설명 중복 제거로 20-30%
- 사용자 경험: 즉시성 향상

**실제 코드 예시**:

```typescript
// ❌ 순차 실행
await readFile('file1.ts');
await readFile('file2.ts');
await readFile('file3.ts');

// ✅ 병렬 실행
await Promise.all([
  readFile('file1.ts'),
  readFile('file2.ts'),
  readFile('file3.ts')
]);
```

#### 규칙 4: 대량 출력(20줄 이상)은 서브에이전트에게 위임하라

**서브에이전트란?**

서브에이전트(Subagent)는 Claude Code CLI에서 제공하는 기능으로, 독립적인 작업을 별도의 컨텍스트에서 실행할 수 있게 합니다. 

**주요 특징**:
- 메인 세션과 분리된 독립 컨텍스트
- 작업 완료 후 요약만 메인으로 전달
- 여러 서브에이전트를 병렬로 실행 가능
- 메인 컨텍스트 보호 및 절약

**사용 방법** (Claude Code CLI):
```
사용자: "대량 분석 작업이 필요한데, 서브에이전트 사용해줘"
Claude: [Agent tool을 사용하여 서브에이전트 생성 및 작업 위임]
```

또는 Claude가 자동으로 판단:
```
Claude: "이 작업은 출력량이 많을 것으로 예상됩니다. 
서브에이전트에게 위임하고 요약만 받겠습니다."
[서브에이전트 자동 실행]
```

**참고**: 이 기능은 Claude Code CLI 전용입니다. Claude API를 직접 사용하는 경우, 별도 세션으로 작업을 분리하여 유사한 효과를 얻을 수 있습니다.

---

##### 왜 (Why)

- **메인 컨텍스트 보호**: 큰 출력은 메인 세션의 컨텍스트를 빠르게 소진시킵니다
- **작업 분리**: 탐색 작업과 구현 작업을 분리하면 각각에 집중 가능
- **효율성**: 서브에이전트는 완료 후 요약만 전달하므로 메인 컨텍스트 절약
- **병렬성**: 여러 서브에이전트가 독립적으로 작업 가능

**실제 영향**:
- 100줄 파일 목록 출력: 3,000-5,000 토큰 소비
- 메인 컨텍스트: 200,000 토큰 중 5,000 토큰 소진 (2.5%)
- 10번 반복시: 25% 컨텍스트 낭비

##### 언제 (When)

**서브에이전트 사용이 적합한 경우**:
- ✅ 코드베이스 전체 탐색 (파일 목록, 구조 분석)

#### 규칙 6: 컨텍스트가 60% 사용되면 자동으로 압축하라

- **컨텍스트 한계 방지**: 200K 토큰 제한에 도달하면 강제로 정보가 손실됩니다
- **품질 유지**: 컨텍스트가 가득 차면 응답 품질이 저하됩니다
- **작업 연속성**: 압축으로 더 긴 세션 유지 가능
- **선제적 관리**: 한계에 도달하기 전에 관리하는 것이 효과적

**실제 영향**:
- 60% 시점 압축: 안정적 작업 지속
- 90% 이상: 긴급 압축 필요, 정보 손실 위험
- 100% 도달: 강제 정보 손실

- ✅ 컨텍스트 사용률 60% 이상
- ✅ 긴 대화 세션 (일반적으로 20회 이상 상호작용)
- ✅ 대량 파일을 읽은 후
- ✅ 작업 단위가 전환될 때 (예: 분석 완료 → 구현 시작)

**참고**: "60%", "20회"는 권장 가이드라인입니다. 작업의 복잡도, 파일 크기, 세션 특성에 따라 조정하세요.

#### 규칙 8: 쉘 명령어 출력을 제한하라

##### 왜 (Why)

- **출력 폭발**: ls, find 같은 명령어는 수천 줄 출력 가능
- **컨텍스트 낭비**: 대부분 불필요한 정보가 포함됨
- **가독성 저하**: 너무 많은 출력은 핵심을 찾기 어렵게 만듦
- **토큰 소비**: 1,000줄 출력 = 3,000-5,000 토큰

**실제 영향**:
- 제한 없는 find: 1,000개 파일 = 3,000 토큰
- head -20 사용: 20줄 = 100 토큰
- 절감: 97%

##### 언제 (When)

**출력 제한이 필요한 명령어**:
- ✅ ls (디렉토리 목록)
- ✅ find (파일 검색)
- ✅ grep (텍스트 검색)
- ✅ cat (파일 내용)
- ✅ git log (커밋 히스토리)
- ✅ npm list (패키지 목록)

**전체 출력이 필요한 경우**:
- ❌ 설정 파일 (보통 짧음)
- ❌ 에러 메시지 (전체 내용 필요)
- ❌ 작은 파일 (100줄 미만)

##### 어떻게 (How)

**방법 1: head/tail로 제한**

❌ **Don't**:
```bash
find . -name "*.ts"
# 결과: 1,000개 파일 출력 = 3,000 토큰
```

✅ **Do**:
```bash
find . -name "*.ts" | head -20
# 결과: 20개 파일 경로만 (평균 5 토큰/경로) = 약 100 토큰
# 절감: 약 2,900 토큰 (97%)
# 참고: 파일 경로 길이에 따라 토큰 수는 달라질 수 있음
```

**방법 2: 카운트만 확인**

❌ **Don't**:
```bash
ls -la
# 결과: 전체 파일 목록 + 상세 정보
```

✅ **Do**:
```bash
ls -la | wc -l
# 결과: 파일 개수만
# 필요시: ls -la | head -10
```

**방법 3: 필터링 + 제한**

✅ **효율적인 패턴**:
```bash
# 큰 파일 찾기
find . -type f -size +1M | head -10

# 최근 수정 파일
ls -lt | head -10

# 에러 로그 확인
tail -50 error.log

# Git 최근 커밋
git log --oneline -10
```

#### 규칙 10: 컨텍스트 초기화 전에 상태를 저장하라

##### 왜 (Why)

- **작업 손실 방지**: 초기화 후 어디까지 했는지 알 수 없으면 재작업 필요
- **연속성 유지**: 저장된 상태로 즉시 작업 재개 가능
- **의사결정 보존**: 왜 그렇게 결정했는지 기록 유지
- **팀 협업**: 다른 팀원도 진행 상황 파악 가능

**실제 영향**:
- 저장 없이 초기화: 30분-2시간 작업 손실
- 저장 후 초기화: 즉시 재개

**상태 저장이 필요한 시점**:
- ✅ 컨텍스트 초기화 전

- ✅ 세션 종료 전
- ✅ 큰 변경 전 (안전망)

#### 규칙 11: .claudeignore를 적극적으로 활용하라

##### 왜 (Why)

- **토큰 낭비 방지**: 불필요한 파일을 읽으면 수만 토큰 낭비
- **보안**: 민감한 파일(.env, 키파일) 노출 방지
- **성능**: 파일 스캔 속도 향상
- **집중도**: Claude가 중요한 파일에만 집중

**실제 영향**:
- node_modules 포함시: 수십만 파일, 수백만 토큰
- .claudeignore 사용시: 필요한 파일만, 토큰 99% 절감

##### 언제 (When)

**반드시 .claudeignore에 추가할 것**:
- ✅ 의존성 디렉토리 (node_modules, vendor, venv)
- ✅ 빌드 결과물 (dist, build, out, target)
- ✅ 로그 파일 (*.log, logs/)
- ✅ 캐시 파일 (__pycache__, .cache, *.pyc)

### 2.2 컨텍스트 캐싱 이해하기

컨텍스트 캐싱은 Claude 사용 비용을 극적으로 줄일 수 있는 핵심 메커니즘입니다. 캐시된 토큰은 원래 비용의 10%만 부과되므로 (2025년 1월 기준, 실제 비용은 공식 문서를 참조하세요), 효과적인 캐싱 전략은 90%의 비용 절감을 의미합니다.

**플랫폼별 캐싱 지원**:
- **Claude API**: 프롬프트 캐싱 완전 지원 (개발자가 직접 제어)
- **Claude Code CLI**: 자동 캐싱 지원 (시스템이 자동 관리)

**중요**: 이 섹션의 원칙(캐시 수명, 무효화 조건, 최적화 전략)은 보편적으로 적용되지만, 구체적인 구현 세부사항(캐시 수명, API 응답 형식 등)은 플랫폼마다 다를 수 있습니다. Claude Code CLI는 자동으로 캐싱을 관리하므로 개발자는 원칙에만 집중하면 됩니다.

---

#### 캐싱 작동 원리

##### 무엇이 캐시되는가

Claude는 다음 요소들을 자동으로 캐시합니다:

**1. 시스템 프롬프트 (System Prompt)**

가장 중요한 캐싱 대상입니다. CLAUDE.md 같은 프로젝트 설정이 여기에 포함됩니다.

```
예시:
첫 요청: CLAUDE.md 읽기 (2,000 토큰)
→ 비용: 2,000 토큰 (100%)

5분 내 두 번째 요청: 같은 CLAUDE.md 참조
→ 비용: 200 토큰 (10%)
→ 절감: 1,800 토큰 (90%)
```

**2. Tool 결과 (Claude Code CLI)**

파일 읽기, 검색 등 Tool 실행 결과가 캐시됩니다.

```
예시:
첫 요청: Read user.service.ts (1,500 토큰)
→ 비용: 1,500 토큰 (100%)

5분 내 같은 파일 재요청:
→ 비용: 150 토큰 (10%)
→ 절감: 1,350 토큰 (90%)
```

**3. 대화 히스토리 일부**

이전 대화의 일부가 컨텍스트로 유지되며 캐시됩니다.

**캐싱되지 않는 것**:
- 사용자의 새로운 프롬프트 (매번 다름)
- Claude의 새로운 응답 (생성되는 내용)
- 변경된 파일 내용

---

##### 캐시 수명 (Cache Lifetime)

**기본 수명**: 5분 (Claude API 기준, 플랫폼에 따라 다를 수 있음)

```
Timeline 예시:

00:00 - 첫 요청, CLAUDE.md 로드 (캐시 생성)
00:30 - 두 번째 요청 (캐시 히트 ✅, 5분 연장)
02:00 - 세 번째 요청 (캐시 히트 ✅, 5분 연장)
04:00 - 네 번째 요청 (캐시 히트 ✅, 5분 연장)
10:00 - 다섯 번째 요청 (6분 경과, 캐시 미스 ❌, 새 캐시 생성)
```

**중요**: 
- 캐시는 마지막 사용으로부터 5분간 유지됩니다
- 5분 내에 재사용하면 캐시 수명이 연장됩니다
- 캐시가 살아있는 동안 계속 사용하면 비용을 최소화할 수 있습니다

**실전 활용**:
```
❌ 비효율적 패턴:
- 작업 시작 → 10분 휴식 → 다시 시작
  (캐시 만료로 처음부터 다시)

✅ 효율적 패턴:
- 관련 작업을 묶어서 한 번에
- 5분 내에 다음 요청 이어가기
- 긴 휴식 전에 작업 마무리
```

---

##### 캐시 키 (Cache Key)

**핵심 원칙**: 정확히 동일한 내용만 캐시 히트

캐시 키는 콘텐츠의 해시값처럼 작동합니다. 즉, 정확히 동일한 내용만 매칭되며, 단 한 글자라도 다르면 완전히 다른 캐시로 인식됩니다.

**이것은 캐시 히트입니다**:
```markdown
<!-- 요청 1 -->
CLAUDE.md 내용:
"# 프로젝트 규칙
1. 이모지 사용 금지
2. 한글 커밋 메시지"

<!-- 요청 2 (5분 내) -->
CLAUDE.md 내용:
"# 프로젝트 규칙
1. 이모지 사용 금지
2. 한글 커밋 메시지"

→ 완전히 동일 ✅ → 캐시 히트
```

**이것은 캐시 미스입니다**:
```markdown
<!-- 요청 1 -->
CLAUDE.md 내용:
"# 프로젝트 규칙
1. 이모지 사용 금지
2. 한글 커밋 메시지"

<!-- 요청 2 -->
CLAUDE.md 내용:
"# 프로젝트 규칙
1. 이모지 사용 금지
2. 한글 커밋 메시지
3. 테스트 필수"  ← 한 줄 추가

→ 내용 변경 ❌ → 캐시 미스 (새 캐시 생성)
```

**민감도**:
- 공백 하나, 줄바꿈 하나, 마침표 하나만 달라져도 캐시 미스
- 파일 순서, 들여쓰기, 주석도 영향을 줍니다
- 대소문자도 구분됩니다

**실전 영향**:
```
시나리오: CLAUDE.md (2,000 토큰)를 하루에 10번 요청, 그 중 3번 수정

수정이 없었다면:
- 1번 × 2,000 토큰 (100%) = 2,000 토큰
- 9번 × 200 토큰 (10%) = 1,800 토큰
- 총 비용: 3,800 토큰

3번 수정 시 (캐시 무효화 발생):
- 4번 × 2,000 토큰 (첫 로드 + 3번 재로드) = 8,000 토큰
- 6번 × 200 토큰 (캐시 히트) = 1,200 토큰
- 총 비용: 9,200 토큰

→ 2.4배 증가 (140% 비용 증가)
```

#### 캐시 가능한 요소 목록

어떤 요소가 캐싱에 적합한지, 그리고 어떻게 최적화할지 알아봅시다.

##### 1. CLAUDE.md (최우선순위)

**왜 중요한가**:
- 모든 세션에서 로드됨
- 크기가 큼 (평균 1,000-3,000 토큰)
- 변경 빈도가 낮음 (잘 설계된 경우)
- 캐싱 효과가 가장 큼

**최적화 전략**:

```markdown
❌ 비효율적 CLAUDE.md:
# 프로젝트

## 할 일 목록
- [ ] 버그 수정 (#123)  ← 매일 변경됨
- [ ] 기능 추가 (#124)  ← 자주 업데이트

## 최근 변경사항
- 2026-04-12: API 엔드포인트 추가  ← 매일 추가됨
- 2026-04-11: 데이터베이스 스키마 변경

→ 자주 변경되어 캐시 무효화 반복
```

```markdown
✅ 효율적 CLAUDE.md:
# 프로젝트

## 불변 규칙 (안정적)
1. 이모지 사용 금지
2. 한글 커밋 메시지
3. TypeScript strict 모드

## 아키텍처 (안정적)
- src/api: REST API
- src/services: 비즈니스 로직
- src/utils: 공통 유틸리티

## 작업 추적
→ 별도 파일 참조: .claude/tasks.md  ← 동적 내용 분리

→ 핵심 규칙만 포함, 자주 변경되지 않음
```

**측정 가능한 효과**:
```

비효율적 CLAUDE.md (하루 5번 수정):
- 5번 × 2,000 토큰 (100%) = 10,000
- 15번 × 200 토큰 (10%) = 3,000
- 합계: 13,000 토큰

효율적 CLAUDE.md (주 1번 수정):
- 1번 × 2,000 토큰 (100%) = 2,000
- 19번 × 200 토큰 (10%) = 3,800
- 합계: 5,800 토큰

→ 절감: 7,200 토큰/일 (55% 절감)
→ 월간: 216,000 토큰 절감
```

##### 3. 변경되지 않는 라이브러리/유틸 코드

**캐싱 효과가 큰 코드**:

```typescript
// ✅ utils/date.ts (안정적인 유틸리티)
// 한 번 작성 후 거의 변경되지 않음
// 여러 작업에서 반복 참조됨
// 크기: ~500 토큰

export function formatDate(date: Date): string {
  return date.toISOString().split('T')[0];
}

export function addDays(date: Date, days: number): Date {
  const result = new Date(date);
  result.setDate(result.getDate() + days);
  return result;
}

// 이런 파일은 캐싱 효과 극대화
```

```typescript
// ❌ services/order.service.ts (자주 변경되는 비즈니스 로직)
// 새 기능 추가로 계속 수정됨
// 캐싱 효과 낮음

export class OrderService {
  async createOrder() { ... }
  async updateOrder() { ... }
  // 매일 새 메서드 추가/수정
}
```

**실전 전략**:

```
프로젝트 구조 최적화:

src/
├── utils/           ← 안정적, 캐싱 효과 높음
│   ├── date.ts
│   ├── string.ts
│   └── validation.ts
├── config/          ← 안정적, 캐싱 효과 높음
│   ├── database.ts
│   └── api.ts
├── types/           ← 비교적 안정적
│   ├── user.ts
│   └── order.ts
└── features/        ← 자주 변경, 캐싱 효과 낮음
    ├── auth/
    └── orders/

세션 전략:
1. 안정적 파일 먼저 로드 (utils, config) → 캐싱
2. 작업 중인 feature 파일 로드
3. 같은 세션 내에서 feature 작업 완료

```

```
캐싱 활용:
- 첫 참조: 3개 × 500 토큰 = 1,500 토큰 (100%)
- 나머지 참조: 12번 × 50 토큰 = 600 토큰 (10%)
- 합계: 2,100 토큰

→ 절감: 5,400 토큰 (72% 절감)
```

#### 캐시 무효화 조건

캐시가 언제 무효화되는지 이해하면, 무효화를 피하고 캐시 히트율을 높일 수 있습니다.

```
10:07 - 파일 수정 (새 메서드 추가)

```

##### 3. 세션 종료시

**세션이 끝나면 캐시도 초기화됩니다.**

```
세션 A:
10:00 - CLAUDE.md 로드 (2,000 토큰, 캐시 생성)
10:05 - 작업 완료, 세션 종료
       → 세션 A의 캐시 소멸

세션 B (새 세션):
11:00 - CLAUDE.md 로드 (2,000 토큰, 새 캐시 생성)
       → 세션 A의 캐시는 사용 불가
```

**세션 간 캐시 공유 안 됨**:
- 각 세션은 독립적인 캐시를 가집니다
- 세션 A에서 캐시한 내용을 세션 B에서는 사용할 수 없습니다
- 새 세션 시작시 항상 처음부터 캐시를 구축합니다

**영향 분석**:

```
시나리오: CLAUDE.md 2,000 토큰

긴 세션 (하루 1개 세션):
- 1번 전체 로드 (2,000 토큰)
- 19번 캐시 사용 (3,800 토큰)
- 합계: 5,800 토큰
- 평균: 290 토큰/요청 ⭐⭐⭐⭐⭐

짧은 세션 (하루 5개 세션, 각 4번 요청):
- 5번 전체 로드 (10,000 토큰)
- 15번 캐시 사용 (3,000 토큰)
- 합계: 13,000 토큰
- 평균: 650 토큰/요청 ⭐⭐
- 증가: 124% 더 많은 비용
```

**최적화 전략**:

```
✅ Do: 세션 길이 최적화

1. 관련 작업을 한 세션에
   예: "user 모듈 전체 리팩토링을 한 세션에"
   - 1번 설정 로드로 전체 작업 커버

2. 적절한 세션 분할
   예: "기능별로 세션 분리"
   - auth 기능: 세션 A
   - order 기능: 세션 B
   - 각 세션 내에서 캐시 최대 활용

3. 세션 시작시 워밍업
   예: "필요한 설정 파일을 먼저 로드"
   - package.json, CLAUDE.md 등을 세션 초반에 로드
   - 이후 작업에서 캐시 활용

❌ Don't: 비효율적 세션 관리

- 작은 질문마다 새 세션
  → 매번 설정 재로드

- 관련 없는 작업을 억지로 한 세션에
  → 컨텍스트 오염, 품질 저하

- 세션 중간에 불필요하게 재시작
  → 캐시 손실
```

**세션 종료 체크리스트**:

```
세션 종료 전 확인:
[ ] 진행 중인 작업 완료했는가?
[ ] 중요 정보를 파일/커밋으로 저장했는가?
[ ] 다음 세션에서 이어서 해야 하는가?
    → Yes: 잠시 기다렸다가 같은 세션에서 계속
    → No: 안심하고 세션 종료
```

#### 캐시 친화적 프로젝트 구조

프로젝트의 파일 구조와 관리 방식이 캐시 히트율에 큰 영향을 줍니다.

##### 설정 파일 안정화

**원칙**: 자주 변경하지 않는 설정 구조를 만들어라.

**Bad 구조** (캐시 무효화 빈번):

```typescript
// config.ts (모든 설정이 한 파일에)

  // 안정적 설정
  app: {
    name: 'MyApp',
    version: '1.0.0',
  },
  
  // 자주 변경되는 설정
  features: {
    newFeatureA: true,  // ← 매일 추가/변경
    newFeatureB: false, // ← 실험 중
    betaFeatureC: true, // ← 자주 토글
  },
  
  // 안정적 설정
  api: {
    baseUrl: 'https://api.example.com',
    timeout: 5000,
  },
};

// 문제: features를 수정할 때마다 전체 파일 캐시 무효화
// 영향: api, app 설정도 캐시 미스
```

**Good 구조** (캐시 히트 최대화):

```typescript
// config/app.ts (안정적 - 거의 변경 안 함)
export const appConfig = {
  name: 'MyApp',
  version: '1.0.0',
};

// config/api.ts (안정적 - 거의 변경 안 함)
export const apiConfig = {
  baseUrl: 'https://api.example.com',
  timeout: 5000,
};

// config/features.ts (동적 - 자주 변경됨)
export const featureFlags = {
  newFeatureA: true,  // 자주 변경
  newFeatureB: false, // 실험 중
  betaFeatureC: true, // 자주 토글
};

// 장점: features만 변경해도 app, api는 캐시 유지
```

**측정 가능한 효과**:

```

- 합계: 2,800 토큰/일

→ 절감: 900 토큰/일 (24% 절감)
```

##### 세션 시작시 캐시 워밍 (Cache Warming)

**원칙**: 세션 초반에 필요한 파일을 미리 로드하여 캐시를 구축하라.

**캐시 워밍이란?**
작업 시작 시 핵심 파일들을 미리 로드하여 캐시를 생성하는 전략입니다. 이후 작업에서 이 캐시를 반복 활용하여 토큰을 절감합니다.

**두 가지 접근법**:

1. **계획적 워밍** (필요한 파일을 아는 경우)
   - 작업 시작 전 필요 파일 목록 작성
   - 세션 초반에 한 번에 로드
   - 이후 작업에서 캐시 활용

2. **점진적 워밍** (탐색적 작업의 경우)
   - 핵심 파일(CLAUDE.md, 설정)부터 먼저 로드
   - 작업하면서 추가 파일 발견 시 로드
   - 자연스럽게 캐시 구축

**계획적 워밍 예시**:

**Bad 패턴** (파일을 필요할 때마다 로드):

```
세션 시작 (10:00):

사용자: "Order 기능을 구현하려고 해"
Claude: "네, 어떻게 도와드릴까요?"

사용자: "먼저 order.service.ts 만들어줘"
Claude: [필요할 때마다 파일 로드]
       [types/order.ts 로드 (500)]

사용자: "이제 order.controller.ts 만들어줘"
Claude: [types/order.ts 재로드? 또는 기억 의존]

사용자: "order 관련 테스트도 작성해줘"
Claude: [이미 작성한 코드를 다시 읽어야 할 수도...]

→ 파일을 여러 번 나눠서 로드
→ 일부는 캐시, 일부는 재로드
→ 비효율적
```

**Good 패턴** (계획적 캐시 워밍):

```
       "구조를 파악했습니다. [요약]"

사용자: "이제 order.service.ts 만들어줘"
Claude: [types/order.ts 캐시 (50)]
       [utils/validation.ts 캐시 (40)]
       [order.service.ts 생성]

사용자: "order.controller.ts도 만들어줘"
```

## 3. 프로젝트 설정

효율적인 Claude 사용의 기반은 **올바른 프로젝트 설정**입니다. 이 섹션에서는 필수 설정 파일과 자동화 구성 방법을 다룹니다.

**왜 중요한가**:
- **비용**: 올바른 설정만으로도 토큰 사용량 30-50% 절감 가능
- **품질**: 일관된 규칙으로 팀 전체가 같은 품질의 결과물 생성
- **자동화**: Hook으로 반복 작업 자동화, 실수 방지

**설정 우선순위**:
1. **필수** (즉시): `.claudeignore` - 불필요한 파일 제외
2. **필수** (1일 내): `CLAUDE.md` - 프로젝트 규칙 정의
3. **권장** (1주 내): Hooks - 자동 검증 및 최적화

### 3.2 .claudeignore 설정

**`.claudeignore`**는 Claude가 읽지 말아야 할 파일/디렉토리를 지정하는 파일입니다 (`.gitignore`와 유사).

**핵심 목적**:
- **비용 절감**: 불필요한 파일 제외로 토큰 낭비 방지
- **보안**: 민감 정보 파일 자동 제외
- **성능**: 파일 검색/읽기 속도 향상

**위치**: 프로젝트 루트에 `.claudeignore` 파일 생성

**문법**: `.gitignore`와 동일
```bash
# 주석
node_modules/          # 디렉토리 전체
*.log                  # 패턴 매칭
dist                   # 특정 이름
!important.log         # 예외 (제외하지 않음)
```

### 3.3 Hooks 설정

**Hooks**는 특정 이벤트 발생시 자동으로 실행되는 스크립트입니다 (Git hooks와 유사).

**핵심 목적**:
- **자동화**: 반복 작업 자동 수행
- **검증**: 실수 방지 (커밋 전 체크)
- **최적화**: 세션 시작시 상태 요약

---

**⚠️ 중요 고지사항**:

이 섹션은 프로젝트 자동화를 위한 권장 패턴을 설명합니다. **Claude Code CLI의 `.claude/hooks/` 디렉토리를 통한 Hook 공식 지원 여부는 Claude Code 공식 문서를 확인하세요.**

**대안**:
- **Git Hooks**: `.git/hooks/`를 사용하여 동일한 검증 로직 구현 가능 (Git 기본 기능)
- **CI/CD**: GitHub Actions, GitLab CI 등으로 자동화
- **Pre-commit Framework**: [pre-commit.com](https://pre-commit.com) 사용

이 섹션의 Hook 예시는 Git hooks나 다른 자동화 도구로도 그대로 적용할 수 있습니다.

#### Hook 종류 및 목적

##### 1. pre-commit Hook

**목적**: 커밋 전 자동 검증으로 실수 방지

**트리거**: `git commit` 실행 전

**주요 용도**:
- 대용량 파일 커밋 방지 (5MB 이상)
- 민감 정보 파일 커밋 방지 (.env, *.pem)
- CLAUDE.md 토큰 최적화 규칙 위반 체크
- 코드 스타일 자동 검사

**예시 시나리오**:

```bash
# 상황: 실수로 .env 파일을 커밋하려고 함

$ git add .env
$ git commit -m "Add config"

# Hook 실행 →

⛔ pre-commit Hook 실패!

문제: .env 파일이 포함되어 있습니다.
파일: .env (민감 정보)

조치:
1. git reset HEAD .env
2. .env를 .gitignore에 추가
3. 다시 커밋

커밋 중단됨.

# 효과: 민감 정보 유출 방지
```

**목적**: 파일 수정 후 자동 체크 및 제안

**트리거**: Claude가 파일을 수정한 직후

**주요 용도**:
- 파일 크기 경고 (500줄 이상시 분할 제안)
- 자동 .claudeignore 업데이트 제안
- 새로운 의존성 추가 감지
- 코드 품질 즉시 체크

**예시 시나리오**:


#### 각 Hook 상세 예시

**완전한 pre-commit Hook 예시**:

다음은 즉시 사용 가능한 pre-commit Hook입니다:

```bash
#!/bin/bash
# .claude/hooks/pre-commit
# 커밋 전 자동 검증으로 실수 방지

echo "🔍 pre-commit Hook 실행 중..."

# 1. 대용량 파일 체크 (5MB 이상)
echo "  - 대용량 파일 체크..."
for file in $(git diff --cached --name-only); do
  if [ -f "$file" ]; then
    size=$(wc -c < "$file" 2>/dev/null || echo 0)
    if [ $size -gt 5242880 ]; then  # 5MB
      echo ""
      echo "⛔ pre-commit Hook 실패!"
      echo ""
      echo "문제: 파일이 너무 큽니다"
      echo "파일: $file ($(($size / 1024 / 1024))MB)"
      echo ""
      echo "조치:"
      echo "  1. git reset HEAD $file"
      echo "  2. 해당 파일을 .claudeignore에 추가"
      echo "  3. Git LFS 사용 고려: git lfs track \"$file\""
      echo ""
      exit 1
    fi
  fi
done

# 2. 민감 정보 파일 체크
echo "  - 민감 정보 파일 체크..."
for file in $(git diff --cached --name-only); do
  if [[ "$file" == *.env* ]] || [[ "$file" == *.pem ]] || \
     [[ "$file" == *credentials* ]] || [[ "$file" == *.key ]]; then
    echo ""
    echo "⛔ pre-commit Hook 실패!"
    echo ""
    echo "문제: 민감 정보 파일이 감지되었습니다"
    echo "파일: $file"
    echo ""
    echo "조치:"
    echo "  1. git reset HEAD $file"
    echo "  2. .gitignore에 추가"
    echo "  3. 이미 커밋된 경우: git filter-repo 사용"
    echo ""
    exit 1
  fi
done

# 3. CLAUDE.md 크기 체크 (2000줄 초과시 경고)
if git diff --cached --name-only | grep -q "CLAUDE.md"; then
  if [ -f ".claude/CLAUDE.md" ]; then
    lines=$(wc -l < ".claude/CLAUDE.md")
    if [ $lines -gt 2000 ]; then
      echo ""
      echo "⚠️  경고: CLAUDE.md가 너무 큽니다 ($lines줄)"
      echo "   권장: 2,000줄 미만 (현재: $lines줄)"
      echo "   상세 내용은 별도 파일로 분리하는 것을 고려하세요."
      echo ""
      # 경고만 표시, 커밋은 허용
    fi
  fi
fi

echo "✅ pre-commit 검사 완료"
exit 0
```

**설치 방법**:

```bash

# 2. Hook 파일 생성

[위 스크립트 내용 붙여넣기]
EOF

# 3. 실행 권한 부여
chmod +x .claude/hooks/pre-commit

# 4. 테스트
.claude/hooks/pre-commit
# 예상 출력: ✅ pre-commit 검사 완료
```

#### 설정 방법 (Claude Code CLI)

**Hook 디렉토리 구조**:

```
.claude/
├── CLAUDE.md
└── hooks/
    ├── pre-commit          # 실행 가능한 스크립트
    ├── post-edit           # 실행 가능한 스크립트
    └── session-start       # 실행 가능한 스크립트
```

**단계별 설정**:

**Step 1**: Hook 디렉토리 생성

```bash
mkdir -p .claude/hooks
```

**Step 2**: Hook 스크립트 작성

```bash
# 예: pre-commit Hook 생성
touch .claude/hooks/pre-commit
chmod +x .claude/hooks/pre-commit
```

**Step 3**: Hook 스크립트 내용 작성

```bash
#!/bin/bash
# .claude/hooks/pre-commit

# 대용량 파일 체크 (5MB 이상)
for file in $(git diff --cached --name-only); do
  if [ -f "$file" ]; then
    size=$(wc -c < "$file")
    if [ $size -gt 5242880 ]; then  # 5MB
      echo "⛔ 파일이 너무 큽니다: $file ($(($size / 1024 / 1024))MB)"
      echo "   .claudeignore에 추가하거나 Git LFS 사용을 고려하세요."
      exit 1
    fi
  fi
done

# 민감 정보 파일 체크
for file in $(git diff --cached --name-only); do
  if [[ "$file" == *.env* ]] || [[ "$file" == *.pem ]] || [[ "$file" == *credentials* ]]; then
    echo "⛔ 민감 정보 파일 감지: $file"
    echo "   이 파일을 정말 커밋하시겠습니까?"
    exit 1
  fi
done

echo "✅ pre-commit 검사 통과"
exit 0
```

**Step 4**: Git 커밋 및 팀 공유

```bash
git add .claude/hooks/pre-commit
git commit -m "Add pre-commit hook"
```

**Step 5**: 팀원 설정

팀원이 저장소를 clone한 후:

```bash
# Hook 실행 권한 설정
chmod +x .claude/hooks/*

# Claude Code CLI가 자동으로 Hook 인식
```

#### 주의사항

**Hook은 빠르게 실행되어야 함 (< 1초)**

```markdown
❌ Don't: 느린 Hook

#!/bin/bash
# 전체 프로젝트 린트 (30초 소요)
npm run lint

# 모든 테스트 실행 (2분 소요)
npm run test

# 문제: 커밋할 때마다 2분 30초 대기
# 결과: 개발자가 Hook을 비활성화하게 됨
```

```markdown
✅ Do: 빠른 Hook

#!/bin/bash
# 변경된 파일만 빠른 체크 (< 1초)

# 1. 간단한 파일 크기 체크 (매우 빠름)
for file in $(git diff --cached --name-only); do
  # ... 크기 체크 로직
done

# 2. 민감 정보 패턴 체크 (빠름)
# ... 패턴 매칭

# 무거운 작업은 CI에서 수행
echo "✅ 빠른 체크 완료 (전체 검증은 CI에서 수행)"
exit 0

# 효과: 1초 미만, 개발 흐름 방해 없음
```

**성능 가이드라인**:
- **pre-commit**: 최대 1초 (이상적: 0.5초)
- **post-edit**: 최대 0.5초 (거의 즉시)
- **session-start**: 최대 2초 (한 번만 실행)

## 4. 프롬프팅 베스트 프랙티스

좋은 설정만으로는 부족합니다. **어떻게 요청하는가**가 결과물의 품질과 토큰 효율성을 결정합니다.

이 섹션에서는 작업 유형별로 효과적인 프롬프트 작성법을 다룹니다.

### 섹션 개요

- **4.1 공통 원칙**: 모든 프롬프트에 적용되는 기본 원칙
- **4.2 코딩 작업용 프롬프팅**: 리팩토링, 버그 수정, 새 기능, 코드 리뷰
- **4.3 문서 작성용 프롬프팅**: README, 가이드, API 문서 등
- **4.4 리팩토링/리뷰용 프롬프팅**: 전체 스캔, 우선순위 지정, 영향도 분석

**💡 핵심 포인트**: 좋은 프롬프트는 토큰을 절약하면서도 더 나은 결과를 만듭니다.

---

### 4.1 공통 원칙

모든 작업 유형에 공통으로 적용되는 프롬프팅 원칙입니다.

#### 원칙 1: 명확한 목표 명시

**왜**: 모호한 목표는 Claude가 불필요한 탐색을 하게 만들어 토큰을 낭비합니다.

**어떻게**:
- 무엇을 원하는지 명확히
- 성공 기준 제시
- 범위 한정

**❌ Bad**:
```
사용자 인증 부분 좀 개선해줘
```

**문제점**:
- "개선"이 무엇을 의미하는지 불명확 (보안? 성능? UX?)
- 범위가 너무 넓음 (전체 인증 시스템?)
- 성공 기준 없음

**토큰 영향**: Claude가 전체 인증 시스템을 분석하고 여러 개선안을 제시 → 3,000-5,000 토큰

**✅ Good**:
```
src/auth/login.ts의 로그인 실패시 에러 메시지를 더 구체적으로 개선해줘.

목표:
- 현재: "로그인 실패"
- 변경 후: "이메일 또는 비밀번호가 일치하지 않습니다" 
         (보안상 어느 쪽이 틀렸는지는 밝히지 않음)

조건:
- 보안 베스트 프랙티스 준수 (타이밍 공격 방지)
- 다국어 지원 고려 (i18n 키 사용)
- 기존 테스트 통과
```

**개선 효과**:
- 정확한 파일과 범위 지정 → 불필요한 탐색 제거
- 명확한 Before/After → 정확한 구현
- 조건 명시 → 재작업 방지

**토큰 영향**: 필요한 파일만 읽고 정확히 구현 → 800-1,200 토큰 (70% 절감)

---

#### 원칙 2: 컨텍스트 최소화 (필요한 것만)

**왜**: 불필요한 파일이나 정보는 토큰을 낭비하고 Claude의 집중력을 분산시킵니다.

**어떻게**:
- 필요한 파일만 명시
- 관련 없는 정보 제외
- 이미 읽은 파일 재언급 방지

**❌ Bad**:
```
프로젝트 전체를 분석해서 API 응답 시간을 개선해줘
```

**문제점**:
- 전체 프로젝트 스캔 → 수만 토큰
- 문제 위치를 찾는 것부터 시작 → 비효율적
- 우선순위 없음

**토큰 영향**: 전체 프로젝트 스캔 + 분석 → 20,000-50,000 토큰

**✅ Good**:
```
/api/products 엔드포인트의 응답 시간을 개선해줘.

현재 상태:
- 평균 응답 시간: 2.5초
- 병목: DB 쿼리 (slow query log 확인)

관련 파일:
- src/api/products.controller.ts (현재 구현)
- src/services/product.service.ts (DB 쿼리 로직)

목표:
- 응답 시간 500ms 이하
- N+1 쿼리 제거
- 기존 API 스펙 유지
```

**개선 효과**:
- 문제 위치 명시 → 탐색 시간 제거
- 필요한 파일만 → 토큰 절약
- 구체적 목표 → 정확한 최적화

**토큰 영향**: 필요한 2개 파일만 분석 + 최적화 → 2,000-3,000 토큰 (85% 절감)

---

#### 원칙 3: 단계별 진행 (한 번에 너무 많이 요청하지 않기)

**왜**: 복잡한 작업을 한 번에 요청하면 불완전하거나 잘못된 결과가 나올 수 있습니다.

**어떻게**:
- 큰 작업을 작은 단계로 분할
- 각 단계를 검증하고 다음으로
- 중간 결과물 확인

**❌ Bad**:
```
새로운 결제 시스템을 구현해줘. Stripe 연동하고, 환불 처리하고, 
관리자 대시보드도 만들고, 이메일 알림도 보내줘.
```

**문제점**:
- 너무 많은 요구사항 → 일부만 구현되거나 품질 저하
- 중간 검증 불가능 → 잘못된 방향으로 진행 가능
- 롤백 어려움

**토큰 영향**: 거대한 구현 시도 → 10,000-30,000 토큰 (실패 가능성 높음)

**✅ Good**:
```
Step 1: 먼저 Stripe 결제 연동 기본 구조를 구현해줘.

범위:
- 결제 생성 (Payment Intent)
- 결제 확인 콜백
- 기본 에러 핸들링

제외:
- 환불 (다음 단계)
- 관리자 UI (다음 단계)
- 이메일 알림 (다음 단계)

완료 조건:
- 테스트 모드에서 결제 성공 확인
- 단위 테스트 통과
```

**Step 1 완료 후**:
```
✅ Step 1 완료 확인

Step 2: 이제 환불 처리를 추가해줘.

현재 상태:
- 기본 결제는 위에서 구현 완료
- src/payment/stripe.service.ts에 코드 있음

추가 기능:
- 전액 환불
- 부분 환불
- 환불 히스토리 저장

[Step 3, 4... 순차적으로 진행]
```

**개선 효과**:
- 각 단계 검증 가능 → 품질 보장

#### 리팩토링

리팩토링은 동작을 유지하면서 구조를 개선하는 작업입니다.

**핵심 요소**:
1. 목표와 제약 명확히
2. 변경 범위 한정
3. 테스트 유지 요구

**❌ Bad**:
```

```

**문제점**:
- 리팩토링 목표 불명확 (가독성? 성능? 구조?)

- 동작 보존 보장 없음

**토큰 영향**: 불필요한 변경 + 재작업 → 5,000-8,000 토큰

**✅ Good**:
```

- 현재 200줄 함수를 작은 단위로 분리
- 각 단계별 책임 명확히 (검증, 계산, 저장)
- 테스트 가능성 향상
```

**개선 효과**:

- 테스트 보존 → 안전성 보장
- 범위 한정 → 불필요한 변경 방지

**토큰 영향**: 정확한 리팩토링 + 검증 → 2,500-3,500 토큰 (50-60% 절감)

#### 버그 수정

버그 수정은 재현 단계와 예상 동작이 중요합니다.

**핵심 요소**:
1. 재현 단계 제공
2. 예상 vs 실제 동작 설명
3. 관련 파일만 언급

**❌ Bad**:
```

```

**문제점**:
- "안 된다"가 무엇인지 불명확
- 재현 방법 없음
- 디버깅을 처음부터 시작

**토큰 영향**: 전체 장바구니 시스템 분석 + 추측 → 8,000-15,000 토큰

**✅ Good**:
```

재현 단계:
1. 상품을 장바구니에 추가 (정상 작동)
2. 장바구니 페이지에서 수량을 2 → 5로 변경
3. 수량은 업데이트되지만 총 가격은 수량 2 기준으로 유지됨

예상 동작:
- 수량 변경시 총 가격 = (단가 × 새 수량)으로 즉시 갱신

실제 동작:
- 수량만 변경되고 총 가격은 이전 값 유지
- 페이지 새로고침하면 정상 가격 표시 (서버는 정상)

관련 파일:
- src/components/Cart.tsx (UI 컴포넌트)

```

**개선 효과**:
- 정확한 재현 → 빠른 버그 발견
- 예상 동작 명시 → 정확한 수정
- 관련 파일 한정 → 토큰 절약

**토큰 영향**: 필요한 파일만 분석 + 수정 → 1,500-2,500 토큰 (80% 절감)

**수정 프로세스**:

```typescript
// Before: 버그 있는 코드
const updateQuantity = (itemId: string, quantity: number) => {
  setItems(items.map(item => 
    item.id === itemId ? { ...item, quantity } : item
  ));
  // total 재계산 누락!
};

// After: 수정된 코드

  const updatedItems = items.map(item => 
    item.id === itemId ? { ...item, quantity } : item
```

```
- 예상 출력 포함 (사용자가 성공 확인 가능하게)
- 문제 발생시 트러블슈팅 팁

예시:
\`\`\`markdown
```

#### API 문서

API 문서는 일관된 형식이 중요합니다.

**❌ Bad**:
```
API 문서를 만들어줘
```

**✅ Good**:
```
REST API 문서를 작성해줘.

대상: src/api/products.controller.ts의 모든 엔드포인트

형식: Markdown (docs/api/products.md)

각 엔드포인트별 포함 내용:
1. HTTP 메서드 및 경로
2. 설명 (1-2줄)
3. 인증 요구사항
4. 요청 파라미터 (Path/Query/Body)
   - 타입, 필수/선택, 설명, 예시
5. 응답 형식 (성공/실패 케이스별)
6. 예시 요청/응답 (curl 명령어)
7. 에러 코드

템플릿:
\`\`\`markdown
### GET /api/products/:id
특정 상품의 상세 정보를 조회합니다.

**Authentication:** Required (Bearer token)

**Path Parameters:**
| Name | Type | Description |
|------|------|-------------|
| id | string | 상품 ID (UUID) |

**Response (200 OK):**
\`\`\`json
{
  "id": "123e4567-e89b-12d3-a456-426614174000",
  "name": "상품명",
  "price": 29900,
  ...
}
\`\`\`

**Errors:**
- 401: 인증 실패
- 404: 상품을 찾을 수 없음

**Example:**
\`\`\`bash
curl -X GET https://api.example.com/api/products/123 \\
  -H "Authorization: Bearer YOUR_TOKEN"
\`\`\`
\`\`\`

한 번에 2-3개 엔드포인트씩 작성하고 확인 후 계속.
```

**개선 효과**:
- 일관된 템플릿 → 읽기 쉬운 문서
- 예시 포함 → 즉시 사용 가능
- 배치 작성 → 효율성

**토큰 영향**: 엔드포인트당 500-800 토큰 (템플릿 재사용으로 효율적)

#### 기술 블로그/가이드

긴 문서는 반드시 섹션별로 작성합니다.

**❌ Bad**:
```
TypeScript 마이그레이션 가이드를 작성해줘. 처음부터 끝까지 다.
```

**문제점**:
- 너무 긴 요청 → 불완전한 결과
- 품질 검증 어려움

**✅ Good**:
```
"JavaScript에서 TypeScript로 마이그레이션" 가이드를 작성해줘.

대상 독자: 중급 JavaScript 개발자 (TypeScript 초심자)
분량: 약 2000-3000 단어
톤: 친근하고 실용적, 예시 중심

Step 1: 먼저 아웃라인만 작성해줘.

포함 섹션:
- 왜 TypeScript인가
- 마이그레이션 전 준비
- 단계별 마이그레이션 (점진적 접근)
- 일반적인 패턴 변환
- 문제 해결
- 다음 단계

각 섹션의 주요 포인트 3-5개씩 나열.

아웃라인 승인 후 섹션별로 작성 진행.
```

**아웃라인 승인 후**:
```
✅ 아웃라인 좋음. Section 1, 2를 작성해줘.

Section 1: 왜 TypeScript인가 (500-700 단어)
- 타입 안전성 이점
- 개발 경험 향상 (IDE 지원)
- 유지보수성
- 실제 사례 1개 (Before/After 코드)

Section 2: 마이그레이션 전 준비 (600-800 단어)
- 테스트 커버리지 확보
- tsconfig.json 초기 설정
- 도구 설치
- 체크리스트 형식으로

Section 1, 2 완성 후 다음 섹션 계속.
```

**개선 효과**:
- 섹션별 품질 관리
- 중간 피드백 가능
- 일관성 유지

**토큰 영향**: 
- 전체 한 번에: 15,000-25,000 토큰 (품질 저하 위험)
- 섹션별: 2,000-3,000 × 6섹션 = 12,000-18,000 토큰 (고품질 보장)

### 4.4 리팩토링/리뷰용 프롬프팅

대규모 코드베이스의 리팩토링이나 리뷰는 전략적 접근이 필요합니다.

#### 전체 스캔 vs 특정 부분

**전체 스캔 (탐색 모드)**:

목적: 문제 영역 파악, 우선순위 결정

**✅ Good**:
```
프로젝트 전체에서 보안 취약점을 스캔해줘.

범위: src/ 디렉토리 전체

집중 항목:
1. SQL Injection 가능성
2. XSS 취약점
3. 하드코딩된 시크릿
4. 인증/인가 누락

출력 형식:
각 발견 사항을:
- [심각도] 파일:라인 - 문제
- 간단한 설명 (1-2줄)
- 영향도 (High/Medium/Low)

상세 분석이나 수정은 하지 말고, 목록만 먼저.
우선순위 High인 항목 발견시 즉시 알려줘.

예상: 20-30개 항목 발견 (목록만이라 토큰 효율적)
```

**개선 효과**:
- 목록만 → 빠른 스캔
- 우선순위 파악 → 중요한 것부터
- 토큰 절약

**토큰 영향**: 전체 스캔 + 목록 생성 → 5,000-8,000 토큰 (상세 분석 대비 70% 절감)

---

**특정 부분 (상세 분석)**:

스캔 후 중요한 부분을 상세 분석합니다.

**✅ Good**:
```
위에서 발견한 High 우선순위 항목 3개를 상세 분석하고 수정해줘.

항목:
1. src/api/users.controller.ts:45 - SQL Injection
2. src/middleware/auth.ts:23 - 인증 우회 가능
3. src/config/database.ts:12 - 하드코딩된 비밀번호

각 항목별:
1. 현재 코드 (문제 부분)
2. 왜 위험한지 (공격 시나리오)
3. 수정 코드
4. 테스트 방법

하나씩 순차적으로 진행. 각 수정 완료 후 다음.
```

**개선 효과**:
- 우선순위 집중 → 효과적
- 상세 분석 → 제대로 된 수정
- 순차 진행 → 검증 가능

**토큰 영향**: 항목당 1,500-2,500 토큰 × 3 = 4,500-7,500 토큰

---

#### 우선순위 지정

리팩토링/리뷰는 우선순위가 중요합니다.

**❌ Bad**:
```
이 프로젝트를 개선해줘
```

**✅ Good**:
```
이 프로젝트의 개선점을 찾고 우선순위별로 분류해줘.

분석 범위: src/ 전체

분류 기준:
- **Critical**: 보안, 데이터 손실, 심각한 버그
- **Important**: 성능 이슈, 확장성 문제, 기술 부채
- **Nice-to-have**: 가독성, 코드 스타일, 최적화

각 카테고리별:
- 문제 설명
- 영향도 (사용자/시스템/개발자)
- 예상 수정 시간
- 수정 난이도

출력 형식:
\`\`\`markdown
## Critical (즉시 수정)
1. [보안] SQL Injection in users.controller.ts
   - 영향: 전체 DB 노출 가능
   - 수정 시간: 2시간
   - 난이도: 중

## Important (이번 주)
1. [성능] N+1 쿼리 in products.service.ts
   - 영향: API 응답 시간 5초
   - 수정 시간: 4시간
   - 난이도: 중

## Nice-to-have (백로그)
1. [가독성] 중복 코드 in utils/
   - 영향: 유지보수성
   - 수정 시간: 3시간
   - 난이도: 하
\`\`\`

분류 후 Critical 항목부터 수정 진행 여부 결정.
```

**개선 효과**:
- 명확한 우선순위 → 효율적 작업
- 영향도 파악 → 의사결정 용이
- 예상 시간 → 계획 수립 가능

**토큰 영향**: 분석 + 분류 → 6,000-10,000 토큰 (가치 있는 투자)

---

#### 변경 영향도 고려

리팩토링시 영향도 분석은 필수입니다.

**✅ Good**:
```
src/services/payment.service.ts를 리팩토링하려고 해.
먼저 변경 영향도를 분석해줘.

분석 항목:
1. 이 파일을 import하는 모든 파일 목록
2. Public API (외부에서 사용하는 메서드)
3. 테스트 파일 존재 여부
4. 의존성 (이 파일이 사용하는 다른 서비스)

출력:
\`\`\`markdown
## 영향도 분석

### 직접 의존 파일 (5개)
- src/api/checkout.controller.ts
- src/api/subscriptions.controller.ts
- src/jobs/payment-retry.job.ts
- src/webhooks/stripe.webhook.ts
- src/services/order.service.ts

### Public API
1. processPayment(amount, method)
2. refundPayment(paymentId, amount?)
3. getPaymentStatus(paymentId)

⚠️ 주의: Public API 변경시 위 5개 파일 모두 수정 필요

### 테스트
✅ src/services/__tests__/payment.service.test.ts 존재
- 커버리지: 85%
- 테스트 수: 23개

### 의존성
- StripeService (외부 API)
- DatabaseService
- LoggerService

## 권장 사항
- Public API는 유지하고 내부 구현만 변경
- 테스트 먼저 100% 통과 확인
- 리팩토링 후 의존 파일들의 테스트도 실행
\`\`\`

이 분석 기반으로 리팩토링 계획 수립.
```

**개선 효과**:
- 영향도 사전 파악 → 안전한 리팩토링
- 테스트 전략 수립 → 품질 보장
- 리스크 최소화

**토큰 영향**: 영향도 분석 → 2,000-3,500 토큰 (리팩토링 실패 방지)

---

#### 단계별 리팩토링 계획

대규모 리팩토링은 계획이 중요합니다.

**✅ Good**:
```
레거시 인증 시스템 (src/auth/)을 현대화하는 계획을 수립해줘.

현재 상태:
- 5년된 코드
- 세션 기반 (JWT로 전환 원함)
- 테스트 없음
- 3,000줄 코드

목표:
- JWT 기반 인증
- 테스트 커버리지 80% 이상
- 기존 사용자 호환성 유지
- 다운타임 없음

계획 형식:
각 Phase별로:
1. 목표 (무엇을)
2. 작업 항목 (상세)
3. 테스트 전략
4. 롤백 계획
5. 예상 시간
6. 리스크 및 완화

Phase는 4-6개로 나누고, 각 Phase는 1-2주 내 완료 가능하게.

계획 승인 후 Phase별로 실행.
```

**출력 예시**:
```markdown
## Phase 1: 기반 준비 (1주)
목표: 기존 시스템 테스트 작성, 새 구조 설계

작업:
1. 레거시 코드 테스트 작성 (현재 동작 보존)
2. JWT 인증 구조 설계 문서
3. 새 auth 모듈 스켈레톤 생성

테스트:
- 레거시 시스템 테스트 80% 커버리지

롤백:
- 새 코드는 아직 사용 안 함 (롤백 불필요)

리스크:
- 테스트 작성이 예상보다 오래 걸릴 수 있음
- 완화: 핵심 플로우 우선, 엣지 케이스는 Phase 2

## Phase 2: 병렬 구현 (2주)
목표: JWT 시스템 구현, 레거시와 병렬 운영

작업:
1. JWT 발급/검증 구현
2. 새 로그인 엔드포인트 (/auth/v2/login)
3. Feature flag로 제어 (기본 off)

테스트:
- 새 시스템 테스트 100%
- 통합 테스트 (레거시 영향 없음 확인)

롤백:
- Feature flag off → 레거시로 즉시 복귀

리스크:
- 두 시스템 병렬 운영 복잡도
- 완화: 명확한 분리, 공통 코드 최소화

[Phase 3, 4, 5 계속...]
```

**개선 효과**:
- 명확한 로드맵 → 팀 정렬
- 리스크 관리 → 안전한 마이그레이션
- 점진적 진행 → 지속적 배포 가능

**토큰 영향**: 계획 수립 → 4,000-6,000 토큰 (실행 실패 방지로 큰 가치)

---

#### 리팩토링/리뷰 체크리스트

```markdown
전체 스캔:
```
