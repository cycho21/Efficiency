### 섹션 개요

- **6.1 결과물 검증 방법**: 코드와 문서 품질 확인 기법

**핵심 포인트**: 검증 없는 결과물은 위험합니다. 체계적인 검증 프로세스를 통해 품질을 보장하세요.

### 6.1 결과물 검증 방법

Claude가 생성한 코드와 문서를 검증하는 체계적인 방법입니다.

#### 6.1.1 코드 품질 검증

**레벨 1: 기본 검증 (필수)**

모든 코드 변경에 적용해야 하는 기본 체크리스트입니다.

```markdown
## 코드 기본 검증 체크리스트

### 1. 문법 및 타입 체크
- [ ] TypeScript 컴파일 에러 없음 (`npm run build`)
- [ ] ESLint 경고 없음 (`npm run lint`)
- [ ] 타입 정의 완전함 (any 사용 최소화)

### 2. 테스트
- [ ] 기존 테스트 모두 통과
- [ ] 새로운 코드에 대한 테스트 추가됨
- [ ] 테스트 커버리지 기준 충족 (80% 이상)

### 3. 기능 검증
- [ ] 의도한 동작 정상 작동
- [ ] 에지 케이스 처리됨
- [ ] 에러 핸들링 적절함

### 4. 부작용 확인
- [ ] 기존 기능에 영향 없음
- [ ] 성능 저하 없음
- [ ] 메모리 누수 없음
```

**실행 예시**:

```bash
# 1. 빌드 검증
npm run build

# 2. 린트 검증
npm run lint

# 3. 테스트 실행
npm test

# 4. 커버리지 확인
npm run test:coverage
```

**예상 결과**:
```
✓ Build: 0 errors
✓ Lint: 0 warnings
✓ Tests: 45 passed
✓ Coverage: 87% (target: 80%)
```

**레벨 2: 보안 검증 (중요 기능)**

인증, 결제, 개인정보 처리 등 중요한 기능에 적용합니다.

```markdown
## 보안 검증 체크리스트

### 1. 입력 검증
- [ ] 모든 사용자 입력 sanitize됨
- [ ] SQL injection 방지 (prepared statements)
- [ ] XSS 방지 (출력 escape)

### 2. 인증/인가
- [ ] 인증 체크 적절히 적용됨
- [ ] 권한 체크 누락 없음
- [ ] 세션/토큰 관리 안전함

### 3. 데이터 보호
- [ ] 민감 정보 암호화됨
- [ ] 로그에 민감 정보 노출 없음
- [ ] HTTPS 사용 강제

### 4. 취약점 스캔
- [ ] npm audit 경고 없음
- [ ] OWASP Top 10 체크
```

**실행 예시**:

```bash
# 1. 의존성 취약점 스캔
npm audit

# 2. 정적 보안 분석 (예: Snyk)
npx snyk test

# 3. 민감 정보 검색
grep -r "password\|secret\|api_key" src/
```

**레벨 3: 성능 검증 (고부하 기능)**

API, 데이터 처리 등 성능이 중요한 기능에 적용합니다.

```markdown
## 성능 검증 체크리스트

### 1. 응답 시간
- [ ] API 응답 시간 목표 달성 (예: 200ms 이하)
- [ ] DB 쿼리 최적화됨 (N+1 쿼리 없음)
- [ ] 캐싱 적절히 적용됨

### 2. 리소스 사용
- [ ] 메모리 사용량 적절함
- [ ] CPU 사용률 정상 범위
- [ ] 동시 접속 목표 달성

### 3. 부하 테스트
- [ ] 예상 트래픽 처리 가능
- [ ] 장애 조치 동작 확인
```

**실행 예시**:

```bash
# 1. 부하 테스트 (예: k6)
k6 run load-test.js

# 2. 프로파일링
node --prof app.js
node --prof-process isolate-*.log

# 3. 메모리 분석
node --inspect app.js
```

#### 6.1.2 문서 품질 검증

Claude가 생성한 문서의 품질을 검증하는 방법입니다.

**체크리스트**:

```markdown
## 문서 품질 검증

### 1. 완성도
- [ ] 목차와 실제 내용 일치
- [ ] 모든 섹션 작성 완료
- [ ] 예시 코드 포함됨
- [ ] 링크 모두 유효함

### 2. 정확성
- [ ] 기술 정보 정확함
- [ ] 코드 예시 동작함
- [ ] 버전 정보 최신임
- [ ] 용어 일관성 있음

### 3. 가독성
- [ ] 논리적 구조
- [ ] 적절한 분량 (너무 길거나 짧지 않음)
- [ ] 표, 다이어그램 활용
- [ ] 문법, 맞춤법 정확

### 4. 유용성
- [ ] 대상 독자 명확
- [ ] 실용적 예시
- [ ] 문제 해결 중심
- [ ] 다음 단계 안내
```

**검증 프로세스**:

1. **자동 검증**
   ```bash
   # 링크 체크
   npx markdown-link-check docs/**/*.md
   
   # 맞춤법 검사 (한글)
   # (적절한 도구 사용)
   
   # 코드 블록 실행 가능 여부
   # (문서의 코드 예시 추출 후 실행)
   ```

2. **수동 검증**
   ```markdown
   - 처음부터 끝까지 읽어보기
   - 코드 예시 직접 실행해보기
   - 대상 독자 입장에서 이해 가능한지 확인
   - 동료에게 리뷰 요청
   ```

3. **피드백 수집**
   ```markdown
   - 문서 하단에 피드백 섹션 추가
   - 실제 사용자 의견 수집
   - 주기적으로 업데이트
   ```

**참고 자료**:
- [docs/checklists/documentation.md](checklists/documentation.md) - 상세 문서 검증 체크리스트

### 6.2 일반적인 실수와 해결책

Claude 사용시 자주 발생하는 실수 패턴과 예방법입니다.

#### 실수 1: 파일 재읽기 (토큰 낭비)

**증상**:
```
대화 중간에 토큰 사용량이 급증
같은 파일을 여러 번 읽는 경우
```

**원인**:
```
"src/auth/login.ts 다시 읽어줘" 같은 요청
이전에 읽은 파일을 다시 참조해야 한다고 생각
```

**해결책**:
```
이미 읽은 파일 목록 추적:
- 대화 시작시 "읽을 파일 목록" 작성
- 읽은 후 체크
- 재읽기 전에 목록 확인

프롬프트 예시:
"src/auth/login.ts (이미 읽음)의 45번째 줄 수정해줘"
```

**토큰 절감**:
- Bad: 파일 재읽기 × 3회 = 6,000 토큰 낭비
- Good: 한 번만 읽기 = 2,000 토큰
- 절감율: 67%

---

#### 실수 2: 너무 큰 범위 요청 (과다 탐색)

**증상**:
```
"전체 프로젝트 스캔"
"모든 파일 분석"
토큰: 50,000+
결과: 느리고 불완전
```

**원인**:
```
문제 위치를 모르는 상태에서 전체 탐색 요청
막연한 개선 요청
```

**해결책**:
```
단계적 접근:
1. 문제 위치 좁히기 (grep, 로그 확인)
2. 관련 파일만 분석
3. 필요한 경우만 확장

프롬프트 예시:
Before: "프로젝트에서 성능 문제 찾아줘"
After: "API 응답 느린 /api/products 엔드포인트 분석"
```

**토큰 절감**:
- Bad: 전체 스캔 = 50,000 토큰
- Good: 타겟 분석 = 3,000 토큰
- 절감율: 94%

#### 실수 4: 모호한 요청 (재작업 발생)

**증상**:
```
"이거 좀 개선해줘"
Claude의 결과가 의도와 다름
다시 설명하고 재작업
```

**원인**:
```
목표가 불명확
성공 기준 없음
제약 조건 미명시
```

**해결책**:
```
명확한 요청:
- 무엇을 (What)
- 왜 (Why)
- 어떻게 (How)
- 성공 기준 (Done)

예시:
Bad: "로그인 개선해줘"
Good: "로그인 에러 메시지를 더 구체적으로 (현재: '로그인 실패' → 변경: '이메일 또는 비밀번호 불일치')"
```

**재작업 감소**:
- Before: 재작업 40%
- After: 재작업 5%
- 개선: 8배

#### 실수 5: 한 번에 너무 많이 요청 (실패 위험)

**증상**:
```
"기능 A, B, C, D 모두 구현해줘"
결과: 일부만 구현됨, 품질 낮음
```

**원인**:
```
큰 작업을 분할하지 않음
한 번에 끝내려는 조급함
```

**해결책**:
```
단계별 진행:
1. 작업 분할 (각 단계 5,000 토큰 이하)
2. 순차 실행
3. 각 단계 검증
4. 다음 단계로

예시:
Step 1: 기능 A 구현 → 테스트 → 커밋
Step 2: 기능 B 구현 → 테스트 → 커밋
...
```

**성공률 향상**:
- Before: 성공률 50% (일부만 완성)
- After: 성공률 95% (모두 완성)
- 개선: 2배

3. 1주일에 1회 이하로 수정
4. 수정시 하위 호환성 고려

예시:
임시: "이번 작업에서만 X 규칙 적용"
영구: 3회 이상 반복된 규칙만 CLAUDE.md 추가

#### 실수 11: 독립적 작업 순차 실행 (시간 낭비)

```
병렬 처리 가능 여부 미파악

```

**해결책**:
```
독립적 작업은 한 번에 요청:

Bad (순차):
"src/auth.ts 읽어줘"
[대기]
"src/user.ts 읽어줘"
[대기]
"src/api.ts 읽어줘"
시간: 30초

Good (병렬):
"다음 파일들 읽어줘:
- src/auth.ts
- src/user.ts
- src/api.ts"
시간: 10초

병렬 가능한 작업:
- 여러 파일 읽기
- 여러 grep 검색
- 독립적인 테스트 실행
- 여러 파일 수정 (서로 의존성 없는 경우)
```

**시간 절감**:
- Before: 순차 실행 3분
- After: 병렬 실행 1분
- 절감율: 67%
