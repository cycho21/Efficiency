### 2.3 캐시 히트율 최적화 전략

이제 캐싱 원리를 이해했으니, 실전에서 캐시 히트율 70% 이상을 달성하는 구체적인 전략을 알아봅시다.

**목표 설정**:
- 70% 이상: 우수 (이 섹션의 목표)
- 50-70%: 양호 (개선 여지 있음)
- 50% 미만: 개선 필요 (즉시 조치)

**섹션 구성**:
1. 캐시 친화적 프로젝트 구조
2. 세션 관리 전략
3. CLAUDE.md 최적화
4. 측정 및 모니터링

---

#### 캐시 친화적 프로젝트 구조

프로젝트의 파일 구조와 관리 방식이 캐시 히트율에 큰 영향을 줍니다.

##### 설정 파일 안정화

**원칙**: 자주 변경하지 않는 설정 구조를 만들어라.

**Bad 구조** (캐시 무효화 빈번):

```typescript
// config.ts (모든 설정이 한 파일에)
export const config = {
  // 안정적 설정
  app: {
    name: 'MyApp',
    version: '1.0.0',
  },
  
  // 자주 변경되는 설정
  features: {
    newFeatureA: true,  // ← 매일 추가/변경
    newFeatureB: false, // ← 실험 중
    betaFeatureC: true, // ← 자주 토글
  },
  
  // 안정적 설정
  api: {
    baseUrl: 'https://api.example.com',
    timeout: 5000,
  },
};

// 문제: features를 수정할 때마다 전체 파일 캐시 무효화
// 영향: api, app 설정도 캐시 미스
```

**Good 구조** (캐시 히트 최대화):

```typescript
// config/app.ts (안정적 - 거의 변경 안 함)
export const appConfig = {
  name: 'MyApp',
  version: '1.0.0',
};

// config/api.ts (안정적 - 거의 변경 안 함)
export const apiConfig = {
  baseUrl: 'https://api.example.com',
  timeout: 5000,
};

// config/features.ts (동적 - 자주 변경됨)
export const featureFlags = {
  newFeatureA: true,  // 자주 변경
  newFeatureB: false, // 실험 중
  betaFeatureC: true, // 자주 토글
};

// 장점: features만 변경해도 app, api는 캐시 유지
```

**측정 가능한 효과**:

```
시나리오: 하루 10번 요청, features를 하루 3번 수정

Bad 구조 (단일 파일):
- config.ts: 1,000 토큰
- 3번 전체 로드 (3,000 토큰) + 7번 캐시 (700 토큰)
- 합계: 3,700 토큰/일

Good 구조 (분리):
- app.ts (200 토큰): 1번 전체 + 9번 캐시 = 380 토큰
- api.ts (300 토큰): 1번 전체 + 9번 캐시 = 570 토큰
- features.ts (500 토큰): 3번 전체 + 7번 캐시 = 1,850 토큰
- 합계: 2,800 토큰/일

→ 절감: 900 토큰/일 (24% 절감)
→ 월간: 27,000 토큰 절감
```

##### 공통 코드 분리

**원칙**: 자주 참조되는 코드를 안정적인 별도 파일로 분리하라.

**Before** (캐싱 비효율):

```typescript

  private validateEmail(email: string): boolean {
    return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email);

  // 공통 유틸리티 중복 (안정적)

// - 공통 유틸이 여러 파일에 중복
```

**After** (캐싱 최적화):

```
- feature 파일 5개: 7,500 토큰 (유틸 제외, 더 작음)
- 합계: 9,150 토큰

→ 절감: 850 토큰 (8.5% 절감)
→ 추가 장점: 코드 품질 향상, 유지보수 개선
```

#### 세션 관리 전략

세션을 어떻게 구성하고 관리하느냐가 캐시 효율성을 크게 좌우합니다.

**원칙**: 같은 컨텍스트를 공유하는 작업은 한 세션에서 처리하라.

```
사용자: "user.service.ts의 updateUser 메서드 개선해줘"
Claude: [CLAUDE.md 로드 (2,000), user.service.ts 로드 (1,500)]
       ← 세션 1과 똑같은 파일을 다시 로드!
```

##### 세션 시작시 캐시 워밍 (Cache Warming)

**원칙**: 세션 초반에 필요한 파일을 미리 로드하여 캐시를 구축하라.

**캐시 워밍이란?**
작업 시작 시 핵심 파일들을 미리 로드하여 캐시를 생성하는 전략입니다. 이후 작업에서 이 캐시를 반복 활용하여 토큰을 절감합니다.

**두 가지 접근법**:

1. **계획적 워밍** (필요한 파일을 아는 경우)
   - 작업 시작 전 필요 파일 목록 작성
   - 세션 초반에 한 번에 로드
   - 이후 작업에서 캐시 활용

2. **점진적 워밍** (탐색적 작업의 경우)
   - 핵심 파일(CLAUDE.md, 설정)부터 먼저 로드
   - 작업하면서 추가 파일 발견 시 로드
   - 자연스럽게 캐시 구축

**계획적 워밍 예시**:

**Bad 패턴** (파일을 필요할 때마다 로드):

```
세션 시작 (10:00):

사용자: "Order 기능을 구현하려고 해"
Claude: "네, 어떻게 도와드릴까요?"

Claude: [필요할 때마다 파일 로드]
       [types/order.ts 로드 (500)]

사용자: "이제 order.controller.ts 만들어줘"
Claude: [types/order.ts 재로드? 또는 기억 의존]

사용자: "order 관련 테스트도 작성해줘"
Claude: [이미 작성한 코드를 다시 읽어야 할 수도...]

→ 파일을 여러 번 나눠서 로드
→ 일부는 캐시, 일부는 재로드
→ 비효율적
```

**Good 패턴** (계획적 캐시 워밍):

```

       "구조를 파악했습니다. [요약]"

사용자: "이제 order.service.ts 만들어줘"
Claude: [types/order.ts 캐시 (50)]

       [order.service.ts 생성]

사용자: "테스트 작성해줘"

       [order.test.ts 생성]

→ 초기 투자: 1,700 토큰
→ 이후 작업: 캐시 활용으로 토큰 최소화

```

##### 캐시 히트율 계산법

**이미 섹션 1.2에서 설명했습니다.** 여기서는 실전 적용에 초점을 맞춥니다.

**공식** (재확인):
```
캐시 히트율 = (캐시 히트 토큰 수) / (전체 입력 토큰 수) × 100%
```

**플랫폼별 측정 방법**:

**Claude API 사용시**:

API 응답에서 직접 확인 가능합니다.

```json
{
  "usage": {
    "input_tokens": 10000,
    "cache_creation_input_tokens": 2000,
    "cache_read_input_tokens": 8000,
    "output_tokens": 500
  }
}
```

**참고**: 위 JSON 구조는 예시이며 실제 API 응답 형식은 다를 수 있습니다 (2025년 1월 API 기준). 최신 형식은 공식 API 문서를 참조하세요.

```
계산:
- 전체 입력: 10,000 토큰
- 캐시 생성: 2,000 토큰 (새로 캐시됨)
- 캐시 읽기: 8,000 토큰 (캐시 히트)
- 캐시 히트율: 8,000 / 10,000 = 80% ✅
```

**Claude Code CLI 사용시**:

현재 자동 측정 기능은 없으므로 간단한 추정으로 충분합니다.

```
간단한 추정 방법:

1. 세션에서 반복 사용된 주요 파일 파악
   예:
   - CLAUDE.md: 2,000 토큰 (매 요청마다 사용)
   - 설정 파일들: 1,000 토큰 (자주 참조)
   - 공통 유틸: 800 토큰 (작업 중 반복 사용)

2. 대략적인 요청 패턴 추정
   예: 총 10번 요청, 5분 내 연속 작업

3. 간략 계산
   - 반복 사용된 콘텐츠: 약 3,800 토큰
   - 첫 로드: 3,800 토큰 (100%)
   - 이후 9번: 약 380 토큰/회 (10%)
   - 캐시 히트율: 약 80% 추정

**팁**: 정확한 수치보다 "대부분 캐시되고 있는가?"를 파악하는 것이 중요합니다. 
관련 작업을 한 세션에 묶고, CLAUDE.md를 자주 수정하지 않으면 자연스럽게 높은 히트율을 달성할 수 있습니다.
```

```
- 개선 여지 있음
- 구조나 세션 관리 최적화 권장
```

- 프로토타입 개발: 빠른 변경으로 히트율이 낮을 수 있음 (수용 가능)

```markdown
   - 작업 완료 후 휴식
   - 5분 미만 휴식은 세션 유지

```

**실전 개선 예시**:

```
Before (히트율 35%):

프로젝트 상태:
- CLAUDE.md: 3,000 토큰, 하루 3번 수정
- 세션: 하루 10개, 평균 5분
- 파일: 대형 파일 5개, 공통 코드 혼재

측정:
- 주간 100번 요청
- 전체 토큰: 약 400,000
- 캐시 히트: 약 140,000 (35%)

After (히트율 75%):

개선 조치:
1. CLAUDE.md 분리
   - 불변 규칙만 유지 (주 1번 미만 수정)
   - 동적 정보는 tasks.md로

2. 세션 재구성
   - 관련 작업 묶음 (하루 3-4개 세션)
   - 세션 시작시 계획적 파일 로드

3. 구조 개선
   - utils/ 분리 (안정적)
   - 설정 파일 모듈화

측정:
- 주간 100번 요청
- 전체 토큰: 약 150,000
- 캐시 히트: 약 112,500 (75%)

결과:
- 히트율: 35% → 75% (+40%p)
- 토큰 절감: 62.5%
- 비용 절감: 약 60%
```
