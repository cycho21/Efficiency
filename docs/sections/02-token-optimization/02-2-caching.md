### 2.2 컨텍스트 캐싱 이해하기

컨텍스트 캐싱은 Claude 사용 비용을 극적으로 줄일 수 있는 핵심 메커니즘입니다. 캐시된 토큰은 원래 비용의 10%만 부과되므로 (2025년 1월 기준, 실제 비용은 공식 문서를 참조하세요), 효과적인 캐싱 전략은 90%의 비용 절감을 의미합니다.

**플랫폼별 캐싱 지원**:
- **Claude API**: 프롬프트 캐싱 완전 지원 (개발자가 직접 제어)
- **Claude Code CLI**: 자동 캐싱 지원 (시스템이 자동 관리)

**중요**: 이 섹션의 원칙(캐시 수명, 무효화 조건, 최적화 전략)은 보편적으로 적용되지만, 구체적인 구현 세부사항(캐시 수명, API 응답 형식 등)은 플랫폼마다 다를 수 있습니다. Claude Code CLI는 자동으로 캐싱을 관리하므로 개발자는 원칙에만 집중하면 됩니다.

---

#### 캐싱 작동 원리

##### 무엇이 캐시되는가

Claude는 다음 요소들을 자동으로 캐시합니다:

**1. 시스템 프롬프트 (System Prompt)**

가장 중요한 캐싱 대상입니다. CLAUDE.md 같은 프로젝트 설정이 여기에 포함됩니다.

```
예시:
첫 요청: CLAUDE.md 읽기 (2,000 토큰)
→ 비용: 2,000 토큰 (100%)

5분 내 두 번째 요청: 같은 CLAUDE.md 참조
→ 비용: 200 토큰 (10%)
→ 절감: 1,800 토큰 (90%)
```

**2. Tool 결과 (Claude Code CLI)**

파일 읽기, 검색 등 Tool 실행 결과가 캐시됩니다.

```
예시:
첫 요청: Read user.service.ts (1,500 토큰)
→ 비용: 1,500 토큰 (100%)

5분 내 같은 파일 재요청:
→ 비용: 150 토큰 (10%)
→ 절감: 1,350 토큰 (90%)
```

**3. 대화 히스토리 일부**

이전 대화의 일부가 컨텍스트로 유지되며 캐시됩니다.

**캐싱되지 않는 것**:
- 사용자의 새로운 프롬프트 (매번 다름)
- Claude의 새로운 응답 (생성되는 내용)
- 변경된 파일 내용

---

##### 캐시 수명 (Cache Lifetime)

**기본 수명**: 5분 (Claude API 기준, 플랫폼에 따라 다를 수 있음)

```
Timeline 예시:

00:00 - 첫 요청, CLAUDE.md 로드 (캐시 생성)
00:30 - 두 번째 요청 (캐시 히트 ✅, 5분 연장)
02:00 - 세 번째 요청 (캐시 히트 ✅, 5분 연장)
04:00 - 네 번째 요청 (캐시 히트 ✅, 5분 연장)
10:00 - 다섯 번째 요청 (6분 경과, 캐시 미스 ❌, 새 캐시 생성)
```

**중요**: 
- 캐시는 마지막 사용으로부터 5분간 유지됩니다
- 5분 내에 재사용하면 캐시 수명이 연장됩니다
- 캐시가 살아있는 동안 계속 사용하면 비용을 최소화할 수 있습니다

**실전 활용**:
```
❌ 비효율적 패턴:
- 작업 시작 → 10분 휴식 → 다시 시작
  (캐시 만료로 처음부터 다시)

✅ 효율적 패턴:
- 관련 작업을 묶어서 한 번에
- 5분 내에 다음 요청 이어가기
- 긴 휴식 전에 작업 마무리
```

---

##### 캐시 키 (Cache Key)

**핵심 원칙**: 정확히 동일한 내용만 캐시 히트

캐시 키는 콘텐츠의 해시값처럼 작동합니다. 즉, 정확히 동일한 내용만 매칭되며, 단 한 글자라도 다르면 완전히 다른 캐시로 인식됩니다.

**이것은 캐시 히트입니다**:
```markdown
<!-- 요청 1 -->
CLAUDE.md 내용:
"# 프로젝트 규칙
1. 이모지 사용 금지
2. 한글 커밋 메시지"

<!-- 요청 2 (5분 내) -->
CLAUDE.md 내용:
"# 프로젝트 규칙
1. 이모지 사용 금지
2. 한글 커밋 메시지"

→ 완전히 동일 ✅ → 캐시 히트
```

**이것은 캐시 미스입니다**:
```markdown
<!-- 요청 1 -->
CLAUDE.md 내용:
"# 프로젝트 규칙
1. 이모지 사용 금지
2. 한글 커밋 메시지"

<!-- 요청 2 -->
CLAUDE.md 내용:
"# 프로젝트 규칙
1. 이모지 사용 금지
2. 한글 커밋 메시지
3. 테스트 필수"  ← 한 줄 추가

→ 내용 변경 ❌ → 캐시 미스 (새 캐시 생성)
```

**민감도**:
- 공백 하나, 줄바꿈 하나, 마침표 하나만 달라져도 캐시 미스
- 파일 순서, 들여쓰기, 주석도 영향을 줍니다
- 대소문자도 구분됩니다

**실전 영향**:
```
시나리오: CLAUDE.md (2,000 토큰)를 하루에 10번 요청, 그 중 3번 수정

수정이 없었다면:
- 1번 × 2,000 토큰 (100%) = 2,000 토큰
- 9번 × 200 토큰 (10%) = 1,800 토큰
- 총 비용: 3,800 토큰

3번 수정 시 (캐시 무효화 발생):
- 4번 × 2,000 토큰 (첫 로드 + 3번 재로드) = 8,000 토큰
- 6번 × 200 토큰 (캐시 히트) = 1,200 토큰
- 총 비용: 9,200 토큰

→ 2.4배 증가 (140% 비용 증가)
```

---

#### 캐시 가능한 요소 목록

어떤 요소가 캐싱에 적합한지, 그리고 어떻게 최적화할지 알아봅시다.

##### 1. CLAUDE.md (최우선순위)

**왜 중요한가**:
- 모든 세션에서 로드됨
- 크기가 큼 (평균 1,000-3,000 토큰)
- 변경 빈도가 낮음 (잘 설계된 경우)
- 캐싱 효과가 가장 큼

**최적화 전략**:

```markdown
❌ 비효율적 CLAUDE.md:
# 프로젝트

## 할 일 목록
- [ ] 버그 수정 (#123)  ← 매일 변경됨
- [ ] 기능 추가 (#124)  ← 자주 업데이트

## 최근 변경사항
- 2026-04-12: API 엔드포인트 추가  ← 매일 추가됨
- 2026-04-11: 데이터베이스 스키마 변경

→ 자주 변경되어 캐시 무효화 반복
```

```markdown
✅ 효율적 CLAUDE.md:
# 프로젝트

## 불변 규칙 (안정적)
1. 이모지 사용 금지
2. 한글 커밋 메시지
3. TypeScript strict 모드

## 아키텍처 (안정적)
- src/api: REST API
- src/services: 비즈니스 로직
- src/utils: 공통 유틸리티

## 작업 추적
→ 별도 파일 참조: .claude/tasks.md  ← 동적 내용 분리

→ 핵심 규칙만 포함, 자주 변경되지 않음
```

**측정 가능한 효과**:
```
시나리오: 하루 20번 요청, CLAUDE.md 2,000 토큰

비효율적 CLAUDE.md (하루 5번 수정):
- 5번 × 2,000 토큰 (100%) = 10,000
- 15번 × 200 토큰 (10%) = 3,000
- 합계: 13,000 토큰

효율적 CLAUDE.md (주 1번 수정):
- 1번 × 2,000 토큰 (100%) = 2,000
- 19번 × 200 토큰 (10%) = 3,800
- 합계: 5,800 토큰

→ 절감: 7,200 토큰/일 (55% 절감)
→ 월간: 216,000 토큰 절감
```

```
- 참조 빈도: 높음 (구조 파악, 스크립트 확인)

- 변경 빈도: 낮음 (규칙 추가시만)
- 참조 빈도: 높음 (코드 스타일 검증)
```

```
다음 요청 (5분 내):

Claude의 내부 워크플로우:
1. package.json 참조 (100 토큰, 캐시 히트 ✅)
2. tsconfig.json 참조 (30 토큰, 캐시 히트 ✅)
3. 테스트 작성...

절감: 1,170 토큰 (90%)
```

##### 3. 변경되지 않는 라이브러리/유틸 코드

**캐싱 효과가 큰 코드**:

```typescript
// ✅ utils/date.ts (안정적인 유틸리티)
// 한 번 작성 후 거의 변경되지 않음
// 여러 작업에서 반복 참조됨
// 크기: ~500 토큰

export function formatDate(date: Date): string {
  return date.toISOString().split('T')[0];
}

export function addDays(date: Date, days: number): Date {
  const result = new Date(date);
  result.setDate(result.getDate() + days);
  return result;
}

// 이런 파일은 캐싱 효과 극대화
```

```typescript
// ❌ services/order.service.ts (자주 변경되는 비즈니스 로직)
// 새 기능 추가로 계속 수정됨
// 캐싱 효과 낮음

export class OrderService {
  async createOrder() { ... }
  async updateOrder() { ... }
  // 매일 새 메서드 추가/수정
}
```

**실전 전략**:

```
프로젝트 구조 최적화:

src/
├── utils/           ← 안정적, 캐싱 효과 높음
│   ├── date.ts
│   ├── string.ts
│   └── validation.ts
├── config/          ← 안정적, 캐싱 효과 높음
│   ├── database.ts
│   └── api.ts
├── types/           ← 비교적 안정적
│   ├── user.ts
│   └── order.ts
└── features/        ← 자주 변경, 캐싱 효과 낮음
    ├── auth/
    └── orders/

세션 전략:
1. 안정적 파일 먼저 로드 (utils, config) → 캐싱
2. 작업 중인 feature 파일 로드
3. 같은 세션 내에서 feature 작업 완료
→ 안정적 파일은 캐시 유지, feature만 새로 읽기
```

**측정 가능한 효과**:

```
시나리오: 5개 기능 구현, 각 기능마다 utils 3개 참조

캐싱 없이:
- utils 참조: 5번 × 3개 × 500 토큰 = 7,500 토큰

캐싱 활용:
- 첫 참조: 3개 × 500 토큰 = 1,500 토큰 (100%)
- 나머지 참조: 12번 × 50 토큰 = 600 토큰 (10%)
- 합계: 2,100 토큰

→ 절감: 5,400 토큰 (72% 절감)
```

#### 캐시 무효화 조건

캐시가 언제 무효화되는지 이해하면, 무효화를 피하고 캐시 히트율을 높일 수 있습니다.

##### 1. 파일 내용 변경시

**가장 흔한 무효화 원인**입니다.

```
예시 시나리오:

10:00 - user.service.ts 읽기 (1,500 토큰, 캐시 생성)
10:05 - 같은 파일 참조 (150 토큰, 캐시 히트 ✅)
10:07 - 파일 수정 (새 메서드 추가)
10:10 - 같은 파일 참조 (1,500 토큰, 캐시 미스 ❌)
        → 내용이 변경되어 새 캐시 생성
```

**영향 분석**:

```
변경 빈도별 캐시 효과:

주 1회 변경 (안정적):
- 주간 50번 요청
- 1번 전체 로드 (100%) + 49번 캐시 (10%)
- 평균 비용: 14.8% ⭐⭐⭐⭐⭐

일 1회 변경 (보통):
- 주간 50번 요청 (하루 평균 7번)
- 7번 전체 로드 (100%) + 43번 캐시 (10%)
- 평균 비용: 22.6% ⭐⭐⭐⭐

일 5회 변경 (잦음):
- 주간 50번 요청
- 35번 전체 로드 (100%) + 15번 캐시 (10%)
- 평균 비용: 73% ⭐⭐

시간당 변경 (매우 잦음):
- 주간 50번 요청
- 50번 전체 로드 (거의 매번 변경)
- 평균 비용: 100% ⭐ (캐싱 효과 없음)
```

**최적화 전략**:

```markdown
✅ Do: 변경 빈도 낮추기

1. 안정적인 부분과 동적 부분 분리
   - CLAUDE.md: 불변 규칙
   - tasks.md: 동적 할 일 목록

2. 배치 업데이트
   - 작은 변경을 여러 번 → ❌
   - 관련 변경을 모아서 한 번에 → ✅

3. 업데이트 타이밍 조절
   - 세션 중간에 CLAUDE.md 수정 → ❌
   - 세션 종료 후 수정 → ✅
```

##### 2. 5분 경과시

**시간 기반 만료**는 자동으로 발생합니다.

```
Timeline 예시:

10:00 - CLAUDE.md 로드 (2,000 토큰, 캐시 생성)
       캐시 만료 시각: 10:05

10:04 - CLAUDE.md 참조 (200 토큰, 캐시 히트 ✅)
       캐시 만료 시각: 10:09 (연장)

10:08 - CLAUDE.md 참조 (200 토큰, 캐시 히트 ✅)
       캐시 만료 시각: 10:13 (연장)

[작업 중단, 15분 휴식]

10:25 - CLAUDE.md 참조 (2,000 토큰, 캐시 미스 ❌)
       → 10:13에 만료되어 새로 로드
```

**영향 분석**:

```
작업 패턴별 영향:

집중 작업 (5분 내 연속):
- 10번 요청, 모두 5분 내
- 1번 전체 + 9번 캐시
- 비용: 19% ⭐⭐⭐⭐⭐

중간 휴식 (10분마다 휴식):
- 10번 요청, 5번씩 묶음
- 2번 전체 + 8번 캐시
- 비용: 28% ⭐⭐⭐⭐

빈번한 중단 (요청 간 10분 이상):
- 10번 요청, 매번 만료
- 10번 전체 + 0번 캐시
- 비용: 100% ⭐ (캐싱 효과 없음)
```

**최적화 전략**:

```
✅ Do: 작업 패턴 최적화

1. 관련 작업 묶기
   "user 기능 3개를 한 세션에 완료"
   → 캐시 유지

2. 긴 휴식은 작업 구분점에
   "기능 A 완료 → 휴식 → 새 세션으로 기능 B"
   → 어차피 새 컨텍스트

3. 짧은 확인은 5분 내에
   "코드 작성 → 바로 리뷰 요청"
   → 캐시 활용

❌ Don't: 캐시 불친화적 패턴

- 요청 → 10분 휴식 → 요청 → 10분 휴식
  (매번 캐시 만료)

- 관련 없는 작업을 하나의 긴 세션에
  (캐시는 유지되지만 컨텍스트 오염)
```

---

##### 3. 세션 종료시

**세션이 끝나면 캐시도 초기화됩니다.**

```
세션 A:
10:00 - CLAUDE.md 로드 (2,000 토큰, 캐시 생성)
10:05 - 작업 완료, 세션 종료
       → 세션 A의 캐시 소멸

세션 B (새 세션):
11:00 - CLAUDE.md 로드 (2,000 토큰, 새 캐시 생성)
       → 세션 A의 캐시는 사용 불가
```

**세션 간 캐시 공유 안 됨**:
- 각 세션은 독립적인 캐시를 가집니다
- 세션 A에서 캐시한 내용을 세션 B에서는 사용할 수 없습니다
- 새 세션 시작시 항상 처음부터 캐시를 구축합니다

**영향 분석**:

```
시나리오: CLAUDE.md 2,000 토큰

긴 세션 (하루 1개 세션):
- 1번 전체 로드 (2,000 토큰)
- 19번 캐시 사용 (3,800 토큰)
- 합계: 5,800 토큰
- 평균: 290 토큰/요청 ⭐⭐⭐⭐⭐

짧은 세션 (하루 5개 세션, 각 4번 요청):
- 5번 전체 로드 (10,000 토큰)
- 15번 캐시 사용 (3,000 토큰)
- 합계: 13,000 토큰
- 평균: 650 토큰/요청 ⭐⭐
- 증가: 124% 더 많은 비용
```

**최적화 전략**:

```
✅ Do: 세션 길이 최적화

1. 관련 작업을 한 세션에
   예: "user 모듈 전체 리팩토링을 한 세션에"
   - 1번 설정 로드로 전체 작업 커버

2. 적절한 세션 분할
   예: "기능별로 세션 분리"
   - auth 기능: 세션 A
   - order 기능: 세션 B
   - 각 세션 내에서 캐시 최대 활용

3. 세션 시작시 워밍업
   예: "필요한 설정 파일을 먼저 로드"
   - package.json, CLAUDE.md 등을 세션 초반에 로드
   - 이후 작업에서 캐시 활용

❌ Don't: 비효율적 세션 관리

- 작은 질문마다 새 세션
  → 매번 설정 재로드

- 관련 없는 작업을 억지로 한 세션에
  → 컨텍스트 오염, 품질 저하

- 세션 중간에 불필요하게 재시작
  → 캐시 손실
```

**세션 종료 체크리스트**:

```
세션 종료 전 확인:
[ ] 진행 중인 작업 완료했는가?
[ ] 중요 정보를 파일/커밋으로 저장했는가?
[ ] 다음 세션에서 이어서 해야 하는가?
    → Yes: 잠시 기다렸다가 같은 세션에서 계속
    → No: 안심하고 세션 종료
```
