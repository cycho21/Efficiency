## 5. 워크플로우 패턴

효율적인 워크플로우는 개인의 생산성을 높이고 팀 전체의 일관성을 보장합니다.

이 섹션에서는 실전에서 검증된 워크플로우 패턴을 다룹니다.

### 섹션 개요

- **5.1 개인 워크플로우**: 일일 시작, 대형 작업 관리, 컨텍스트 압축 타이밍
- **5.2 팀 협업 워크플로우**: CLAUDE.md 표준화, PR 리뷰, 공유 템플릿, 모니터링

**핵심 포인트**: 좋은 워크플로우는 반복적인 작업을 자동화하고, 실수를 방지하며, 팀의 베스트 프랙티스를 자연스럽게 적용하게 만듭니다.

---

### 5.1 개인 워크플로우

개인 사용자가 Claude를 효율적으로 활용하기 위한 일상적인 작업 패턴입니다.

#### 5.1.1 일일 시작 루틴

매일 작업을 시작할 때 따르는 체크리스트입니다.

**목표**:
- 작업 환경을 빠르게 설정
- 불필요한 컨텍스트 로드 방지
- 캐시 효율성 최대화

**체크리스트**:

1. **작업 범위 확인** (5분)
   ```
   오늘 할 작업:
   - [ ] 구체적인 파일/모듈 식별
   - [ ] 예상 토큰 사용량 추정
   - [ ] 대형 작업인지 판단 (10,000+ 토큰)
   ```

2. **프로젝트 설정 점검** (2분)
   ```
   설정 확인:
   - [ ] CLAUDE.md 최신 상태인가?
   - [ ] .claudeignore에 빌드 결과물 포함되었나?
   - [ ] hooks 설정이 활성화되어 있나?
   ```

3. **첫 프롬프트 작성** (3분)
   ```
   명확한 시작:
   - [ ] 구체적인 목표 명시
   - [ ] 필요한 파일만 지정
   - [ ] 제외할 범위 명시
   ```

**예시 - 좋은 시작**:

```
오늘 작업: src/auth/login.ts의 에러 핸들링 개선

1단계: 현재 에러 처리 코드 리뷰
- src/auth/login.ts 읽기
- 에러 케이스 목록 작성
- 개선이 필요한 부분 식별

목표: 리뷰 결과만 요약 (코드 수정은 아직 안 함)
```

**나쁜 시작**:
```
인증 시스템 전체를 개선해줘
```

**토큰 절감 효과**:
- Good: 500-1,000 토큰 (명확한 범위)
- Bad: 5,000-10,000 토큰 (불필요한 탐색)
- 절감율: 80-90%

---

#### 5.1.2 대형 작업 관리

10,000+ 토큰이 예상되는 큰 작업을 효율적으로 처리하는 방법입니다.

**작업 분할 전략**:

1. **작업 크기 추정**
   ```
   예상 토큰 계산:
   - 읽어야 할 파일 수 × 평균 파일 크기
   - 분석 + 구현 + 검증 오버헤드
   - 총 예상치가 10,000 초과시 분할 고려
   ```

2. **단계 정의**
   ```
   큰 작업을 작은 단계로:
   - 각 단계는 독립적으로 검증 가능
   - 각 단계는 5,000 토큰 이하
   - 단계 간 명확한 인터페이스
   ```

3. **단계별 실행**
   ```
   각 단계마다:
   1. 목표 명시
   2. 구현
   3. 검증 (테스트 실행)
   4. 커밋
   5. 다음 단계로
   ```

**예시: 새로운 결제 시스템 구현**

**잘못된 접근** (한 번에 전부):
```
새로운 결제 시스템 구현:
- Stripe 연동
- 환불 처리
- 관리자 대시보드
- 이메일 알림
- 결제 히스토리
```

**예상 토큰**: 30,000-50,000 (실패 위험 높음)

**올바른 접근** (단계별):

**Step 1: 기본 결제 플로우** (3,000 토큰)
```
Stripe Payment Intent 생성 및 확인
- 파일: src/payment/stripe.service.ts
- 기능: 결제 생성, 확인 콜백
- 테스트: 테스트 모드 결제 성공 확인
```

**Step 2: 환불 처리** (2,500 토큰)
```
환불 기능 추가
- 파일: src/payment/refund.service.ts
- 기능: 전액/부분 환불
- 테스트: 환불 시나리오 테스트
```

**Step 3: 데이터 저장** (2,000 토큰)
```
결제 히스토리 저장
- 파일: src/payment/history.repository.ts
- 기능: DB 스키마, CRUD
- 테스트: 히스토리 조회 테스트
```

**Step 4: 관리자 UI** (4,000 토큰)
```
관리자 대시보드
- 파일: src/admin/payment-dashboard.tsx
- 기능: 결제 목록, 환불 처리
- 테스트: UI 렌더링 테스트
```

**Step 5: 알림** (2,500 토큰)
```
이메일 알림
- 파일: src/notification/payment-email.service.ts
- 기능: 결제 성공/실패 이메일
- 테스트: 이메일 발송 테스트
```

**총 토큰**: 14,000 (각 단계 검증됨, 품질 높음)

**장점**:
- 각 단계마다 검증 가능
- 문제 발생시 롤백 쉬움
- 점진적 통합으로 리스크 감소
- 중간에 방향 수정 가능

#### 5.1.3 컨텍스트 압축 타이밍

언제 컨텍스트를 압축하거나 새 대화를 시작해야 하는지 판단하는 기준입니다.

**신호 1: 토큰 사용량 증가**

```
컨텍스트 압축 필요 신호:
- 현재 대화의 토큰이 60% 이상
- 같은 정보를 반복해서 언급
- Claude의 응답이 이전 컨텍스트를 놓치는 경우
```

**행동**:
1. 현재 상태 저장 (progress.md 작성)
2. 중요한 결정사항 기록
3. 새 대화 시작
4. progress.md 참조하여 컨텍스트 복원

**신호 2: 작업 전환**

```
새 대화 시작이 필요한 경우:
- 완전히 다른 작업으로 전환
- 이전 컨텍스트가 더 이상 필요 없음
- 새 작업이 독립적
```

**행동**:
1. 현재 작업 마무리 (커밋)
2. 새 대화 시작
3. 새 작업의 목표만 로드

**신호 3: 반복적인 오류**

```
컨텍스트 리셋 필요 신호:
- Claude가 같은 실수를 반복
- 이전 잘못된 접근을 계속 참조
- 문제의 원인이 컨텍스트 혼란
```

**행동**:
1. 현재 대화 중단
2. 새 대화에서 깨끗하게 시작
3. 문제 상황만 명확히 설명

**예시 - 컨텍스트 압축**:

**Before 압축** (60,000 토큰 누적):
```
[대화 1] 기능 A 구현 (20,000 토큰)
[대화 2] 기능 B 구현 (18,000 토큰)
[대화 3] 기능 A 수정 (15,000 토큰)
[대화 4] 기능 C 시작... (느려짐)
```

**progress.md 작성**:
```markdown
# 진행 상황 (2024-04-13)

## 완료된 작업
- 기능 A: src/features/a.ts 구현 완료 (커밋: abc123)
- 기능 B: src/features/b.ts 구현 완료 (커밋: def456)

## 진행 중인 작업
- 기능 C: 인증 플로우 추가 중

## 다음 단계
1. 기능 C: OAuth 콜백 핸들러 구현
2. 기능 C: 토큰 저장 로직 추가
3. 통합 테스트 작성

## 중요한 결정사항
- 인증 토큰은 httpOnly 쿠키에 저장 (XSS 방지)
- 리프레시 토큰은 7일 유효
```

**After 압축** (새 대화):
```
현재 작업: 기능 C (OAuth 인증) 구현 중

완료 상태:
- 기능 A, B는 이미 완료 (코드는 src/features/에 있음)

다음 할 일:
1. OAuth 콜백 핸들러 (src/auth/oauth-callback.ts)
2. 토큰 저장 (httpOnly 쿠키 사용)
3. 토큰 리프레시 로직 (7일 유효)

현재 파일: src/auth/oauth.service.ts (기본 구조만 있음)
```

**효과**:
- 토큰: 60,000 → 2,000 (97% 감소)
- 응답 속도: 느림 → 빠름
- 집중도: 낮음 → 높음

#### 5.1.4 실전 패턴: 일주일 작업 예시

**월요일 오전: 새 프로젝트 시작**
```
1. CLAUDE.md 작성 (10분)
2. .claudeignore 설정 (5분)
3. 전체 아키텍처 설계 (30분, 새 대화)
4. 첫 기능 구현 시작 (새 대화)
```

**월요일 오후 - 목요일: 기능 구현**
```
각 기능마다:
1. 새 대화 시작
2. 기능 구현
3. 테스트 작성
4. 커밋
5. 필요시 progress.md 업데이트
```

**금요일: 정리 및 리뷰**
```
1. 주간 작업 리뷰 (새 대화)
2. 코드 품질 점검
3. 문서 업데이트
4. 다음 주 계획
```

**토큰 사용 패턴**:
```
월요일: 15,000 토큰 (설계 + 첫 기능)
화-목: 각 10,000 토큰/일 (기능 구현)
금요일: 5,000 토큰 (리뷰 + 정리)

주간 총: 50,000 토큰
```

**최적화 전 (비효율적)**:
```
모든 작업을 하나의 긴 대화로:
- 토큰: 100,000+
- 속도: 느림
- 실수: 많음
```

**절감율: 50%**

### 5.2 팀 협업 워크플로우

팀 전체가 Claude를 효율적으로 사용하기 위한 협업 패턴입니다.

#### 5.2.1 CLAUDE.md를 팀 표준으로

**목표**:
- 모든 팀원이 동일한 가이드라인 따르기
- 프로젝트별 컨벤션 자동 적용
- 신규 팀원 온보딩 간소화

**단계**:

**1단계: 베이스 CLAUDE.md 작성**

프로젝트 루트에 `.claude/CLAUDE.md` 생성:

```markdown
# 프로젝트명 Claude 사용 가이드

## 토큰 최적화 규칙
1. 파일 재읽기 금지
2. 병렬 도구 호출 사용
3. 대형 출력은 서브에이전트에 위임

## 코딩 컨벤션
- TypeScript strict mode 사용
- 함수형 컴포넌트 + hooks (React)
- Jest + React Testing Library (테스트)
- ESLint + Prettier (코드 스타일)

## 프로젝트 구조
- src/features/ : 기능별 모듈
- src/shared/ : 공유 유틸리티
- src/components/ : 재사용 컴포넌트

## 테스트 정책
- 모든 비즈니스 로직은 단위 테스트 필수
- 컴포넌트는 렌더링 테스트 필수
- 통합 테스트는 주요 플로우만

## 커밋 컨벤션
- feat: 새 기능
- fix: 버그 수정
- refactor: 리팩토링
- docs: 문서
- test: 테스트
```

**2단계: 팀 공유 템플릿 작성**

`docs/examples/prompt-templates.md` 참조:
- 기능 구현 템플릿
- 버그 수정 템플릿
- 리팩토링 템플릿
- 코드 리뷰 템플릿

**3단계: Hooks 설정**

`.claude/settings.json`:
```json
{
  "hooks": {
    "before:commit": {
      "run": "npm test",
      "description": "커밋 전 테스트 실행"
    },
    "after:read": {
      "run": "echo 'File read: {file}'",
      "description": "파일 읽기 로깅"
    }
  }
}
```

**4단계: 온보딩 문서**

`ONBOARDING.md`:
```markdown
# 신규 팀원 가이드

## Claude 설정 (10분)
1. .claude/CLAUDE.md 읽기
2. 로컬에서 hooks 활성화
3. 첫 프롬프트 테스트

## 첫 작업 해보기 (30분)
1. docs/examples/prompt-templates.md 참조
2. 간단한 버그 수정 작업 선택
3. 템플릿 따라 프롬프트 작성
4. PR 생성

## 체크리스트
- [ ] CLAUDE.md 읽음
- [ ] 템플릿 사용해봄
- [ ] 첫 PR 생성함
- [ ] 코드 리뷰 받음
```

**효과**:
- 온보딩 시간: 2일 → 2시간
- 일관성: 팀원별 편차 감소
- 품질: 표준화된 패턴 적용

#### 5.2.2 PR 리뷰 워크플로우

**목표**:
- Claude를 활용한 효율적인 코드 리뷰
- 일관된 리뷰 품질 유지
- 리뷰어 부담 감소

**프로세스**:

**1단계: PR 작성자 - 셀프 리뷰**

PR 생성 전에 Claude로 셀프 리뷰:

```
내가 작성한 PR을 리뷰해줘.

변경 파일:
- src/features/auth/login.ts
- src/features/auth/login.test.ts

체크 포인트:
- [ ] 테스트 커버리지 충분한가?
- [ ] 에러 핸들링 적절한가?
- [ ] 타입 안전성 확보되었나?
- [ ] 성능 이슈 없나?
- [ ] 보안 취약점 없나?

CLAUDE.md의 코딩 컨벤션 준수 여부도 확인.
```

**2단계: 리뷰어 - Claude 보조 리뷰**

리뷰어는 Claude를 활용하여 빠르게 리뷰:

```
PR #123 리뷰 도와줘.

변경 내용: 로그인 기능 에러 핸들링 개선
변경 파일: src/features/auth/login.ts

우선 체크:
1. 보안 이슈 (인증, 에러 메시지 노출)
2. 에지 케이스 처리
3. 테스트 누락

심각한 이슈만 보고. 사소한 스타일은 건너뛰어.
```

**3단계: 리뷰 결과 정리**

Claude가 발견한 이슈를 PR 코멘트로:

```
Claude 리뷰 결과:

심각:
- [ ] 에러 메시지에 민감 정보 노출 가능성 (line 45)

중요:
- [ ] null 체크 누락 (line 78)
- [ ] 테스트 케이스 부족 (실패 시나리오)

제안:
- 에러 메시지 일반화
- null 체크 추가
- 테스트 커버리지 확대
```

**4단계: 피드백 반영**

PR 작성자가 피드백 반영:

```
PR #123 피드백 반영해줘.

리뷰 코멘트:
1. 에러 메시지 민감 정보 제거
2. null 체크 추가
3. 테스트 케이스 추가

변경 파일: src/features/auth/login.ts
기존 코드 읽지 말고, 위 3가지만 수정.
```

**효과**:
- 리뷰 시간: 30분 → 10분
- 발견되는 이슈: 2-3개 → 5-7개
- 품질: 일관성 향상

---

#### 5.2.3 공유 템플릿 관리

**목표**:
- 팀 전체가 검증된 프롬프트 패턴 사용
- 반복 작업 자동화
- 베스트 프랙티스 전파

**템플릿 카테고리**:

1. **기능 구현 템플릿**
   - 위치: `docs/examples/prompt-templates.md` → "새 기능 구현"
   - 용도: 새로운 기능 개발시
   - 포함: 명세 작성, 구현, 테스트, 문서화

2. **버그 수정 템플릿**
   - 위치: `docs/examples/prompt-templates.md` → "버그 수정"
   - 용도: 버그 리포트 받았을 때
   - 포함: 재현, 원인 분석, 수정, 검증

3. **리팩토링 템플릿**
   - 위치: `docs/examples/prompt-templates.md` → "리팩토링"
   - 용도: 코드 개선 작업
   - 포함: 현황 분석, 개선 계획, 단계별 실행

4. **코드 리뷰 템플릿**
   - 위치: `docs/checklists/code-review.md`
   - 용도: PR 리뷰시
   - 포함: 보안, 성능, 테스트, 스타일 체크

**템플릿 사용 예시**:

**상황**: 새로운 API 엔드포인트 추가

**Before (템플릿 없음)**:
```
/api/users 엔드포인트 만들어줘
```
결과: 불완전한 구현, 테스트 누락, 문서 없음

**After (템플릿 사용)**:
```
docs/examples/prompt-templates.md의 "새 기능 구현" 템플릿 사용.

기능: 사용자 목록 조회 API
엔드포인트: GET /api/users
명세:
- 페이지네이션 지원 (page, limit)
- 필터링: role, status
- 정렬: createdAt, name
- 응답: { users: [], total: number, page: number }

제약:
- 인증 필수 (admin role만)
- Rate limit: 100 req/min
- 최대 limit: 100

테스트 요구사항:
- 단위 테스트 (서비스 로직)
- 통합 테스트 (API 엔드포인트)
- 권한 테스트 (비인증, 비admin)
```

결과: 완전한 구현, 테스트 포함, 문서화됨

**효과**:
- 구현 누락: 40% → 5%
- 테스트 커버리지: 60% → 90%
- 문서화율: 30% → 95%

#### 5.2.4 팀 모니터링 및 개선

**목표**:
- 팀 전체의 토큰 사용 패턴 파악
- 비효율적인 패턴 식별
- 지속적인 개선

**추적 메트릭**:

1. **팀 레벨 메트릭**
   ```
   주간 리포트:
   - 총 토큰 사용량
   - 팀원별 사용 패턴
   - 캐시 히트율
   - 평균 대화 길이
   - 재작업 비율
   ```

2. **개인 레벨 메트릭**
   ```
   개인 대시보드:
   - 일일 토큰 사용량
   - 효율성 점수
   - 자주 사용하는 패턴
   - 개선 제안
   ```

**모니터링 프로세스**:

**매주 금요일: 주간 리뷰**
```markdown
## Week 15 Claude 사용 리포트

### 팀 메트릭
- 총 토큰: 450,000 (목표: 500,000 이하) ✓
- 캐시 히트율: 72% (목표: 70% 이상) ✓
- 재작업 비율: 8% (목표: 10% 이하) ✓

### 개선 필요 영역
1. 대화 길이가 너무 긴 케이스 3건 발견
   - 원인: 작업 범위 미리 분할 안 함
   - 해결: 대형 작업 체크리스트 공유

2. 캐시 미스가 많은 팀원 2명
   - 원인: CLAUDE.md 자주 수정
   - 해결: 안정화 후 수정 최소화

### 베스트 프랙티스 공유
- 김개발: 템플릿 활용으로 재작업 0%
- 이코더: 단계별 진행으로 토큰 40% 절감
```

**개선 사이클**:

1. **측정** → 매주 메트릭 수집
2. **분석** → 비효율적 패턴 식별
3. **공유** → 팀 미팅에서 베스트 프랙티스 전파
4. **적용** → CLAUDE.md, 템플릿 업데이트
5. **검증** → 다음 주 개선 확인

**효과** (3개월 후):
- 팀 토큰 사용량: 30% 감소
- 코드 품질: 일관성 향상
- 개발 속도: 20% 증가
- 온보딩 시간: 50% 단축
