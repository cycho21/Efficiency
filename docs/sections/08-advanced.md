## 8. 고급 테크닉

기본적인 최적화를 넘어서, Claude를 더욱 효과적으로 활용하는 고급 기법들을 소개합니다.

---

### 8.1 Subagent 활용

Subagent는 Claude Code CLI의 강력한 기능으로, 메인 대화의 컨텍스트를 오염시키지 않으면서 독립적인 작업을 수행할 수 있습니다.

---

#### Subagent란?

**정의**: 
메인 세션과 분리된 독립적인 Claude 인스턴스. 별도의 컨텍스트를 가지며, 결과만 메인 세션으로 반환합니다.

**언제 사용하는가**:
```
✅ 사용해야 할 때:
- 큰 파일 분석 (20줄 이상 출력)
- 여러 파일 검색/탐색
- 독립적인 리팩토링 제안
- 대량 데이터 처리

❌ 사용하지 말아야 할 때:
- 간단한 질문 (한 문장 답변)
- 메인 컨텍스트 필요한 작업
- 실시간 협업 필요한 작업
```

**장점**:
```
1. 컨텍스트 절약:
   - 큰 출력이 메인 컨텍스트를 오염시키지 않음
   - 메인 세션은 깔끔하게 유지

2. 병렬 처리:
   - 여러 subagent 동시 실행 가능
   - 전체 작업 시간 단축

3. 독립성:
   - 실패해도 메인 세션 영향 없음
   - 재시도 쉬움
```

**단점**:
```
1. 컨텍스트 공유 안 됨:
   - 메인 세션의 정보를 subagent가 모름
   - 명시적으로 전달 필요

2. 추가 토큰:
   - Subagent도 토큰 소비
   - 불필요한 사용은 비효율적
```

---

#### 사용 예시

**예시 1: 큰 파일 분석**

메인 세션에서:
```
사용자: "utils/ 디렉토리의 모든 파일 분석해서 중복 함수 찾아줘"

Claude: "20개 파일이 있네요. 이건 subagent로 처리하겠습니다."
→ Subagent 실행
→ 결과 요약만 메인 세션으로 반환

Subagent 결과:
"3개의 중복 함수 발견:
1. formatDate() - utils/date.ts, utils/helpers.ts
2. isEmpty() - utils/validation.ts, utils/string.ts
3. deepClone() - utils/object.ts, utils/helpers.ts"

메인 세션 컨텍스트:
- 요약만 포함 (3줄)
- 전체 분석 내용 (500줄)은 제외
- 토큰 절약: 95%
```

**예시 2: 병렬 탐색**

```
사용자: "프로젝트에서 API 엔드포인트를 찾고, 각각의 에러 핸들링 분석해줘"

Claude:
→ Subagent 1: API 엔드포인트 찾기
→ Subagent 2: 에러 핸들링 패턴 분석
→ Subagent 3: 테스트 커버리지 확인
(병렬 실행)

결과 통합:
"15개 엔드포인트 발견
- 12개: try-catch 있음
- 3개: 에러 핸들링 없음 (수정 필요)
- 테스트 커버리지: 60%"

시간 절감: 순차 처리 대비 70%
```

---

#### 최적 사용 패턴

**패턴 1: 탐색 후 집중**

```
1단계 (Subagent): 넓게 탐색
"전체 프로젝트에서 deprecated API 사용처 찾기"
→ 리스트 반환

2단계 (메인): 집중 처리
"위에서 찾은 3개 파일을 하나씩 수정"
→ 필요한 파일만 메인 컨텍스트에 로드
```

**패턴 2: 대량 처리**

```
Subagent로 100개 파일 분석:
"모든 컴포넌트에서 PropTypes 검증 누락 찾기"

메인 세션은 요약만 받음:
"23개 컴포넌트에서 PropTypes 누락"

후속 작업:
"가장 중요한 5개만 수정하자. 어떤 게 우선순위인가?"
```

**패턴 3: 독립적 품질 검사**

```
메인: 기능 구현 진행 중

Subagent (백그라운드):
- 린트 체크
- 테스트 커버리지 분석
- 보안 취약점 스캔

메인 컨텍스트 영향: 0
결과: 요약만 확인
```

---

#### 실제 효과 측정

**사례: 레거시 코드 분석**

**Without Subagent**:
```
메인 세션에서 50개 파일 분석:
- 전체 출력: 5,000줄
- 컨텍스트 사용: 80%
- 이후 작업 불가 (컨텍스트 포화)
- 새 세션 시작 필요
```

**With Subagent**:
```
Subagent로 50개 파일 분석:
- Subagent 출력: 5,000줄 (메인에 영향 없음)
- 메인 세션 수신: 요약 10줄
- 컨텍스트 사용: 15%
- 이후 작업 계속 가능
- 토큰 절약: 85%
```

### 8.2 컨텍스트 관리 전략

200K 토큰 윈도우를 효율적으로 활용하는 고급 전략입니다.

---

#### 컨텍스트 수명 주기

**단계별 관리**:

```
0-20% (초기): 자유롭게 탐색
- 필요한 파일 로드
- 프로젝트 이해
- 방향 설정

20-40% (안정): 집중 작업
- 핵심 작업 진행
- 불필요한 파일 읽기 자제
- 효율적 프롬프트

40-60% (주의): 선택적 작업
- 중요한 작업만
- 새 파일 읽기 최소화
- 압축 고려 시작

60-80% (경고): 압축 또는 종료
- progress.md 저장
- 핵심 정보만 유지하고 압축
- 또는 새 세션 시작

80%+ (위험): 즉시 조치
- 응답 품질 저하 시작
- 반드시 새 세션
- 상태 저장 필수
```

---

#### 컨텍스트 압축 기법

**방법 1: 점진적 요약**

작업하면서 주기적으로 요약:
```
사용자: "지금까지 한 작업을 3-5줄로 요약해줘"

Claude:
"1. 로그인 API 버그 수정 (auth.ts)
2. 테스트 3개 추가 (auth.test.ts)
3. 문서 업데이트 (API.md)"

→ 이 요약을 기록
→ 이후 필요시 요약 참조
→ 원본 대화는 필요 없음
```

**방법 2: progress.md 패턴**

컨텍스트 60% 도달시:
```
사용자: "progress.md 파일 생성해서 현재 상태 저장해줘"

progress.md 내용:
"""
# OAuth 구현 진행 상황

## 완료
- [x] Google OAuth 설정
- [x] 인증 플로우 구현
- [x] 토큰 저장 로직

## 진행 중
- [ ] 리프레시 토큰 처리
  - 현재 위치: src/auth/oauth.ts의 refreshToken 함수
  - 다음 단계: 만료 시간 체크 로직 추가

## 다음 작업
- [ ] 에러 핸들링
- [ ] 테스트 작성
- [ ] 문서화

## 중요 파일
- src/auth/oauth.ts (핵심 로직)
- src/types/auth.ts (타입 정의)
- .env.example (설정 예시)

## 알려진 이슈
- 리프레시 토큰 만료시 처리 미비
"""

새 세션에서:
"progress.md를 읽고 리프레시 토큰 처리 계속해줘"
→ 이전 컨텍스트 없이도 작업 계속
```

**방법 3: 선택적 정보 유지**

```
컨텍스트 60% 상황:

필수 정보 (유지):
- 현재 작업 중인 파일 (2-3개)
- 핵심 요구사항
- 진행 상태

제거 가능 (압축):
- 탐색했던 다른 파일들
- 시도했다가 폐기한 방법
- 중간 과정의 긴 출력
```

---

#### 컨텍스트 분할 전략

**전략 1: 작업 단위 분할**

```
하나의 큰 작업을 여러 세션으로:

세션 1: 설계 및 계획
- 요구사항 분석
- 아키텍처 설계
- 작업 분해
→ design.md 저장

세션 2: 핵심 구현
- design.md 기반 구현
- 단위 테스트
→ implementation-notes.md 저장

세션 3: 통합 및 테스트
- 통합 테스트
- 버그 수정
- 문서화
```

**전략 2: 모듈별 분할**

```
큰 프로젝트:

세션 A: 인증 모듈
- auth/ 디렉토리만 집중
- 다른 모듈 참조 최소화

세션 B: API 모듈
- api/ 디렉토리만 집중
- 인터페이스만 참조

세션 C: 통합
- 각 모듈 연결
- E2E 테스트
```

---

#### 효율적 파일 로딩

**전략: 계층적 로딩**

```
1단계: 타입/인터페이스 (항상 필요)
"다음 파일 읽어줘:
- src/types/*.ts (타입 정의)"

2단계: 핵심 파일 (자주 참조)
"핵심 파일 추가로 읽어줘:
- src/core/auth.ts
- src/core/api.ts"

3단계: 특정 기능 (필요시만)
"OAuth 구현 관련 파일만:
- src/auth/oauth.ts
- src/auth/providers/google.ts"

장점:
- 공통 정보는 한 번만 로드
- 캐시 효과 극대화
- 불필요한 파일 배제
```

---

#### 컨텍스트 관리 실전 예시

**예시: 대규모 리팩토링**

**초기 (0-20%)**:
```
"프로젝트 구조 파악:
- package.json 읽기
- 주요 디렉토리 구조 확인
- README 읽기"

컨텍스트: 5,000 토큰 (2.5%)
```

**탐색 (20-40%)**:
```
"리팩토링 대상 찾기:
- 레거시 패턴 검색 (subagent)
- 중복 코드 분석 (subagent)
- 결과 요약만 수신"

컨텍스트: 15,000 토큰 (7.5%)
메인 컨텍스트 증가: 10,000 토큰 (5%)
```

**실행 (40-60%)**:
```
"가장 중요한 5개 파일 리팩토링:
- 한 번에 1개 파일만
- 수정 후 다음 파일"

컨텍스트: 40,000 토큰 (20%)
```

**마무리 (60% 전에 종료)**:
```
"progress.md 저장:
- 완료: 5개 파일
- 남은 작업: 12개 파일 (우선순위 포함)"

새 세션으로 이어서:
"progress.md 읽고 리팩토링 계속"
```

**효과**:
```
Without 관리:
- 1개 세션에서 모두 처리 시도
- 컨텍스트 90% 도달
- 응답 느려짐, 품질 저하
- 작업 시간: 3시간

With 관리:
- 2개 세션으로 분할
- 각 세션 컨텍스트 50% 이하
- 빠른 응답, 고품질 유지
- 작업 시간: 2시간
```

### 8.3 메모리 시스템 활용

Claude Code의 메모리 기능을 활용하여 장기적인 컨텍스트를 유지합니다.

---

#### CLAUDE.md를 메모리로 활용

**개념**: 
CLAUDE.md는 모든 세션에서 자동으로 로드되는 "영구 메모리"입니다.

**최적화 원칙**:

1. **안정적인 정보만 포함**:
```markdown
✅ 좋은 내용:
- 프로젝트 아키텍처 (거의 안 바뀜)
- 코딩 컨벤션 (팀 표준)
- 금지 사항 (일관성 중요)
- 디렉토리 구조 (자주 안 바뀜)

❌ 피해야 할 내용:
- 현재 작업 중인 태스크 (매일 바뀜)
- 진행 상황 (계속 업데이트 필요)
- 임시 메모 (세션 종료시 불필요)
- 상세한 구현 (너무 구체적)
```

2. **간결하게 유지**:
```
Bad CLAUDE.md (5,000 토큰):
- 모든 함수 설명
- 상세한 API 문서
- 전체 히스토리
→ 매 세션 5,000 토큰 소비
→ 캐시되어도 비효율적

Good CLAUDE.md (500 토큰):
- 핵심 원칙만
- 참조할 문서 링크
- 필수 규칙만
→ 매 세션 500 토큰
→ 10배 효율적
```

3. **계층적 구조**:
```
.claude/
  CLAUDE.md (핵심, 거의 수정 안 함)
  architecture.md (아키텍처 상세, 가끔 수정)
  conventions.md (컨벤션 상세, 가끔 수정)
  current.md (현재 작업, 자주 수정)

CLAUDE.md는 다른 파일 참조:
"상세 아키텍처는 .claude/architecture.md 참고"
```

---

#### 세션 간 정보 전달

**방법 1: progress.md 패턴 (재방문)**

컨텍스트 관리 섹션에서 다뤘지만, 메모리 관점에서 재조명:

```
세션 1 종료시:
"progress.md 작성해줘:
- 완료한 작업
- 현재 상태
- 다음 단계
- 중요 파일 목록
- 결정 사항 및 이유"

세션 2 시작시:
"progress.md 읽고 작업 계속해줘"
→ 이전 세션의 "메모리" 복원
```

**방법 2: decision-log.md**

중요한 결정 사항 기록:
```markdown
# 기술 결정 기록

## 2026-04-10: JWT 대신 세션 기반 인증 선택

**결정**: 세션 기반 인증 사용

**이유**:
- 서버 측 제어 필요 (즉시 로그아웃)
- JWT 리프레시 복잡도 회피
- 트래픽 낮아서 세션 부담 적음

**영향**:
- Redis 세션 스토어 필요
- 서버 상태 있음 (stateful)

## 2026-04-12: PostgreSQL 선택

**결정**: PostgreSQL over MongoDB

**이유**:
- 관계형 데이터 많음 (사용자-권한-조직)
- ACID 보장 필요
- 팀이 SQL 경험 많음
```

새 세션에서:
```
"decision-log.md 확인 후 권한 시스템 설계해줘"
→ 이전 결정 컨텍스트 파악
→ 일관된 설계
```

---

#### 팀 메모리 공유

**패턴: 팀 공통 CLAUDE.md**

```markdown
# [프로젝트명] - 팀 공통 가이드

## 아키텍처
- 마이크로서비스 (API Gateway + 4개 서비스)
- 이벤트 주도 (Kafka)
- 자세한 내용: docs/architecture.md

## 코딩 컨벤션
- TypeScript strict mode
- ESLint + Prettier
- 함수형 우선, 클래스 최소화

## 팀 결정 사항
최신 기술 결정은 .claude/decision-log.md 참고

## 금지 사항
- 직접 DB 접근 (반드시 ORM 사용)
- any 타입 사용
- console.log (logger 사용)
- 테스트 없는 PR
```

**효과**:
```
팀원 A, B, C 모두:
- 같은 컨벤션으로 코드 생성
- 일관된 아키텍처 결정
- 중복 질문 감소

토큰 절감:
- 매번 "우리 프로젝트는..."로 시작 안 해도 됨
- Claude가 프로젝트 컨텍스트 이미 파악
```

---

### 8.4 캐시 친화적 프로젝트 구조

프로젝트 구조 자체를 캐싱에 유리하게 설계합니다.

---

#### 원칙 1: 안정적 파일과 변동적 파일 분리

**나쁜 구조**:
```
src/
  config.ts (설정 + 비즈니스 로직 혼재)
  utils.ts (안정적 유틸 + 자주 바뀌는 임시 코드)
  
→ config.ts 수정 → 전체 파일 캐시 무효화
→ 캐시 히트율 낮음
```

**좋은 구조**:
```
src/
  config/
    constants.ts (거의 안 바뀜)
    environment.ts (환경별 설정, 가끔 바뀜)
    feature-flags.ts (자주 바뀜)
  utils/
    stable/ (안정적 유틸)
      date.ts
      string.ts
    experimental/ (자주 바뀌는 코드)
      new-feature.ts

→ constants.ts 읽음 → 캐시 오래 유지
→ feature-flags.ts 수정 → 다른 파일 캐시 유지
→ 캐시 히트율 높음
```

---

#### 원칙 2: 작은 파일로 분할

**나쁜 구조**:
```
src/
  auth.ts (2,000줄 - 인증, 권한, 세션 모두)
  
→ 세션 로직 1줄 수정
→ 전체 2,000줄 캐시 무효화
→ 재로드 비용: 높음
```

**좋은 구조**:
```
src/auth/
  authentication.ts (300줄 - 인증만)
  authorization.ts (200줄 - 권한만)
  session.ts (150줄 - 세션만)
  types.ts (100줄 - 타입, 거의 안 바뀜)

→ session.ts 1줄 수정
→ session.ts만 캐시 무효화
→ authentication.ts, authorization.ts 캐시 유지
→ 재로드 비용: 낮음
```

**원칙**:
```
파일당 최대 300-500줄 권장
→ 수정 영향 범위 최소화
→ 캐시 효율 극대화
```

---

#### 원칙 3: 의존성 방향 관리

**나쁜 구조** (순환 참조):
```
auth.ts imports user.ts
user.ts imports auth.ts

→ 하나 읽으면 둘 다 읽어야 함
→ 캐시 효과 반감
```

**좋은 구조** (단방향):
```
types.ts (기본 타입)
  ↑
utils.ts (types 사용)
  ↑
services.ts (utils 사용)
  ↑
controllers.ts (services 사용)

→ types.ts 한 번 읽으면 캐시
→ 다른 파일들이 재사용
→ 캐시 효과 배가
```

---

#### 원칙 4: 공통 타입 분리

**나쁜 구조**:
```
user.service.ts:
  export interface User { ... }
  export class UserService { ... }

auth.service.ts:
  import { User } from './user.service'
  
→ User 타입 보려고 전체 UserService 읽어야 함
→ 불필요한 토큰 소비
```

**좋은 구조**:
```
types/
  user.types.ts:
    export interface User { ... }
    
services/
  user.service.ts:
    import { User } from '../types/user.types'
    export class UserService { ... }
    
  auth.service.ts:
    import { User } from '../types/user.types'
    
→ user.types.ts만 읽으면 됨 (작고 안정적)
→ 캐시 효율 높음
```

---

#### 원칙 5: 문서 파일 구조화

**나쁜 구조**:
```
README.md (10,000줄 - 모든 문서)

→ 일부만 필요해도 전체 읽어야 함
→ 10,000줄 × 토큰 계수 = 비효율
```

**좋은 구조**:
```
docs/
  README.md (목차 + 개요, 200줄)
  getting-started.md (300줄)
  architecture.md (400줄)
  api/
    auth.md (150줄)
    users.md (150줄)
  guides/
    deployment.md (200줄)
    
→ 필요한 문서만 선택적 읽기
→ 각 문서 작고 캐시 가능
```

---

#### 실전 적용: 리팩토링 예시

**Before**:
```
프로젝트 구조:
src/
  app.ts (1,500줄 - 모든 로직)
  config.ts (800줄 - 모든 설정)
  utils.ts (1,200줄 - 모든 유틸)
  
캐시 효과:
- app.ts 수정시 1,500줄 전체 무효화
- 평균 히트율: 35%
- 세션당 파일 읽기: 8,000 토큰
```

**After**:
```
프로젝트 구조:
src/
  types/
    *.types.ts (각 100줄, 안정적)
  config/
    constants.ts (200줄, 안정적)
    environment.ts (150줄, 가끔 수정)
  services/
    auth.service.ts (300줄)
    user.service.ts (250줄)
  utils/
    date.ts (80줄, 안정적)
    string.ts (100줄, 안정적)
    
캐시 효과:
- 수정 파일만 캐시 무효화 (평균 200줄)
- 평균 히트율: 72%
- 세션당 파일 읽기: 2,800 토큰
- 토큰 절감: 65%
```

#### 체크리스트: 캐시 친화적 구조

```markdown
- [ ] 파일 크기 확인
  - 500줄 초과 파일이 있는가?
  - 있다면 분할 가능한가?
  
- [ ] 안정성 분리 확인
  - 자주 바뀌는 코드와 안정적 코드가 같은 파일에 있는가?
  - 분리 가능한가?
  
- [ ] 타입 분리 확인
  - 타입이 구현 파일에 포함되어 있는가?
  - types/ 디렉토리로 분리 가능한가?
  
- [ ] 순환 참조 확인
  - 순환 참조가 있는가?
  - 의존성 방향을 단방향으로 만들 수 있는가?
  
- [ ] 문서 구조 확인
  - 하나의 큰 README인가?
  - 주제별로 분할 가능한가?
```
