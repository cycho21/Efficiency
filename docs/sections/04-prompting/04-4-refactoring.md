### 4.4 리팩토링/리뷰용 프롬프팅

대규모 코드베이스의 리팩토링이나 리뷰는 전략적 접근이 필요합니다.

#### 전체 스캔 vs 특정 부분

**전체 스캔 (탐색 모드)**:

목적: 문제 영역 파악, 우선순위 결정

**✅ Good**:
```
프로젝트 전체에서 보안 취약점을 스캔해줘.

범위: src/ 디렉토리 전체

집중 항목:
1. SQL Injection 가능성
2. XSS 취약점
3. 하드코딩된 시크릿
4. 인증/인가 누락

출력 형식:
각 발견 사항을:
- [심각도] 파일:라인 - 문제
- 간단한 설명 (1-2줄)
- 영향도 (High/Medium/Low)

상세 분석이나 수정은 하지 말고, 목록만 먼저.
우선순위 High인 항목 발견시 즉시 알려줘.

예상: 20-30개 항목 발견 (목록만이라 토큰 효율적)
```

**개선 효과**:
- 목록만 → 빠른 스캔
- 우선순위 파악 → 중요한 것부터
- 토큰 절약

**토큰 영향**: 전체 스캔 + 목록 생성 → 5,000-8,000 토큰 (상세 분석 대비 70% 절감)

---

**특정 부분 (상세 분석)**:

스캔 후 중요한 부분을 상세 분석합니다.

**✅ Good**:
```
위에서 발견한 High 우선순위 항목 3개를 상세 분석하고 수정해줘.

항목:
1. src/api/users.controller.ts:45 - SQL Injection
2. src/middleware/auth.ts:23 - 인증 우회 가능
3. src/config/database.ts:12 - 하드코딩된 비밀번호

각 항목별:
1. 현재 코드 (문제 부분)
2. 왜 위험한지 (공격 시나리오)
3. 수정 코드
4. 테스트 방법

하나씩 순차적으로 진행. 각 수정 완료 후 다음.
```

**개선 효과**:
- 우선순위 집중 → 효과적
- 상세 분석 → 제대로 된 수정
- 순차 진행 → 검증 가능

**토큰 영향**: 항목당 1,500-2,500 토큰 × 3 = 4,500-7,500 토큰

---

#### 우선순위 지정

리팩토링/리뷰는 우선순위가 중요합니다.

**❌ Bad**:
```
이 프로젝트를 개선해줘
```

**✅ Good**:
```
이 프로젝트의 개선점을 찾고 우선순위별로 분류해줘.

분석 범위: src/ 전체

분류 기준:
- **Critical**: 보안, 데이터 손실, 심각한 버그
- **Important**: 성능 이슈, 확장성 문제, 기술 부채
- **Nice-to-have**: 가독성, 코드 스타일, 최적화

각 카테고리별:
- 문제 설명
- 영향도 (사용자/시스템/개발자)
- 예상 수정 시간
- 수정 난이도

출력 형식:
\`\`\`markdown
## Critical (즉시 수정)
1. [보안] SQL Injection in users.controller.ts
   - 영향: 전체 DB 노출 가능
   - 수정 시간: 2시간
   - 난이도: 중

## Important (이번 주)
1. [성능] N+1 쿼리 in products.service.ts
   - 영향: API 응답 시간 5초
   - 수정 시간: 4시간
   - 난이도: 중

## Nice-to-have (백로그)
1. [가독성] 중복 코드 in utils/
   - 영향: 유지보수성
   - 수정 시간: 3시간
   - 난이도: 하
\`\`\`

분류 후 Critical 항목부터 수정 진행 여부 결정.
```

**개선 효과**:
- 명확한 우선순위 → 효율적 작업
- 영향도 파악 → 의사결정 용이
- 예상 시간 → 계획 수립 가능

**토큰 영향**: 분석 + 분류 → 6,000-10,000 토큰 (가치 있는 투자)

---

#### 변경 영향도 고려

리팩토링시 영향도 분석은 필수입니다.

**✅ Good**:
```
src/services/payment.service.ts를 리팩토링하려고 해.
먼저 변경 영향도를 분석해줘.

분석 항목:
1. 이 파일을 import하는 모든 파일 목록
2. Public API (외부에서 사용하는 메서드)
3. 테스트 파일 존재 여부
4. 의존성 (이 파일이 사용하는 다른 서비스)

출력:
\`\`\`markdown
## 영향도 분석

### 직접 의존 파일 (5개)
- src/api/checkout.controller.ts
- src/api/subscriptions.controller.ts
- src/jobs/payment-retry.job.ts
- src/webhooks/stripe.webhook.ts
- src/services/order.service.ts

### Public API
1. processPayment(amount, method)
2. refundPayment(paymentId, amount?)
3. getPaymentStatus(paymentId)

⚠️ 주의: Public API 변경시 위 5개 파일 모두 수정 필요

### 테스트
✅ src/services/__tests__/payment.service.test.ts 존재
- 커버리지: 85%
- 테스트 수: 23개

### 의존성
- StripeService (외부 API)
- DatabaseService
- LoggerService

## 권장 사항
- Public API는 유지하고 내부 구현만 변경
- 테스트 먼저 100% 통과 확인
- 리팩토링 후 의존 파일들의 테스트도 실행
\`\`\`

이 분석 기반으로 리팩토링 계획 수립.
```

**개선 효과**:
- 영향도 사전 파악 → 안전한 리팩토링
- 테스트 전략 수립 → 품질 보장
- 리스크 최소화

**토큰 영향**: 영향도 분석 → 2,000-3,500 토큰 (리팩토링 실패 방지)

---

#### 단계별 리팩토링 계획

대규모 리팩토링은 계획이 중요합니다.

**✅ Good**:
```
레거시 인증 시스템 (src/auth/)을 현대화하는 계획을 수립해줘.

현재 상태:
- 5년된 코드
- 세션 기반 (JWT로 전환 원함)
- 테스트 없음
- 3,000줄 코드

목표:
- JWT 기반 인증
- 테스트 커버리지 80% 이상
- 기존 사용자 호환성 유지
- 다운타임 없음

계획 형식:
각 Phase별로:
1. 목표 (무엇을)
2. 작업 항목 (상세)
3. 테스트 전략
4. 롤백 계획
5. 예상 시간
6. 리스크 및 완화

Phase는 4-6개로 나누고, 각 Phase는 1-2주 내 완료 가능하게.

계획 승인 후 Phase별로 실행.
```

**출력 예시**:
```markdown
## Phase 1: 기반 준비 (1주)
목표: 기존 시스템 테스트 작성, 새 구조 설계

작업:
1. 레거시 코드 테스트 작성 (현재 동작 보존)
2. JWT 인증 구조 설계 문서
3. 새 auth 모듈 스켈레톤 생성

테스트:
- 레거시 시스템 테스트 80% 커버리지

롤백:
- 새 코드는 아직 사용 안 함 (롤백 불필요)

리스크:
- 테스트 작성이 예상보다 오래 걸릴 수 있음
- 완화: 핵심 플로우 우선, 엣지 케이스는 Phase 2

## Phase 2: 병렬 구현 (2주)
목표: JWT 시스템 구현, 레거시와 병렬 운영

작업:
1. JWT 발급/검증 구현
2. 새 로그인 엔드포인트 (/auth/v2/login)
3. Feature flag로 제어 (기본 off)

테스트:
- 새 시스템 테스트 100%
- 통합 테스트 (레거시 영향 없음 확인)

롤백:
- Feature flag off → 레거시로 즉시 복귀

리스크:
- 두 시스템 병렬 운영 복잡도
- 완화: 명확한 분리, 공통 코드 최소화

[Phase 3, 4, 5 계속...]
```

**개선 효과**:
- 명확한 로드맵 → 팀 정렬
- 리스크 관리 → 안전한 마이그레이션
- 점진적 진행 → 지속적 배포 가능

**토큰 영향**: 계획 수립 → 4,000-6,000 토큰 (실행 실패 방지로 큰 가치)

---

#### 리팩토링/리뷰 체크리스트

```markdown
전체 스캔:
[ ] 명확한 스캔 목적 (보안/성능/품질)
[ ] 목록 우선 (상세 분석은 나중에)
[ ] 우선순위 기준 명시
[ ] 예상 발견 개수 언급

특정 부분 분석:
[ ] 스캔 결과 기반 선택
[ ] 우선순위 높은 것부터
[ ] 상세 분석 + 수정 제안
[ ] 순차 진행 (한 번에 하나씩)

우선순위:
[ ] Critical/Important/Nice-to-have 분류
[ ] 영향도 평가 (사용자/시스템/개발자)
[ ] 예상 수정 시간
[ ] 난이도 평가

영향도 분석:
[ ] 의존 파일 목록
[ ] Public API 식별
[ ] 테스트 존재 여부
[ ] 리스크 평가 및 완화 방안
```

---

### 섹션 4 요약

**프롬프팅 원칙 총정리**:

```markdown
공통 원칙 (모든 작업):
✓ 명확한 목표 명시
✓ 컨텍스트 최소화 (필요한 것만)
✓ 단계별 진행 (한 번에 하나씩)
✓ 결과 형식 지정

코딩 작업:
✓ 리팩토링: 목표/제약/범위 명확히
✓ 버그 수정: 재현 단계 + 예상/실제 동작
✓ 새 기능: 명세 우선 → 단계별 구현
✓ 코드 리뷰: 범위 지정 + 우선순위

문서 작성:
✓ 아웃라인 먼저 승인
✓ 섹션별 작성 (한 번에 전체 X)
✓ 톤 앤 매너 + 대상 독자 명시
✓ 템플릿/형식 제공

리팩토링/리뷰:
✓ 전체 스캔 → 목록 우선
✓ 우선순위 지정 (Critical/Important/Nice-to-have)
✓ 영향도 분석 → 안전한 변경
✓ 단계별 계획 → 점진적 개선
```

**토큰 절감 효과**:

```
Bad 프롬프트:
- 모호한 요청 → 불필요한 탐색
- 전체 컨텍스트 로드 → 과다 토큰
- 한 번에 많은 작업 → 재작업 발생
평균 토큰: 10,000-20,000 (품질 낮음)

Good 프롬프트:
- 명확한 목표 → 정확한 실행
- 필요한 것만 → 토큰 절약
- 단계별 진행 → 고품질 결과
평균 토큰: 3,000-7,000 (품질 높음)

절감율: 50-70%
```

**품질 향상 효과**:

```
Before (Bad Prompts):
- 재작업: 30-50%
- 불완전한 구현: 40%
- 방향 수정: 20%

After (Good Prompts):
- 재작업: 5-10%
- 불완전한 구현: 5%
- 방향 수정: 거의 없음

효과:
- 시간 절약: 60%
- 토큰 절약: 50-70%
- 품질 향상: 2-3배
```

**빠른 참조 가이드**:

| 작업 유형 | 핵심 포인트 | 예상 토큰 (Good) |
|----------|------------|----------------|
| 리팩토링 | 목표/제약/범위 | 2,500-3,500 |
| 버그 수정 | 재현 단계 | 1,500-2,500 |
| 새 기능 | 명세 우선 | 7,500-10,500 |
| 코드 리뷰 | 범위 한정 | 2,000-4,000 |
| README | 아웃라인 + 섹션별 | 5,800-8,700 |
| API 문서 | 템플릿 + 배치 | 500-800/endpoint |
| 전체 스캔 | 목록만 우선 | 5,000-8,000 |
| 리팩토링 계획 | 단계별 + 리스크 | 4,000-6,000 |
