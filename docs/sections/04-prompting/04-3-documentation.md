### 4.3 문서 작성용 프롬프팅

문서 작성은 아웃라인을 먼저 승인하고 섹션별로 작성하는 것이 효율적입니다.

#### 핵심 원칙

**아웃라인 우선 접근**:
1. 아웃라인 먼저 승인
2. 섹션별 작성 (한 번에 전체 X)
3. 톤 앤 매너 지정
4. 대상 독자 명시

#### README 작성

**❌ Bad**:
```
README 파일을 작성해줘
```

**문제점**:
- 대상 독자 불명확
- 포함할 내용 모호
- 톤 앤 매너 정의 없음

**토큰 영향**: 일반적인 README + 재작업 → 4,000-6,000 토큰

**✅ Good (2단계 접근)**:

**Step 1: 아웃라인 작성**
```
이 프로젝트의 README 아웃라인을 작성해줘.

프로젝트: E-commerce API (Node.js + TypeScript + PostgreSQL)

대상 독자:
1. 신규 개발자 (팀 온보딩)
2. API 사용자 (프론트엔드 개발자)

목적:
- 5분 안에 프로젝트 이해
- 10분 안에 로컬 환경 구성
- 기본 API 사용법 파악

톤 앤 매너:
- 친근하고 명확하게
- 전문 용어는 설명과 함께
- 예시 코드 포함

포함 항목:
- 프로젝트 개요
- 주요 기능
- 기술 스택
- 시작하기 (설치/실행)
- API 기본 사용법
- 환경 변수 설정

제외 항목:
- 상세 API 문서 (별도 문서)
- 내부 아키텍처 (ARCHITECTURE.md)
- 기여 가이드 (CONTRIBUTING.md)

아웃라인 작성 후 승인받고 섹션별로 작성 시작.
```

**Step 2: 아웃라인 승인 후 섹션별 작성**
```
✅ 아웃라인 승인. 이제 섹션 1, 2를 작성해줘.

Section 1: 프로젝트 개요
- 1-2 문단으로 간결하게
- 주요 기능 3-5개 bullet points
- 데모 링크 (있다면)

Section 2: 기술 스택
- 주요 기술만 (세부 dependencies는 package.json 참조)
- 각 기술 선택 이유 간단히 (1줄)

예시 형식:
\`\`\`markdown
## 기술 스택

- **Node.js 18+**: LTS 버전, 최신 기능 활용
- **TypeScript**: 타입 안전성 및 개발 경험 향상
- **PostgreSQL**: 관계형 데이터 및 트랜잭션 지원
...
\`\`\`

Section 1, 2 완료 후 Section 3, 4 진행.
```

**Step 3: 다음 섹션**
```
✅ Section 1, 2 확인. 이제 Section 3 (시작하기)를 작성해줘.

Section 3: 시작하기

포함:
1. 사전 요구사항 (Node.js 버전, PostgreSQL 등)
2. 설치 (클론, npm install, 환경변수 설정)
3. 데이터베이스 설정 (마이그레이션)
4. 실행 (개발 모드)
5. 테스트 실행

각 단계:
- 명령어는 코드 블록으로
- 예상 출력 포함 (사용자가 성공 확인 가능하게)
- 문제 발생시 트러블슈팅 팁

예시:
\`\`\`markdown
### 설치

\`\`\`bash
# 저장소 클론
git clone https://github.com/example/ecommerce-api.git
cd ecommerce-api

# 의존성 설치
npm install

# 환경변수 설정
cp .env.example .env
# .env 파일을 열어서 DATABASE_URL 등을 설정하세요
\`\`\`

**예상 출력:**
\`\`\`
added 234 packages, and audited 235 packages in 12s
\`\`\`
...
\`\`\`
```

**개선 효과**:
- 아웃라인 승인 → 방향 검증
- 섹션별 작성 → 품질 집중
- 명확한 형식 → 일관성 유지

**토큰 영향**: 
- 아웃라인: 800-1,200 토큰
- 섹션별 작성: 1,000-1,500 토큰 × 5섹션 = 5,000-7,500 토큰
- 총: 5,800-8,700 토큰 (고품질, 재작업 최소화)

---

#### API 문서

API 문서는 일관된 형식이 중요합니다.

**❌ Bad**:
```
API 문서를 만들어줘
```

**✅ Good**:
```
REST API 문서를 작성해줘.

대상: src/api/products.controller.ts의 모든 엔드포인트

형식: Markdown (docs/api/products.md)

각 엔드포인트별 포함 내용:
1. HTTP 메서드 및 경로
2. 설명 (1-2줄)
3. 인증 요구사항
4. 요청 파라미터 (Path/Query/Body)
   - 타입, 필수/선택, 설명, 예시
5. 응답 형식 (성공/실패 케이스별)
6. 예시 요청/응답 (curl 명령어)
7. 에러 코드

템플릿:
\`\`\`markdown
### GET /api/products/:id
특정 상품의 상세 정보를 조회합니다.

**Authentication:** Required (Bearer token)

**Path Parameters:**
| Name | Type | Description |
|------|------|-------------|
| id | string | 상품 ID (UUID) |

**Response (200 OK):**
\`\`\`json
{
  "id": "123e4567-e89b-12d3-a456-426614174000",
  "name": "상품명",
  "price": 29900,
  ...
}
\`\`\`

**Errors:**
- 401: 인증 실패
- 404: 상품을 찾을 수 없음

**Example:**
\`\`\`bash
curl -X GET https://api.example.com/api/products/123 \\
  -H "Authorization: Bearer YOUR_TOKEN"
\`\`\`
\`\`\`

한 번에 2-3개 엔드포인트씩 작성하고 확인 후 계속.
```

**개선 효과**:
- 일관된 템플릿 → 읽기 쉬운 문서
- 예시 포함 → 즉시 사용 가능
- 배치 작성 → 효율성

**토큰 영향**: 엔드포인트당 500-800 토큰 (템플릿 재사용으로 효율적)

---

#### 기술 블로그/가이드

긴 문서는 반드시 섹션별로 작성합니다.

**❌ Bad**:
```
TypeScript 마이그레이션 가이드를 작성해줘. 처음부터 끝까지 다.
```

**문제점**:
- 너무 긴 요청 → 불완전한 결과
- 품질 검증 어려움

**✅ Good**:
```
"JavaScript에서 TypeScript로 마이그레이션" 가이드를 작성해줘.

대상 독자: 중급 JavaScript 개발자 (TypeScript 초심자)
분량: 약 2000-3000 단어
톤: 친근하고 실용적, 예시 중심

Step 1: 먼저 아웃라인만 작성해줘.

포함 섹션:
- 왜 TypeScript인가
- 마이그레이션 전 준비
- 단계별 마이그레이션 (점진적 접근)
- 일반적인 패턴 변환
- 문제 해결
- 다음 단계

각 섹션의 주요 포인트 3-5개씩 나열.

아웃라인 승인 후 섹션별로 작성 진행.
```

**아웃라인 승인 후**:
```
✅ 아웃라인 좋음. Section 1, 2를 작성해줘.

Section 1: 왜 TypeScript인가 (500-700 단어)
- 타입 안전성 이점
- 개발 경험 향상 (IDE 지원)
- 유지보수성
- 실제 사례 1개 (Before/After 코드)

Section 2: 마이그레이션 전 준비 (600-800 단어)
- 테스트 커버리지 확보
- tsconfig.json 초기 설정
- 도구 설치
- 체크리스트 형식으로

Section 1, 2 완성 후 다음 섹션 계속.
```

**개선 효과**:
- 섹션별 품질 관리
- 중간 피드백 가능
- 일관성 유지

**토큰 영향**: 
- 전체 한 번에: 15,000-25,000 토큰 (품질 저하 위험)
- 섹션별: 2,000-3,000 × 6섹션 = 12,000-18,000 토큰 (고품질 보장)

---

#### 문서 작성 체크리스트

```markdown
시작 전:
[ ] 대상 독자 정의
[ ] 문서 목적 명확히
[ ] 톤 앤 매너 결정
[ ] 분량 목표 설정

진행 방식:
[ ] 아웃라인 먼저 작성 및 승인
[ ] 섹션별 작성 (한 번에 1-2개 섹션)
[ ] 형식/템플릿 제공
[ ] 예시 코드 포함 요청

품질 검증:
[ ] 각 섹션 완성 후 확인
[ ] 일관성 체크 (톤, 형식)
[ ] 예시 코드 실행 가능성 확인
[ ] 링크 유효성 (내부/외부)
```

**추가 리소스**: 상세한 문서 작성 템플릿은 `examples/prompt-templates.md` 참조
