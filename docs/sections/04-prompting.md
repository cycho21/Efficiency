## 4. 프롬프팅 베스트 프랙티스

좋은 설정만으로는 부족합니다. **어떻게 요청하는가**가 결과물의 품질과 토큰 효율성을 결정합니다.

이 섹션에서는 작업 유형별로 효과적인 프롬프트 작성법을 다룹니다.

### 섹션 개요

- **4.1 공통 원칙**: 모든 프롬프트에 적용되는 기본 원칙
- **4.2 코딩 작업용 프롬프팅**: 리팩토링, 버그 수정, 새 기능, 코드 리뷰
- **4.3 문서 작성용 프롬프팅**: README, 가이드, API 문서 등
- **4.4 리팩토링/리뷰용 프롬프팅**: 전체 스캔, 우선순위 지정, 영향도 분석

**💡 핵심 포인트**: 좋은 프롬프트는 토큰을 절약하면서도 더 나은 결과를 만듭니다.

### 4.1 공통 원칙

모든 작업 유형에 공통으로 적용되는 프롬프팅 원칙입니다.

#### 원칙 1: 명확한 목표 명시

**왜**: 모호한 목표는 Claude가 불필요한 탐색을 하게 만들어 토큰을 낭비합니다.

**어떻게**:
- 무엇을 원하는지 명확히
- 성공 기준 제시
- 범위 한정

**❌ Bad**:
```
사용자 인증 부분 좀 개선해줘
```

**문제점**:
- "개선"이 무엇을 의미하는지 불명확 (보안? 성능? UX?)
- 범위가 너무 넓음 (전체 인증 시스템?)
- 성공 기준 없음

**토큰 영향**: Claude가 전체 인증 시스템을 분석하고 여러 개선안을 제시 → 3,000-5,000 토큰

**✅ Good**:
```
src/auth/login.ts의 로그인 실패시 에러 메시지를 더 구체적으로 개선해줘.

목표:
- 현재: "로그인 실패"
- 변경 후: "이메일 또는 비밀번호가 일치하지 않습니다" 
         (보안상 어느 쪽이 틀렸는지는 밝히지 않음)

조건:
- 보안 베스트 프랙티스 준수 (타이밍 공격 방지)
- 다국어 지원 고려 (i18n 키 사용)
- 기존 테스트 통과
```

**개선 효과**:
- 정확한 파일과 범위 지정 → 불필요한 탐색 제거
- 명확한 Before/After → 정확한 구현
- 조건 명시 → 재작업 방지

**토큰 영향**: 필요한 파일만 읽고 정확히 구현 → 800-1,200 토큰 (70% 절감)

---

#### 원칙 2: 컨텍스트 최소화 (필요한 것만)

**왜**: 불필요한 파일이나 정보는 토큰을 낭비하고 Claude의 집중력을 분산시킵니다.

**어떻게**:
- 필요한 파일만 명시
- 관련 없는 정보 제외
- 이미 읽은 파일 재언급 방지

**❌ Bad**:
```
프로젝트 전체를 분석해서 API 응답 시간을 개선해줘
```

**문제점**:
- 전체 프로젝트 스캔 → 수만 토큰
- 문제 위치를 찾는 것부터 시작 → 비효율적
- 우선순위 없음

**토큰 영향**: 전체 프로젝트 스캔 + 분석 → 20,000-50,000 토큰

**✅ Good**:
```
/api/products 엔드포인트의 응답 시간을 개선해줘.

현재 상태:
- 평균 응답 시간: 2.5초
- 병목: DB 쿼리 (slow query log 확인)

관련 파일:
- src/api/products.controller.ts (현재 구현)
- src/services/product.service.ts (DB 쿼리 로직)

목표:
- 응답 시간 500ms 이하
- N+1 쿼리 제거
- 기존 API 스펙 유지
```

**개선 효과**:
- 문제 위치 명시 → 탐색 시간 제거
- 필요한 파일만 → 토큰 절약
- 구체적 목표 → 정확한 최적화

**토큰 영향**: 필요한 2개 파일만 분석 + 최적화 → 2,000-3,000 토큰 (85% 절감)

---

#### 원칙 3: 단계별 진행 (한 번에 너무 많이 요청하지 않기)

**왜**: 복잡한 작업을 한 번에 요청하면 불완전하거나 잘못된 결과가 나올 수 있습니다.

**어떻게**:
- 큰 작업을 작은 단계로 분할
- 각 단계를 검증하고 다음으로
- 중간 결과물 확인

**❌ Bad**:
```
새로운 결제 시스템을 구현해줘. Stripe 연동하고, 환불 처리하고, 
관리자 대시보드도 만들고, 이메일 알림도 보내줘.
```

**문제점**:
- 너무 많은 요구사항 → 일부만 구현되거나 품질 저하
- 중간 검증 불가능 → 잘못된 방향으로 진행 가능
- 롤백 어려움

**토큰 영향**: 거대한 구현 시도 → 10,000-30,000 토큰 (실패 가능성 높음)

**✅ Good**:
```
Step 1: 먼저 Stripe 결제 연동 기본 구조를 구현해줘.

범위:
- 결제 생성 (Payment Intent)
- 결제 확인 콜백
- 기본 에러 핸들링

제외:
- 환불 (다음 단계)
- 관리자 UI (다음 단계)
- 이메일 알림 (다음 단계)

완료 조건:
- 테스트 모드에서 결제 성공 확인
- 단위 테스트 통과
```

**Step 1 완료 후**:
```
✅ Step 1 완료 확인

Step 2: 이제 환불 처리를 추가해줘.

현재 상태:
- 기본 결제는 위에서 구현 완료
- src/payment/stripe.service.ts에 코드 있음

추가 기능:
- 전액 환불
- 부분 환불
- 환불 히스토리 저장

[Step 3, 4... 순차적으로 진행]
```

**개선 효과**:
- 각 단계 검증 가능 → 품질 보장
- 문제 발생시 롤백 쉬움
- 점진적 개선

**토큰 영향**: 각 단계별 2,000-3,000 토큰 × 4단계 = 8,000-12,000 토큰 (전체 품질 훨씬 향상)

#### 원칙 4: 결과 형식 지정 (출력 포맷 명시)

- 예시 제공
- 포함/제외할 내용 지정

**❌ Bad**:

**문제점**:

**✅ Good**:

```
예시 형식:

```

**개선 효과**:
- 정확한 형식 → 재작업 없음
- 명확한 요구사항 → 완성도 높은 결과

### 4.2 코딩 작업용 프롬프팅

코딩 작업은 가장 흔한 사용 케이스입니다. 작업 유형별로 효과적인 프롬프팅 방법을 다룹니다.

#### 리팩토링

리팩토링은 동작을 유지하면서 구조를 개선하는 작업입니다.

**핵심 요소**:
1. 목표와 제약 명확히
2. 변경 범위 한정
3. 테스트 유지 요구

**❌ Bad**:
```
이 코드를 리팩토링해줘
```

**문제점**:
- 리팩토링 목표 불명확 (가독성? 성능? 구조?)
- 범위 모호
- 동작 보존 보장 없음

**토큰 영향**: 불필요한 변경 + 재작업 → 5,000-8,000 토큰

**✅ Good**:
```
src/services/order.service.ts의 processOrder 메서드를 리팩토링해줘.

목표:
- 현재 200줄 함수를 작은 단위로 분리
- 각 단계별 책임 명확히 (검증, 계산, 저장)
- 테스트 가능성 향상

제약:
- 기존 public API 유지 (processOrder 시그니처 동일)
- 모든 기존 테스트 통과
- 동작 변경 없음 (순수 리팩토링)

방향:
- 검증 로직 → validateOrder()
- 가격 계산 → calculateTotalPrice()
- DB 저장 → saveOrderToDatabase()
- 각 함수는 private 메서드로

리팩토링 후:
- 기존 테스트 실행해서 통과 확인
- 변경 사항 요약
```

**개선 효과**:
- 명확한 목표 → 정확한 리팩토링
- 테스트 보존 → 안전성 보장
- 범위 한정 → 불필요한 변경 방지

**토큰 영향**: 정확한 리팩토링 + 검증 → 2,500-3,500 토큰 (50-60% 절감)

**Before/After 비교**:

```typescript
// Before: 200줄 단일 함수
async processOrder(orderData: OrderInput) {
  // 검증 로직 50줄
  // 계산 로직 80줄
  // 저장 로직 40줄
  // 에러 처리 30줄
}

// After: 작은 함수들로 분리
async processOrder(orderData: OrderInput) {
  const validated = await this.validateOrder(orderData);
  const calculated = await this.calculateTotalPrice(validated);
  return await this.saveOrderToDatabase(calculated);
}

private async validateOrder(data: OrderInput) { /* 50줄 */ }
private async calculateTotalPrice(data: ValidatedOrder) { /* 80줄 */ }
private async saveOrderToDatabase(data: CalculatedOrder) { /* 40줄 */ }
```

#### 버그 수정

버그 수정은 재현 단계와 예상 동작이 중요합니다.

**핵심 요소**:
1. 재현 단계 제공
2. 예상 vs 실제 동작 설명
3. 관련 파일만 언급

**❌ Bad**:
```
장바구니가 안 돼. 고쳐줘.
```

**문제점**:
- "안 된다"가 무엇인지 불명확
- 재현 방법 없음
- 디버깅을 처음부터 시작

**토큰 영향**: 전체 장바구니 시스템 분석 + 추측 → 8,000-15,000 토큰

**✅ Good**:
```
장바구니에서 수량 업데이트시 총 가격이 갱신되지 않는 버그를 수정해줘.

재현 단계:
1. 상품을 장바구니에 추가 (정상 작동)
2. 장바구니 페이지에서 수량을 2 → 5로 변경
3. 수량은 업데이트되지만 총 가격은 수량 2 기준으로 유지됨

예상 동작:
- 수량 변경시 총 가격 = (단가 × 새 수량)으로 즉시 갱신

실제 동작:
- 수량만 변경되고 총 가격은 이전 값 유지
- 페이지 새로고침하면 정상 가격 표시 (서버는 정상)

관련 파일:
- src/components/Cart.tsx (UI 컴포넌트)
- src/hooks/useCart.ts (상태 관리)

추측:
- updateQuantity 호출시 total 재계산 누락?
```

**개선 효과**:
- 정확한 재현 → 빠른 버그 발견
- 예상 동작 명시 → 정확한 수정
- 관련 파일 한정 → 토큰 절약

**토큰 영향**: 필요한 파일만 분석 + 수정 → 1,500-2,500 토큰 (80% 절감)

**수정 프로세스**:

```typescript
// Before: 버그 있는 코드
const updateQuantity = (itemId: string, quantity: number) => {
  setItems(items.map(item => 
    item.id === itemId ? { ...item, quantity } : item
  ));
  // total 재계산 누락!
};

// After: 수정된 코드
const updateQuantity = (itemId: string, quantity: number) => {
  const updatedItems = items.map(item => 
    item.id === itemId ? { ...item, quantity } : item
  );
  setItems(updatedItems);
  
  // total 재계산 추가
  const newTotal = calculateTotal(updatedItems);
  setTotal(newTotal);
};
```

#### 새 기능 추가

새 기능은 명세를 먼저 작성하고 단계별로 진행합니다.

**핵심 요소**:
1. 명세 우선 작성
2. 단계별 구현 요청
3. 테스트와 함께 진행

**❌ Bad**:
```
즐겨찾기 기능을 추가해줘
```

**문제점**:
- 기능 스펙 불명확
- UI/UX 정의 없음
- 테스트 계획 없음

**토큰 영향**: 불완전한 구현 + 재작업 → 10,000-20,000 토큰

**✅ Good (2단계 접근)**:

**Step 1: 명세 작성**
```
즐겨찾기 기능 명세를 작성해줘.

요구사항:
- 사용자가 상품을 즐겨찾기에 추가/제거
- 즐겨찾기 목록 페이지
- 상품 카드에 즐겨찾기 아이콘 표시

고려사항:
- 로그인 필수 (비로그인 시 로그인 유도)
- 즐겨찾기 상태 실시간 반영
- 서버/클라이언트 상태 동기화

명세 형식:
- 기능 설명
- API 스펙 (엔드포인트, 파라미터, 응답)
- UI 컴포넌트 목록
- 상태 관리 방법
- 테스트 시나리오

명세 작성 후 구현은 내가 승인한 다음에 진행해줘.
```

**Step 2: 명세 승인 후 구현**
```

Phase 1: 백엔드 API 구현
- POST /api/favorites (추가)

Phase 1 완료 후 확인하고 Phase 2 진행.

Phase 2: 프론트엔드 상태 관리
- useFavorites 훅 구현

- 즐겨찾기 버튼 컴포넌트
```

**개선 효과**:

- 단계별 진행 → 중간 검증 가능
- 테스트 포함 → 품질 보장

**토큰 영향**: 명세 1,500 + 구현 Phase별 2,000-3,000 → 총 7,500-10,500 토큰 (고품질 보장)

#### 코드 리뷰

코드 리뷰는 범위와 우선순위를 명확히 해야 합니다.

**핵심 요소**:
1. 리뷰 범위 지정 (보안/성능/가독성 중 선택)
2. 변경 사항에만 집중
3. 자동 수정 vs 제안만 구분

**❌ Bad**:
```
이 PR 리뷰해줘
```

**문제점**:
- 리뷰 관점 불명확
- 전체 파일 vs 변경사항 모호
- 자동 수정 여부 불명확

**토큰 영향**: 전체 파일 읽기 + 포괄적 리뷰 → 8,000-15,000 토큰

**✅ Good**:

**일반 리뷰**:
```
PR #123의 변경사항을 리뷰해줘.

리뷰 범위:
1. 보안 (높은 우선순위)
   - SQL Injection 가능성
   - XSS 취약점
   - 인증/인가 누락
2. 버그 가능성 (중간 우선순위)
   - Null/undefined 처리
   - 에러 핸들링
   - Edge case
3. 코드 품질 (낮은 우선순위)
   - 가독성
   - 중복 코드

제외:
- 스타일/포매팅 (ESLint가 처리)
- 타이포 (별도로 확인함)

출력 형식:
각 이슈별로:
- [심각도] 파일명:라인 - 문제 설명
- 이유
- 수정 제안

자동 수정 안 함. 제안만 해줘.
```

**긴급 보안 리뷰**:
```
PR #123을 보안 관점에서만 긴급 리뷰해줘.

집중 항목:
- SQL Injection
- XSS
- CSRF
- 인증/인가 우회
- 민감 정보 노출

변경된 파일만:
- src/api/users.controller.ts
- src/middleware/auth.ts

발견시 즉시 리포트하고, Critical 이슈는 수정 제안까지.

다른 항목 (성능, 가독성 등)은 나중에 별도 리뷰.
```

**자동 수정 포함 리뷰**:
```
src/utils/validators.ts의 변경사항을 리뷰하고 문제 발견시 자동 수정해줘.

자동 수정 허용 항목:
- Null 체크 누락
- 에러 핸들링 개선
- 타입 안정성 강화
- 명백한 버그

수정 후:
- 변경 내용 요약
- 테스트 실행
```

**개선 효과**:
- 범위 한정 → 집중 리뷰
- 우선순위 명확 → 중요한 것 먼저
- 자동 수정 여부 명시 → 작업 흐름 명확

**토큰 영향**: 
- 전체 리뷰: 8,000-15,000 토큰
- 범위 한정 리뷰: 2,000-4,000 토큰 (70% 절감)

#### API 문서

API 문서는 일관된 형식이 중요합니다.

**❌ Bad**:
```

```

**✅ Good**:
```

\`\`\`markdown

\`\`\`

**Example:**
\`\`\`bash
curl -X GET https://api.example.com/api/products/123 \\

\`\`\`
\`\`\`

한 번에 2-3개 엔드포인트씩 작성하고 확인 후 계속.
```

**개선 효과**:
- 일관된 템플릿 → 읽기 쉬운 문서

**토큰 영향**: 엔드포인트당 500-800 토큰 (템플릿 재사용으로 효율적)

### 4.4 리팩토링/리뷰용 프롬프팅

대규모 코드베이스의 리팩토링이나 리뷰는 전략적 접근이 필요합니다.

#### 전체 스캔 vs 특정 부분

**전체 스캔 (탐색 모드)**:

목적: 문제 영역 파악, 우선순위 결정

**✅ Good**:
```
프로젝트 전체에서 보안 취약점을 스캔해줘.

범위: src/ 디렉토리 전체

집중 항목:
1. SQL Injection 가능성
2. XSS 취약점
3. 하드코딩된 시크릿
4. 인증/인가 누락

출력 형식:
각 발견 사항을:
- [심각도] 파일:라인 - 문제
- 간단한 설명 (1-2줄)
- 영향도 (High/Medium/Low)

상세 분석이나 수정은 하지 말고, 목록만 먼저.
우선순위 High인 항목 발견시 즉시 알려줘.

예상: 20-30개 항목 발견 (목록만이라 토큰 효율적)
```

**개선 효과**:
- 목록만 → 빠른 스캔
- 우선순위 파악 → 중요한 것부터
- 토큰 절약

**토큰 영향**: 전체 스캔 + 목록 생성 → 5,000-8,000 토큰 (상세 분석 대비 70% 절감)

---

**특정 부분 (상세 분석)**:

스캔 후 중요한 부분을 상세 분석합니다.

**✅ Good**:
```
위에서 발견한 High 우선순위 항목 3개를 상세 분석하고 수정해줘.

항목:
1. src/api/users.controller.ts:45 - SQL Injection
2. src/middleware/auth.ts:23 - 인증 우회 가능
3. src/config/database.ts:12 - 하드코딩된 비밀번호

각 항목별:
1. 현재 코드 (문제 부분)
2. 왜 위험한지 (공격 시나리오)
3. 수정 코드
4. 테스트 방법

하나씩 순차적으로 진행. 각 수정 완료 후 다음.
```

**개선 효과**:
- 우선순위 집중 → 효과적
- 상세 분석 → 제대로 된 수정
- 순차 진행 → 검증 가능

**토큰 영향**: 항목당 1,500-2,500 토큰 × 3 = 4,500-7,500 토큰

---

#### 우선순위 지정

리팩토링/리뷰는 우선순위가 중요합니다.

**❌ Bad**:
```
이 프로젝트를 개선해줘
```

**✅ Good**:
```
이 프로젝트의 개선점을 찾고 우선순위별로 분류해줘.

분석 범위: src/ 전체

분류 기준:
- **Critical**: 보안, 데이터 손실, 심각한 버그
- **Important**: 성능 이슈, 확장성 문제, 기술 부채
- **Nice-to-have**: 가독성, 코드 스타일, 최적화

각 카테고리별:
- 문제 설명
- 영향도 (사용자/시스템/개발자)
- 예상 수정 시간
- 수정 난이도

출력 형식:
\`\`\`markdown
## Critical (즉시 수정)
1. [보안] SQL Injection in users.controller.ts
   - 영향: 전체 DB 노출 가능
   - 수정 시간: 2시간
   - 난이도: 중

## Important (이번 주)
1. [성능] N+1 쿼리 in products.service.ts
   - 영향: API 응답 시간 5초
   - 수정 시간: 4시간
   - 난이도: 중

## Nice-to-have (백로그)
1. [가독성] 중복 코드 in utils/
   - 영향: 유지보수성
   - 수정 시간: 3시간
   - 난이도: 하
\`\`\`

분류 후 Critical 항목부터 수정 진행 여부 결정.
```

**개선 효과**:
- 명확한 우선순위 → 효율적 작업
- 영향도 파악 → 의사결정 용이
- 예상 시간 → 계획 수립 가능

**토큰 영향**: 분석 + 분류 → 6,000-10,000 토큰 (가치 있는 투자)

---

#### 변경 영향도 고려

리팩토링시 영향도 분석은 필수입니다.

**✅ Good**:
```
src/services/payment.service.ts를 리팩토링하려고 해.
먼저 변경 영향도를 분석해줘.

분석 항목:
1. 이 파일을 import하는 모든 파일 목록
2. Public API (외부에서 사용하는 메서드)
3. 테스트 파일 존재 여부
4. 의존성 (이 파일이 사용하는 다른 서비스)

출력:
\`\`\`markdown
## 영향도 분석

### 직접 의존 파일 (5개)
- src/api/checkout.controller.ts
- src/api/subscriptions.controller.ts

- src/services/order.service.ts

### Public API

⚠️ 주의: Public API 변경시 위 5개 파일 모두 수정 필요

### 테스트
✅ src/services/__tests__/payment.service.test.ts 존재
- 커버리지: 85%
- 테스트 수: 23개

### 의존성
- StripeService (외부 API)
- DatabaseService
- LoggerService

## 권장 사항

- 테스트 먼저 100% 통과 확인
- 리팩토링 후 의존 파일들의 테스트도 실행
\`\`\`

이 분석 기반으로 리팩토링 계획 수립.
```

**개선 효과**:
- 영향도 사전 파악 → 안전한 리팩토링
- 테스트 전략 수립 → 품질 보장
- 리스크 최소화

**토큰 영향**: 영향도 분석 → 2,000-3,500 토큰 (리팩토링 실패 방지)

#### 단계별 리팩토링 계획

대규모 리팩토링은 계획이 중요합니다.

**✅ Good**:
```
레거시 인증 시스템 (src/auth/)을 현대화하는 계획을 수립해줘.

현재 상태:
- 5년된 코드
- 세션 기반 (JWT로 전환 원함)
- 테스트 없음
- 3,000줄 코드

목표:
- JWT 기반 인증
- 테스트 커버리지 80% 이상
- 기존 사용자 호환성 유지
- 다운타임 없음

계획 형식:
각 Phase별로:
1. 목표 (무엇을)
2. 작업 항목 (상세)
3. 테스트 전략
4. 롤백 계획
5. 예상 시간
6. 리스크 및 완화

Phase는 4-6개로 나누고, 각 Phase는 1-2주 내 완료 가능하게.

계획 승인 후 Phase별로 실행.
```

**출력 예시**:
```markdown
## Phase 1: 기반 준비 (1주)
목표: 기존 시스템 테스트 작성, 새 구조 설계

작업:
1. 레거시 코드 테스트 작성 (현재 동작 보존)
2. JWT 인증 구조 설계 문서
3. 새 auth 모듈 스켈레톤 생성

테스트:
- 레거시 시스템 테스트 80% 커버리지

롤백:
- 새 코드는 아직 사용 안 함 (롤백 불필요)

리스크:
- 테스트 작성이 예상보다 오래 걸릴 수 있음
- 완화: 핵심 플로우 우선, 엣지 케이스는 Phase 2

## Phase 2: 병렬 구현 (2주)
목표: JWT 시스템 구현, 레거시와 병렬 운영

작업:
1. JWT 발급/검증 구현
2. 새 로그인 엔드포인트 (/auth/v2/login)
3. Feature flag로 제어 (기본 off)

테스트:
- 새 시스템 테스트 100%
- 통합 테스트 (레거시 영향 없음 확인)

롤백:
- Feature flag off → 레거시로 즉시 복귀

리스크:
- 두 시스템 병렬 운영 복잡도
- 완화: 명확한 분리, 공통 코드 최소화

[Phase 3, 4, 5 계속...]
```

**개선 효과**:
- 명확한 로드맵 → 팀 정렬
- 리스크 관리 → 안전한 마이그레이션
- 점진적 진행 → 지속적 배포 가능

**토큰 영향**: 계획 수립 → 4,000-6,000 토큰 (실행 실패 방지로 큰 가치)

```
- 필요한 것만 → 토큰 절약
- 단계별 진행 → 고품질 결과
평균 토큰: 3,000-7,000 (품질 높음)
```
